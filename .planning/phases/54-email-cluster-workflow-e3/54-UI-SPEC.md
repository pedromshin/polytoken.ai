# Phase 54 — UI Design Contract: Email-Cluster Workflow (E3)

> Visual and interaction contract for the killer-feature phase: email-thread cards on the
> chat canvas, thread-bound chats, `web_search` tool rounds, suggest-only source capture, and
> a cluster-context indicator. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
> **This phase is a CHROME EXTENSION over three already-shipped machines — the versioned
> canvas node-type registry (CANVAS-03), the tool-round chrome (Phase 39), and the
> confirm-action widget spine (Phase 40) — not a redesign.** It consumes the Phase 48 token
> system and the Phase 51 interactive-state convention unchanged. Depth-first mandate
> (54-CONTEXT.md): every visual decision below optimizes for the ONE CLUS-07 live scenario
> (real thread → attach chat → web research with thread in context → sources captured →
> promotion confirmed → a follow-up chat sees cluster context) working end-to-end on the
> user's real inbox — no multi-cluster management chrome, no cluster UI polish beyond what
> that scenario needs.

Unattended overnight run (54-CONTEXT.md `Mode`): every open design question below is resolved
with an explicit, documented judgment call (see **Judgment Calls**) rather than left ambiguous
for the executor.

### 1. `EmailThreadNode` — the canvas node type (CLUS-01)

alongside `ChatNode`/`GenuiPanelNode`/`KnowledgePreviewNode`, registered in `node-types.ts` as

**`node.data` shape** (new `EmailThreadNodeDataSchema` in `node-data-schemas.ts`, mirroring
`KnowledgePreviewNodeDataSchema`'s exact provenance-ref-only discipline — `.strict()`, never
carries fetched content):

```

  threadId: z.string().uuid(),
  label: z.string().max(120).optional(),   // optional custom override, same as KP's `label`
}).strict();
```

**Shell** (byte-identical recipe to `KnowledgePreviewNode`'s outer shell, per the constraint's
explicit "closest node-type analog" pointer):

```

"ring-2 ring-primary ring-offset-1 shadow-elevation-2" : "shadow-elevation-1"}`}>
  <Handle type="target" position={Position.Left} />
  {/* header */}
  {/* body */}
  {/* footer */}
  <Handle type="source" position={Position.Right} />
</div>
```

**Header** (`h-9 node-drag-handle`, `bg-muted/40 border-b border-border/60`, matches
`GenuiPanelNode`'s header bar exactly):

```
<div className="node-drag-handle flex h-9 shrink-0 cursor-grab items-center justify-between gap-2 border-b border-border/60 bg-muted/40 px-3 active:cursor-grabbing">
  <span className="flex min-w-0 items-center gap-2">
    <Mail className="size-3 shrink-0 text-graph-email" aria-hidden />
    <span className="truncate text-xs font-normal text-muted-foreground">{headerLabel}</span>
  </span>

    className="flex size-6 shrink-0 items-center justify-center rounded-md text-muted-foreground transition-colors hover:text-destructive focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-1 pointer-coarse:touch-target"
    onClick={(e) => { e.stopPropagation(); void deleteElements({ nodes: [{ id }] }); }}>
    <X className="size-3.5" aria-hidden />
  </button>
</div>
```

`headerLabel` resolution order (mirrors `resolveHeaderLabel`'s exact 3-step precedent from
`knowledge-preview-node.tsx`): explicit `data.label` → the fetched thread's own `subject` once
the query settles → the fallback literal **"Untitled thread"** (new fallback string — `Threads.subject`
is nullable per `packages/db/src/schema/threads.ts`, so this branch is a real, reachable case,
not defensive-only).

**Body** — loading/error/empty/success branch order **identical to
`KnowledgePreviewMiniGraph`'s established precedent** (loading → error → not-found/empty →
success), rendered inside a `relative flex-1 flex-col gap-1 px-3 py-2` wrapper:

| Branch | Condition | Rendering |
|---|---|---|
| Loading | `query.isPending` | `role="status" aria-label="Loading thread"` — two `Skeleton` bars (`h-3 w-3/4` for a participants-row placeholder, `h-3 w-full` × 3 for a summary placeholder), same `@polytoken/ui/skeleton` primitive `KnowledgePreviewMiniGraph` already imports |
| Error | `query.isError` | `EmptyState icon={AlertCircle} tone="destructive" layout="centered" size="compact" heading="Couldn't load this thread." body="Try again, or open it from your inbox." action={{label:"Retry", onClick: () => void query.refetch()}}` — same `~/components/empty-state` primitive, same copy shape as `KnowledgePreviewMiniGraph`'s own error branch |
| Empty / not found | `query.data === null` (thread deleted, or importer no longer owned) | `EmptyState icon={Mail} tone="muted" layout="centered" size="compact" heading="This thread is unavailable." body="It may have been removed or is no longer accessible."` |
| Success | `query.data` present | Participants row + summary text (below) |

**Success-state body markup:**

```
<div className="flex min-w-0 items-center gap-1.5 text-xs text-muted-foreground">
  <Users className="size-3 shrink-0" aria-hidden />
  <span className="truncate">{participantsSummary}</span>
</div>
<p className="mt-1 line-clamp-4 text-xs font-normal text-foreground">{summaryText}</p>
```

`participantsSummary` — same "+{n} more" overflow convention as
`resolveFooterCopy`/`ProvenanceLink`'s dedupe idiom: join up to 3 participant display names
(sender name, falling back to sender address, deduped) with `", "`, then `" +{n} more"` if the
thread has more. `summaryText` is the thread's latest-message snippet (54-CONTEXT.md's
"prefer cheap/latest-preview tonight" resolution — never an LLM-generated summary this phase).

**Footer** (`h-9`, `border-t border-border/60`, two actions):

```
<div className="flex h-9 shrink-0 items-center justify-between gap-1 border-t border-border/60 px-2">
  <Link href={hrefFor("email", latestMessageId)}
    className="flex h-7 shrink-0 items-center gap-1 rounded-md px-2 text-xs text-muted-foreground transition-colors hover:bg-accent hover:text-accent-foreground focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-1 pointer-coarse:h-11">
    Open thread →
  </Link>

    className="h-7 gap-1 pointer-coarse:h-11" disabled={isAttaching} onClick={handleAttachChat}>
    {isAttaching ? <Loader2 className="size-3.5 animate-spin" aria-hidden /> : <MessageSquarePlus className="size-3.5" aria-hidden />}
    Attach chat
  </Button>
</div>
```

`hrefFor("email", latestMessageId)` reuses `provenance-link.tsx`'s existing route switch
verbatim — **zero new route is invented**; "Open thread" deep-links to the thread's latest
member email's detail page (`/emails/{id}`), the only thread-scoped surface that already
exists (Judgment Call #4).

## Judgment Calls (unattended-run discretion resolutions)

Explicit record per the constraints — each is a deliberate design decision, not an oversight,
made because 54-CONTEXT.md left it to "Claude's Discretion" or because implementing
CLUS-01/02/03/04/06 required a call CONTEXT didn't fully spell out visually.

1. **`EmailThreadNode`'s shell is a fixed `h-[220px] w-[320px]`, byte-identical recipe to
   `KnowledgePreviewNode`'s outer shell** (border/shadow/selection-ring/entrance-animation
   classes copied verbatim) — the objective's own `<required_reading>` names
   `knowledge-preview-node.tsx` as "the closest node-type analog for the thread card," and its
   content (server-truncated to `THREAD_SNIPPET_CHARS=240`) is bounded the same way
   `KnowledgePreviewMiniGraph`'s capped mini-graph is, licensing the same fixed-dimension
   choice over `GenuiPanelNode`'s `min-h`/unbounded-content approach.
2. **The header `Mail` icon does NOT use `text-primary`**, diverging from
   `KnowledgePreviewNode`'s pre-existing `text-primary` `Share2` icon. Research found
   `GenuiPanelNode`'s own header icon (`PanelsTopLeft`) is `text-muted-foreground` and
   `ChatNode`'s node-kind identity signal is its allowlisted `border-l-2 border-l-primary`
   stripe, NOT an icon color — `KnowledgePreviewNode`'s icon is the outlier among the three
   existing node types, most plausibly a pre-51-audit straggler never caught (51-UI-SPEC's
   scout was grep-scoped to specific file lists that didn't include this file). This phase
   follows the CURRENT majority convention (neutral icon, accent reserved for selection state
   only) rather than propagating the drift to a 4th node type.
3. **Registry key is `"email-thread"` (kebab-case), not `email_thread`** (the requirement
   prose's snake_case). Matches the sibling registry-key convention exactly (`"chat"`,
   `"genui-panel"`, `"knowledge-preview"`) in `node-types.ts` — `email_thread` is the
   conceptual/DB-adjacent name (CLUS-01's own prose, migration-adjacent), the concrete React
   Flow `type` string follows the established web-layer naming.
4. **"Open thread" deep-links to `/emails/{latestMessageId}` via the EXISTING
   `hrefFor("email", id)` switch**, not a new `/threads/[id]` route. Research confirmed no
   thread-detail route exists yet (grep-verified); inventing one is out of this phase's
   depth-first scope. The thread's latest member email is already a fully-formed detail
   surface (subject, body, entities) and is the closest existing "open this thread" landing
   spot — zero new routing surface, consistent with 41-UI-SPEC's "never mint a new alias"
   spirit applied to routes.
5. **`AddEmailThreadPopover` is a search-select `Command` list, not a manual-paste form**
   (diverging from `AddKnowledgePreviewPopover`'s precedent). 41-CONTEXT.md's manual-paste
   choice was explicit because NO knowledge-node picker UI existed at the time; 54-CONTEXT.md
   explicitly specifies "a thread picker listing the user's threads," and `emails.listThreads`
   already returns exactly the rows a picker needs at personal-use scale — reusing
   `ModelPicker`'s exact `Popover`+`Command` composition is lower-risk than either building a
   new picker shape or force-fitting the manual-paste form onto data that's already listable.
6. **One `ThreadClusterIndicator` component serves both CLUS-02's linked-thread indicator and
   CLUS-06's cluster-context indicator**, as two sections of one popover, rather than two
   separate header elements. The `h-11` top bar has a proven, already-tight width budget
   (53-UI-SPEC's own note about `ChatCanvasViewToggle`/`ModelPicker`/`CostMeter` fitting at
   360px) — a second competing pill would erode that margin for no user benefit, since both
   concepts ("this chat is linked to a thread" / "this chat can see cluster context") are
   facets of the SAME fact (`conversation.threadId` is set) and are naturally read together.
7. **No citation chips for raw `web_search` results this phase** — a search result URL isn't
   yet an internal `ProvenanceKind` object (`email`/`entity`/`knowledge`), and minting a new
   `ProvenanceKind` ("web"/"source") for pre-capture results would create a citation affordance
   that immediately becomes stale/orphaned the moment the source IS captured (it would then
   need a SECOND representation as a `knowledge`-kind chip). Keeping raw search rounds
   chip-free until confirmed capture avoids a doubled citation vocabulary and keeps CLUS-04's
   "suggest-only, nothing writes without confirmation" boundary legible in the UI too — nothing
   is presented as "citable" until a human has actually confirmed it.
8. **The source-capture confirm widget introduces zero new visible copy or components** —
   54-CONTEXT.md's own instruction is to reuse "the existing confirm-action widget machinery
   (v1.6) — nothing writes without confirmation," and research confirmed that machinery's
   visible surface (fixed "Confirm"/"Reject" cards + the agent's own preceding text carrying
   the actual question) is already fully generalized over `widgetKind`. Inventing new card
   copy or a distinct visual treatment for this one `suggestionRef.kind` would fragment a
   pattern that intentionally looks identical across every confirm-action use case today.
9. **`EmailThreadNode`/`AddEmailThreadPopover` are canvas-only and unreachable below `md`,
   with no bespoke mobile presentation built this phase.** This mirrors the EXISTING,
   already-shipped precedent for `KnowledgePreviewNode`/`AddKnowledgePreviewPopover` — 53-UI-SPEC
   never built a mobile list/feed equivalent for canvas-only node types (unlike genui panels,
   which have a message-stream provenance ref and therefore a natural inline-feed rendering
   path). CLUS-07's live acceptance scenario is not specified as a mobile scenario, and the
   depth-first mandate (54-CONTEXT.md: "ONE scenario fully working beats five half-working")
   argues against building new mobile canvas-node infrastructure this phase. `ThreadClusterIndicator`
   — the one genuinely cross-viewport-persistent piece of this phase's UI — DOES get a full
   mobile contract (Component 6) since it lives in the always-mounted header bar, not the canvas.
