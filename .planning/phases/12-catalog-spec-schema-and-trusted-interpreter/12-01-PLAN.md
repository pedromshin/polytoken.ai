<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| spec JSON → schema parse | An untrusted-shaped spec object (in Phase 13, LLM output; this phase, a hardcoded fixture) crosses into typed territory only via `SpecRootSchema.safeParse`. The schema is the gate. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-01 | Tampering | SpecRootSchema (extra/unknown keys) | mitigate | Every `z.object` ends in `.strict()` (D-22) — `additionalProperties:false` rejects injected keys including `__proto__`/`constructor` at the JSON-Schema level before any object is constructed. |
| T-12-02 | Denial of Service | recursive spec tree (deep/wide) | mitigate | Schema authors `MAX_SPEC_NODES=200` + `MAX_SPEC_DEPTH=8` via `.refine()` + `countNodes`/`specDepth` walkers (D-24, SAFE-06 seam) so an unbounded tree fails validation, not the renderer. |
| T-12-03 | Elevation of Privilege | `action`/`dataRef` fields as code | mitigate | `action` and `dataRef`/`valueRef` are typed `z.string()` only — declared identifiers, never expressions. The schema makes executable code unrepresentable (SPEC-04/SPEC-05); no `eval` surface is introduced. |
| T-12-04 | Spoofing | spec version | mitigate | `v: z.literal(1)` forces exact-version emission; a stale or forged version literal fails parse immediately (SEAM-01, D-10). |
| T-12-SC | Tampering | npm installs (zod, react, vitest) | mitigate | No NEW third-party runtime packages introduced — zod/react already in the monorepo at the pinned versions; vitest is an existing devDependency. No package-legitimacy checkpoint required (RESEARCH has no new-install audit row because there is no new install). |
</threat_model>
