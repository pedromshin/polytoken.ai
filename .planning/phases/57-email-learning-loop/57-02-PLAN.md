<objective>
Close LEARN-02 gap 3: `EntityTypeClassifierProtocol.classify()` is the one
classification call site with zero learning mechanism — cold every time. This plan
adds a few-shot `examples` parameter (mirroring the already-shipped
`AutofillProtocol.autofill` mechanism), feeds it from Plan 57-01's importer-scoped
`entity_type_corrections` retrieval, and renders it as a delimited block in the
Bedrock user turn (D-14). The accumulated corrections now bias the classifier's
SUGGESTION — never its decision.

Purpose: Satisfies ROADMAP success criterion 2 for the classification axis — a later
region that resembles a previously-corrected one is classified using the accumulated
signal, measurably differing from the pre-correction behavior. Proven with a
deterministic test at the mockable classifier boundary (Bedrock may be unavailable
overnight — the LLM call stays behind the protocol seam).

Output: `classify()` gains `examples`; the adapter renders `<entity_type_examples>`;
`SuggestEntityTypesUseCase` retrieves + passes them; a deterministic differ-with-vs-without
test and an explicit suggest-only invariant test.

Decisions applied (from 57-RESEARCH.md, locked in 57-01):
- Q2 → retrieval is **trgm-only, importer-scoped** (reuses 57-01's RPC). NO new embed
  call is added to the classification hot path — this preserves the "one Bedrock call
  per document" reliability constraint (entity_type_classifier_protocol.py:1-5). The
  candidate region's raw text is the trgm query; degrade-to-nothing on failure (D-13).
- Q1 → corrections include both pre- and post-confirm reclassifications (captured that
  way in 57-01); no additional filtering here.
</objective>
