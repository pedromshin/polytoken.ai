<objective>
Build the gate chain (ANTIC-02): each `AnticipatoryCandidate` from Plan 25-01 is filtered by an
appropriateness eval (gate #1, D-07) AND a hard frequency cap (gate #2, D-10) as INDEPENDENT checks
(D-08), must pass BOTH, and only a survivor becomes a Phase-24 proposal card requiring explicit user
acceptance (D-11). Every candidate's lifecycle is recorded as run-event-shaped events (D-13). The
whole pipeline stays dark by default (D-12) and adds NO new DB table (D-14 planner decision — see below).

Purpose: Prove the safety gates deterministically against the Plan 25-01 fixtures — that a
false-positive-averse eval AND an independent frequency cap both stand between a trigger and the user,
and that nothing ever fires without an explicit click. This is the evidence Plan 25-03's go/no-go
findings rest on.

Output: The two gate ports + their spike adapters (a Haiku appropriateness judge mirroring the
code-island judge, and an in-memory cap store), the gate-chain use case with lifecycle recording and
explicit-accept mapping, a stub judge for deterministic tests, and light DI wiring proving the dark
pipeline boots.

## Planner decision (D-14): NO new DB table for the spike
The candidate lifecycle (D-13) is modeled as ordered `AnticipatoryLifecycleEvent` records that mirror
the Phase-22 `ChatRunEvent` `type`+`data` shape (SEAM-04) and are returned from the pipeline + emitted
via structlog — reusing the run-event *schema* without a migration. The frequency cap reads prior
"shown" state through the narrow `AnticipatoryCapStore` port; the spike adapter is IN-MEMORY, seeded
from fixtures to simulate a reloaded conversation. D-10's "persists / survives reload" is satisfied
*architecturally* by the port — the production adapter projects the same events off the existing
`chat_run_events` substrate (a run-event projection, not a new table). This is the minimal, defensible
spike choice D-14 explicitly delegates to the planner; the persistence seam is documented in Plan 25-03's findings.
</objective>
