# Phase 23 — UI Design Contract

> Visual and interaction contract for the `/chat` Canvas view — a 2D infinite React Flow surface
> hosting genui panels and the chat itself as draggable/pannable nodes, wired by data-carrying
> edges over a per-chat shared-state store. Generated by gsd-ui-researcher (autonomous session —
> no user prompts; all choices derived from `23-CONTEXT.md`'s 10 locked decisions [D-01..D-10],
> the `/knowledge` React Flow reference implementation, and the approved `22-UI-SPEC.md` chat
> contract this phase extends). Verified by gsd-ui-checker.

**Extends, does not replace:** the Phase 22 chat surface (`ConversationView`'s h-11 frosted
toolbar, `Composer`, `MessageList`, `GenuiPartBoundary`'s `GenuiCard` wrapper) and the `/knowledge`
React Flow chrome (`Background`/`Controls`/`MiniMap` composition, node color-by-type language,
`nodeClasses` helper, dagre auto-layout, no-font-medium rule). No new brand colors, radii, or
fonts are introduced anywhere in this spec. Canvas is a VIEW of a conversation, not a new page —
it lives inside `/chat/[conversationId]`.

## Layout & Structure

### View toggle (D-02)

Placement: inside `ConversationView`'s existing `h-11` frosted toolbar
(`border-b border-border/50 bg-background/70 backdrop-blur-md px-4`, from `22-UI-SPEC.md`), as a
new leftmost toolbar group, before `ModelPicker`:

```
[ Chat | Canvas ]  ModelPicker  ················  CostMeter
```

- Component: a two-segment toggle built from `@nauta/ui/tabs` (Radix `Tabs`, used as a
  segmented-control, `TabsList` with two `TabsTrigger`s: "Chat" / "Canvas") — NOT a `Switch`
  (this is a mode selector with two named destinations, not a boolean). `size="sm"` scale to fit
  the h-11 row without changing toolbar height.
- Default: "Chat" (docked classic view) per D-02 — canvas is opt-in per conversation, remembered
  in `localStorage["nauta.chat.canvas-view:{conversationId}"]` (mirrors the existing
  `nauta.knowledge.taxonomy-banner-dismissed` per-conversation-scoped-key convention) so returning
  to a conversation restores the last-used view without a server round-trip.
- `aria-label="View"` on the `TabsList`; each trigger's accessible name is its visible label
  ("Chat view" / "Canvas view" via `aria-label` override so screen readers don't just hear "Chat").
- Switching to "Canvas" mounts the React Flow surface via `dynamic(ssr: false)` (mirrors
  `knowledge-graph-island.tsx`'s pattern) — never SSR the canvas.

### Canvas surface layout

Canvas occupies the SAME main-column slot the docked `MessageList`+`Composer` occupy today — full
remaining height below the h-11 toolbar, full width (no rail changes; `ConversationRail` stays
mounted and functions identically in both view modes, since it operates at the conversation level,
not the view level).

```
┌─────────────────────────────────────────────────────────┐
│ h-11 toolbar: [Chat|Canvas] toggle · ModelPicker · CostMeter │
├─────────────────────────────────────────────────────────┤
│                                                           │
│   React Flow surface (Background + Controls, no MiniMap  │
│   by default — see Claude's Discretion resolution below) │
│                                                           │
│   ┌──────────────┐        ┌──────────────────┐          │
│   │  chat node   │──edge──▶│  genui-panel node │         │
│   └──────────────┘        └──────────────────┘          │
│                                                           │
└─────────────────────────────────────────────────────────┘
```

**Minimap decision (Claude's Discretion, D-02 canvas chrome):** OFF by default. Rationale: unlike
`/knowledge` (which can hold hundreds of taxonomy nodes), a per-chat canvas is expected to hold a
handful of panels (one chat node + N genui panels from that conversation's turns) — a minimap adds
chrome without a navigational payoff at this scale and would be the first UI a first-time canvas
user sees, competing with the empty-canvas state. Provide a toolbar toggle (`Map` icon, `size="sm"`
`variant="ghost"`, `aria-pressed`, `aria-label="Toggle minimap"`) to enable it per-session if a conversation grows large; state is
NOT persisted (session-only, resets to off on reload — avoids a second persisted-preference axis
beyond the view-mode toggle).

**Zoom controls:** `<Controls />` (React Flow's built-in, same as `/knowledge`) bottom-left,
default position. `showZoom`, `showFitView`, `showInteractive` all `true` (interactive-lock toggle
is useful here since STATE-01/02 wiring benefits from a "lock positions" mode while inspecting
edges — reuse the built-in rather than building a custom one).

**Background:** `<Background gap={16} size={1} />` — identical props to `/knowledge` (dot-grid,
16px gap) for visual consistency across the app's two React Flow surfaces.

### Empty-canvas state

When a conversation has zero genui-panel nodes yet (fresh canvas, no genui turns emitted): render
inside the canvas pane (not as a React Flow node — a plain absolutely-centered div, same technique
as `GraphNoSchemaState`):

- Icon: `LayoutGrid` (lucide), `size-8 text-muted-foreground`
- Heading: `text-base font-semibold` — "No panels yet"
- Body: `text-sm text-muted-foreground` — "Genui responses from this chat will appear here as
  panels. Switch to Chat view to start a conversation."
- No button (the "New chat"/send action lives in the chat node or the docked Chat view — this
  empty state is informational only, mirroring `GraphNoSchemaState`'s no-button precedent for a
  state whose remedy lives on a different surface/view).
- The chat node itself IS always present once a conversation exists (D-02: chat is a first-class
  node) — so in practice this empty state only shows before the chat node's own first layout pass
  resolves, OR (edge case) if a user deletes/hides the chat node in a future phase. Treat it as a
  transient/defensive state, not the primary first-run experience — the primary first-run
  experience is: canvas opens with ONE chat node centered, ready to compose.

## Spacing Scale

Declared values (multiples of 4 only) — unchanged 8-point scale from `22-UI-SPEC.md`:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Node internal icon-to-label gaps, edge-picker option padding |
| sm | 8px | Node header internal padding, toolbar icon gaps |
| md | 16px | genui-panel node inner content padding, chat-node composer padding |
| lg | 24px | Empty-canvas icon-to-heading gap |
| xl | 32px | Default gap between auto-placed sibling panels (dagre `nodesep`/`ranksep` seed) |
| 2xl | 48px | not used this phase |
| 3xl | 64px | Empty-canvas state vertical centering padding |

Exceptions:
- Node drag handle / header row: `36px` height (not a multiple-of-4 exception — `36px` = `h-9`,
  which IS on-scale under Tailwind's spacing scale even though it isn't in the 8-pt table above;
  documented here because it's the one fixed node-chrome dimension worth calling out explicitly).
- Touch targets: edge-connection `Handle` hit areas use React Flow's default (larger than the
  visible dot) — do not shrink via CSS; this is a per-project WCAG 2.5.5 carry-over from
  `22-UI-SPEC.md`'s 44×44px rule applied to the canvas's own interactive affordances (zoom
  buttons, minimap toggle, edge-picker trigger all use `size-11` hit areas per that precedent).
- genui-panel / chat node MINIMUM dimensions: `320×240px` (see Node Visual Language) — deliberately
  larger than `/knowledge`'s compact 160×48-class taxonomy nodes, since these nodes host live
  interactive content (forms, charts, composer), not a label.

## Interaction Contracts

**View toggle (D-02):** Switching "Chat" ↔ "Canvas" is instantaneous (no transition animation
beyond `motion-safe:` cross-fade of the main-column content, ≤150ms) — both views read the SAME
underlying conversation state (messages, streaming status); switching never interrupts an
in-flight generation. The docked Chat view remains fully functional regardless of canvas state
(D-02: "docked view remains the default").

**Canvas mount/restore (CANVAS-02, D-05/D-06):** On mounting the Canvas view for a conversation,
fetch the persisted `chat_canvas_layouts` row (nodes/edges/viewport/shared_state) via a tRPC query;
apply positions/viewport exactly (`toObject()`-style restore, D-06) before the first paint settles
— use the `CanvasSkeleton` while this fetch is in flight so there's no flash of an unlaid-out
default. If no row exists yet (first-ever canvas visit for this conversation): synthesize a default
layout — one `chat` node centered at `(0,0)`, dagre auto-placement for any existing genui-panel
nodes derived from that conversation's `genui_spec` message parts (D-03) — and persist it on the
first debounced save.

**Live panel materialization (D-03, CANVAS-04):** A new `genui_spec` message part arriving during
an active canvas view (live turn completing while the user is looking at Canvas) triggers dagre
placement for the new node with an offset-cascade fallback (if dagre would overlap an existing
node, nudge by a fixed `+32,+32` step until clear) — this positioning event does NOT touch the
`nodes` array's volatile streaming content (D-07): position/size are assigned once, at
materialization time, and never recomputed while that panel's content streams. The panel fades in
(`motion-safe:animate-in fade-in duration-200`) rather than popping abruptly.

**Streaming responsiveness (CANVAS-04, D-07):** `GenuiPanelNode` and `ChatNode` are both wrapped in
`React.memo`; the `nodeTypes` map is defined at module scope (never inline in render, matching
`/knowledge`'s established pattern). Streaming text/partial-spec content is read from the
Phase-22 `useChatStream` hook's state (or the shared Zustand store) INSIDE the node component —
never lifted into the React Flow `nodes` array's `data` field, which would force a `setNodes` call
(and therefore a full nodes-array identity change) on every streamed token. Node dimensions stay
visually fixed (`min-h-[240px]` with `overflow-y-auto` inner scroll) during streaming so the graph
never relayouts mid-stream.

**Drag/pan/zoom (CANVAS-01):** Standard React Flow gestures — drag a node by its header (grip
icon + `cursor-grab`/`cursor-grabbing`), pan the canvas by dragging empty space, zoom via
scroll/pinch or the `Controls` buttons. Node body content (inside `GenuiPanelNode`/`ChatNode`)
does NOT initiate a node drag — only the header row is the drag handle (React Flow
`dragHandle=".node-drag-handle"` prop) so that clicking/scrolling/typing inside a panel's rendered
content or the chat composer never accidentally moves the node.

**Node selection:** Clicking a node (not its drag handle) selects it — `ring-2 ring-primary
ring-offset-1` (identical selection idiom to `/knowledge`). Selecting a `GenuiPanelNode` focuses
that panel for keyboard interaction (Tab now cycles within the panel's rendered content before
moving to the next node — see Accessibility). Clicking empty canvas space deselects; Escape
deselects (same as `/knowledge`).

**Edge creation (D-09, "Claude's Discretion" resolved):** User drags from a source panel's output
`Handle` (right side, `Position.Right`) to a target panel's input `Handle` (left side,
`Position.Left`). On drop, `EdgeCreationPicker` opens as a `Popover` anchored at the drop point:
two `Select` dropdowns (Source field / Target field, populated from each panel's declared-state
keys) + "Connect fields" / "Don't connect" (context-specific pair — same convention as Phase 22's
"Keep conversation"). The edge is NOT created until the user confirms — an accidental drag never
silently wires two panels together (mirrors DCUI's "never auto-fire" posture applied to canvas
wiring). Confirming validates the value's shape against the target's expected schema (FOUND-6)
before committing; a validation failure keeps the picker open with the inline error copy (see
Copywriting Contract) rather than creating a broken edge.

**Editing an existing data edge:** Clicking an edge's midpoint label pill re-opens
`EdgeCreationPicker` pre-filled with its current source/target, allowing re-pick or delete
(`Button variant="ghost" size="sm"` "Remove connection" inside the same popover, `destructive`
text color only on this one action). Deliberately NO confirmation dialog on edge removal: a data
edge is low-stakes, locally-scoped canvas wiring that is trivially re-created via the same picker —
unlike Delete-conversation (irreversible data loss, full `AlertDialog` in Phase 22), the cost of an
accidental removal is one drag, so extra chrome would be friction without protection value.

**Debounced save (D-06):** Any of {node drag end, node resize end (if resizing is ever exposed —
not in this phase's scope), edge create/delete, viewport pan/zoom settling} schedules a debounced
save (~800ms after the last such event) via `chat.saveCanvasLayout`. Rapid successive interactions
(e.g. dragging three nodes in a row) coalesce into one save. `SaveStatusIndicator` shows "Saved"
briefly on success; failures retry silently per Copywriting Contract (no user-facing retry button —
this is local/sandbox single-user data, not worth a manual-retry affordance per REQUIREMENTS Out of
Scope's "no CRDT/multiplayer" posture).

**Unknown node type (D-04, CANVAS-03):** If a persisted layout references a node `type` the
current session's `NODE_TYPE_REGISTRY` doesn't recognize, that ONE node renders as
`UnknownNodeTypePlaceholder` — position/size from the persisted layout are still honored (so the
canvas's overall spatial arrangement isn't disturbed), but no content renders and no interaction
(drag is still permitted — repositioning is harmless — but no click-to-select-and-focus content
since there IS no content). This never throws or blanks the whole canvas.

**Shared state read/write (STATE-01):** Panels read/write the per-chat Zustand store via the SAME
`useDeclaredState` 5-mutation-enum grammar the v1.1 genui renderer already uses (D-08, FOUND-4) —
this is a data/logic contract, not a new visual affordance; the ONLY new visual surface it produces
is the data-edge label pill (above) making cross-panel wiring legible.

## Accessibility

- **Keyboard pan/zoom fallback (CANVAS-01):** When canvas has focus (not inside a specific node),
  arrow keys pan the viewport (React Flow's built-in `panOnScroll`/keyboard nav, or an explicit
  `onKeyDown` handler if the library default is insufficient — verify at implementation time),
  `+`/`-` zoom, `0` resets to `fitView`. This is a REQUIRED fallback, not a nice-to-have — canvas
  must be fully operable without a pointer.
- **Focus management into nodes:** `Tab` from the toolbar moves focus into the canvas container;
  subsequent `Tab` presses cycle through nodes in a stable order (left-to-right, top-to-bottom by
  position — not DOM/z-order, which can be arbitrary after drags). Pressing `Enter`/`Space` on a
  focused-but-unselected node selects it and moves focus INTO its content (first focusable element
  inside — e.g. the chat node's composer, or a genui-panel's first interactive control). `Escape`
  from inside a node's content returns focus to the node shell (selected-but-not-entered), a second
  `Escape` deselects and returns focus to the canvas container. This two-level focus model prevents
  Tab from "getting stuck" inside a busy genui-panel's many controls when the user just wants to
  move between panels.
- **Aria labels on canvas controls:** `Controls`' built-in buttons already ship accessible labels
  (React Flow default) — verify they aren't stripped by custom styling. The canvas container itself
  gets `role="application"` + `aria-label="Conversation canvas"` (React Flow's recommended pattern
  for a custom-keyboard-nav'd graph surface, since it isn't a normal document-flow region).
  `aria-roledescription="node-based diagram"` on the same container per React Flow's own a11y
  guidance.
- **Reduced motion:** Under `prefers-reduced-motion`, disable: the new-panel fade-in animation
  (`motion-safe:animate-in`), the streaming-indicator pulsing dot (renders as a static solid dot
  instead), the "Saved" fade transition (renders/clears instantly, no fade), and any edge-drawing
  animation React Flow applies by default (set `animated: false` unconditionally on `DataEdge` —
  do NOT use React Flow's animated-dashed-edge style at all, motion-reduce or not, since a
  continuously-animated edge is exactly the kind of always-on motion `prefers-reduced-motion`
  users want eliminated and there's no functional reason a data edge needs to visually "flow").
- **Provenance/streaming indicators never rely on color alone:** the streaming dot has an
  `aria-label="Streaming"` (not just a colored pulse); the data-edge vs. plain-edge distinction is
  ALSO conveyed by the always-visible label pill (source→target text), not `primary`-stroke color
  alone, so colorblind users can distinguish edge types.
- **Live region:** canvas-level state changes (layout restored, save succeeded/failed, a new panel
  materialized from a live turn) announce via a visually-hidden `aria-live="polite"` region —
  "Canvas layout restored", "New panel added", "Layout saved" — same restrained
  transitions-only-never-content posture as `22-UI-SPEC.md`'s chat live region (never announce
  streaming token deltas).
- **Touch targets:** edge-creation `Handle` elements, the minimap toggle, and the keyboard-hint
  dismiss button all use `size-11` (44×44px) hit areas per the project-wide WCAG 2.5.5 carry-over.
- **Color contrast:** no new colors — all canvas chrome inherits already-shipped token contrast
  ratios.
