<objective>
Wire the exact-cache into the live generation path: a new `UiSpecTemplateRepository` port + Supabase
adapter (mirroring the 13-02 audit repo), then integrate it into `GenerateUiSpecUseCase` so the cache
CHECK runs as step 0 (before quarantine/generate/audit — D-02) and a validated spec PERSISTS after
generation (D-11). This is where all three success criteria become true: persist (1), zero-Bedrock hit
with no new audit row (2), and lazy registry-version invalidation (3, inherited from the 14-02 key).

Purpose: The cache must sit where the Bedrock calls are (FastAPI/ECS, D-01) so a hit can skip them. The
port/adapter shape, best-effort swallow+log, and `asyncio.to_thread` are copied verbatim from the
audit repo (13-02). The use case stays domain-pure: it imports the new Protocol port + the pure
cache_key helper (from 14-02), never infrastructure.

Output: the port + DTOs, the Supabase adapter (find/persist-upsert/increment), the modified use case
(step-0 check, validated-only persist, `cache_hit` result flag), DI wiring, a thin additive `cache_hit`
endpoint field (for the Phase 15 studio), and tests proving the zero-Bedrock-on-hit + no-new-audit-row
+ never-persist-fallback + registry-version-invalidation guarantees.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Request → GenerateUiSpecUseCase (cache check) | Request-controlled intent/raw_content/importer_id derive the cache_key. A hit returns a stored spec for this caller — must be tenant-scoped (importer in the key) and validity-filtered. |
| Generated spec → ui_spec_templates (persist) | A fallback/invalid spec written here poisons every future identical intent. |
| Stored spec_json → web SpecRenderer | The cached spec is data returned to the web boundary, which re-validates with SpecRootSchema.safeParse (13-04) before render — the cache adds a DB read, not a new execution surface. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-09 | Tampering (cache poisoning) | persisting SAFE_FALLBACK_SPEC / an invalid spec | mitigate | Persist gated on `_determine_outcome(...) != "fallback"` (D-11); test asserts persist NOT called on the fallback path. DB-level validation_status CHECK (14-01) is the second line. |
| T-14-10 | Information Disclosure (cross-tenant leak) | tenant B served tenant A's cached spec | mitigate | importer_id in the cache_key (D-08, proven in 14-02) + find_by_cache_key returns only the exact-key row; RLS deny-all (14-01) blocks direct client access. |
| T-14-11 | Spoofing (stale-serve) | a registry_version bump still serving an old spec | mitigate | registry_version is a key field → new version = new key = miss = cold regen (D-13/CACHE-04); Task 2 invalidation test asserts the generator runs again after a version change. |
