<feature>
  <name>cache_key — deterministic SHA-256 exact-match key + canonicalizers</name>
  <files>apps/email-listener/app/application/use_cases/cache_key.py, apps/email-listener/tests/application/test_cache_key.py</files>
  <behavior>
    canonicalize_intent(intent: str) -> str (D-05):
      - "  Show   Invoice  " → "show invoice" (strip + lower + collapse whitespace)
      - "Show invoice" and "show  Invoice" → SAME output ("show invoice")
      - applies unicodedata.normalize("NFC", ...) BEFORE strip/lower so NFC-equivalent strings canonicalize equal
      - collapses ALL Unicode whitespace runs (\s+) to a single ASCII space

    compute_data_shape_hash(raw_content: str) -> str (D-06):
      - JSON object {"amount": 500, "lines": [1,2]} and {"amount": 9, "lines": [7]} → SAME hash (values excluded; shape {"amount":"number","lines":["number"]})
      - {"a":1,"b":2} and {"b":2,"a":1} → SAME hash (object keys sorted before hashing)
      - opaque non-JSON text "Invoice #123" → the constant "text" shape → stable hash
      - "" (empty) or whitespace-only → the "∅" sentinel shape → stable hash, DISTINCT from "text"
      - mixed array element types collapse to a deduped sorted set of type names (deterministic; Claude's-discretion rule from D-06, applied consistently)
      - recursion is depth-capped (pick a cap, e.g. 8, matching MAX_SPEC_DEPTH spirit); values NEVER enter the hash at any depth

    compute_cache_key(*, intent, raw_content, registry_version, importer_id, catalog_id) -> str (D-04/D-08):
      - returns a 64-char lowercase hex SHA-256 digest
      - deterministic: same args → same key (call twice, assert equal — success criterion 2 "second identical intent → same key")
      - fields joined by the 0x1f unit-separator byte in FIXED order: canonical_intent, data_shape_hash, registry_version, context_descriptor
      - registry_version change → DIFFERENT key (CACHE-04 lazy-invalidation lever)
      - importer_id change → DIFFERENT key (cross-tenant isolation; tenant A's key ≠ tenant B's key for identical intent)
      - importer_id=None → "__system__" sentinel folded into context_descriptor; None and the literal string "__system__" importer must NOT alias only if a real importer could legitimately be "__system__" — assert None maps to the system sentinel deterministically
      - delimiter prevents field-boundary collisions: intent="ab"+shape="c" must NOT equal intent="a"+shape="bc" (assert different keys)
  </behavior>
  <implementation>
    Create cache_key.py importing ONLY stdlib: `hashlib`, `json`, `re`, `unicodedata` (NO infrastructure, NO app.infrastructure, NO supabase — lint-imports must stay clean; this lives in the application layer next to generate_ui_spec.py). Three module-level pure functions with explicit type annotations, named exports (no `__all__` games — plain `def`). Define a module constant `_FIELD_SEP = "\x1f"`. canonicalize_intent: `unicodedata.normalize("NFC", intent)` then `.strip().lower()` then `re.sub(r"\s+", " ", ...)`. compute_data_shape_hash: try `json.loads(raw_content)`; on success recurse a value-free shape (dict→sorted-key map of recursed types; list→deduped sorted list of element type descriptors; scalar→type name like "string"/"number"/"boolean"/"null"); on JSONDecodeError treat as the "text" constant; empty/whitespace-only raw_content → "∅". Serialize the shape descriptor with `json.dumps(shape, sort_keys=True, ensure_ascii=False)` then SHA-256 hex. compute_cache_key: build context_descriptor as `f"{importer_id or '__system__'}|{catalog_id}"`, assemble the 4 fields in fixed order joined by `_FIELD_SEP`, encode UTF-8, `hashlib.sha256(...).hexdigest()`. RED first: write test_cache_key.py asserting every behavior above (it must fail with ImportError/NameError), commit `test(14-02): add failing cache_key tests`; then implement to GREEN, commit `feat(14-02): implement deterministic cache_key module`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Request payload (intent, raw_content, importer_id) → cache_key | The key is derived from request-controlled inputs. A weak key composition could (a) leak a spec across tenants (collision) or (b) let one tenant's intent collide with another's. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-05 | Spoofing/Information Disclosure | cross-tenant cache_key collision (tenant B gets tenant A's spec) | mitigate | importer_id folded into context_descriptor (D-08); unit test asserts tenant A's key ≠ tenant B's key for identical intent + shape. |
| T-14-06 | Tampering | field-boundary collision (concatenation ambiguity producing an unintended cache hit) | mitigate | 0x1f unit-separator delimiter in fixed field order (D-04); unit test asserts the "ab"+"c" vs "a"+"bc" anti-collision case yields different keys. |
| T-14-07 | Information Disclosure | live values entering the hash (value-bearing key would leak data shape/values via timing or stored hash) | mitigate | data_shape_hash is value-free by construction (D-06); unit test asserts identical-shape-different-values → identical hash, proving no value influence. |
| T-14-08 | Tampering (stale-serve) | a registry_version bump NOT changing the key would serve a stale spec | mitigate | registry_version is a key field (D-04/D-13); unit test asserts a version change → a new key (CACHE-04). |
</threat_model>
