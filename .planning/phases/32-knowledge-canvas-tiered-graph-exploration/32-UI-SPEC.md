## Color

### GRAPH-01 — Tier visual encoding (`kne-*` edges only)

Applies exclusively to edges whose `id` is prefixed `kne-` (the `knowledge_node_edges` UNION seam,
carrying `tier` since 30-01 — see `shapeExplicitEdgeRow`). The 8 FK-derived structural edge types
(`has_field`, `instance_of`, `component_linked_to`, `component_type_of`, `belongs_to_email`,
`nested_in`, `scope_of_*`, `retrieval_context`) have no `tier` field and are **completely untouched**
by this phase — same `type: "smoothstep"`, same conditional `markerEnd`, same default stroke.

**Token-only encoding (no raw hex, no new color):**

| Tier | Stroke | Stroke width | Dash | Opacity | Label opacity |
|------|--------|--------------|------|---------|----------------|
| EXTRACTED | `hsl(var(--foreground))` (React Flow default — unchanged) | `1.5` (React Flow default) | none (solid) | `1` | `1` |
| INFERRED | `hsl(var(--muted-foreground))` | `1.5` | `strokeDasharray: "5 3"` | `1` | `1` |
| AMBIGUOUS | `hsl(var(--muted-foreground))` | `1.5` | none (solid) | `0.45` | `0.6` |

Rationale: EXTRACTED keeps React Flow's existing default stroke (`--foreground`-derived) — this is
the "reusing the existing edge-style conventions" instruction from GRAPH-01, i.e. do not invent a new
"confirmed" color, the CURRENT look already means confirmed/default. INFERRED differentiates via
`strokeDasharray` only (shape, not color) — this repo's `--muted-foreground` token, already used
everywhere for de-emphasized text, is reused here for de-emphasized edges too (no new token).
AMBIGUOUS uses the SAME `--muted-foreground` stroke as INFERRED but drops opacity to `0.45` (faint) —
distinguishing "faint" (AMBIGUOUS) from "dashed" (INFERRED) purely by opacity+dash-shape, never by a
second hue, keeping the 60/30/10 discipline (edges are structural chrome, not the 10% accent budget).

**Arrowhead convention preserved:** taxonomy edges (`TAXONOMY_RELATION_TYPES` = `has_field`) have no
arrowhead — untouched, this set is never `kne-*` prefixed anyway (has_field is FK-derived). All
`kne-*` edges keep their existing `markerEnd: MarkerType.ArrowClosed` (unaffected by tier).

**Implementation shape (`toFlowEdges`, `apps/web/src/app/knowledge/_components/knowledge-graph.tsx`):**

```tsx
function tierEdgeStyle(tier: string | undefined): {
  style?: React.CSSProperties;
  labelStyle?: React.CSSProperties;
} {
  if (tier === "INFERRED") {
    return { style: { strokeDasharray: "5 3", stroke: "hsl(var(--muted-foreground))" } };
  }
  if (tier === "AMBIGUOUS") {
    return {
      style: { stroke: "hsl(var(--muted-foreground))", opacity: 0.45 },
      labelStyle: { opacity: 0.6 },
    };
  }
  // EXTRACTED or undefined (structural edges) — no override, React Flow default
  return {};
}
```

Merged into each edge's returned object in `toFlowEdges` only when `ge.id.startsWith("kne-")` — do
not apply `tierEdgeStyle` to structural edges even if a caller accidentally passes a `tier` field
(defensive: `GraphEdge.tier` stays `undefined` on structural rows today, so this guard is currently
redundant but documents intent for future edge-type additions).

## Interaction Contracts

### GRAPH-02 — Bounded click-expand

- **Trigger:** clicking a knowledge-graph node (any `type` currently rendered by `nodeTypes`) fires
  `knowledge.expandNode({ nodeId, depth: 1 })` via tRPC (`api.knowledge.expandNode.useMutation` or a
  lazy query — planner's call; `useMutation` is more idiomatic since it's an explicit user-triggered
  side-effecting-but-read-only fetch, mirroring how `useQuery` is reserved for the initial graph load).
  This does NOT replace the existing `handleNodeClick` selection behavior (still sets
  `selectedNodeId`) — expand and select happen on the SAME click, both effects fire.
- **Depth:** default `1` (one hop) per click, clamped server-side to `≤2` total. Re-clicking an
  already-expanded node re-issues the same request (idempotent merge, per CONTEXT) rather than
  compounding depth — this phase does NOT expose a depth selector in the UI (Claude's discretion,
  documented): one-hop-per-click keeps the interaction predictable and avoids an unbounded "expand the
  whole graph" footgun via repeated clicks past the server's hard node/edge budget cap (~50).
- **Loading state:** while `expandNode` is in flight, the clicked node gets a subtle pulse — reuse
  the existing `ring-2 ring-primary` selection ring (already applied via `selected: true` in
  `displayedNodes`) with an ADDED `animate-pulse` class scoped to the node's outer wrapper for the
  mutation's duration only (no new spinner icon on the canvas — keeps the node shape untouched, mirrors
  the "skeleton over spinner" product-register guidance where feasible, but a raw pulse on an existing
  selection ring is the pragmatic minimal treatment for a graph-node async state).
- **Merge:** on success, new nodes/edges are merged into `nodes`/`edges` state via `setNodes`/
  `setEdges` functional updates, deduping by `id` (a node/edge already present is not duplicated or
  repositioned) — new nodes are placed via the SAME `layoutGraph` (dagre) helper already used for the
  initial load, re-run over the UNION of old+new nodes/edges (not a fresh from-scratch layout that
  would jarringly reposition already-visible nodes — planner should confirm dagre's positional
  stability or, if it repositions everything, only lay out the NEW nodes near their expand-anchor and
  leave existing node positions untouched; either is acceptable, documented as an implementation
  choice, not a visual regression).
- **Budget-exceeded state:** if the server's ~50-node/edge cap truncates the expansion, show a toast
  ("Showing the first 50 related items — narrow the tier filter to see more.") rather than silently
  dropping data — ties into GRAPH-03's filter as the stated mitigation path.
- **No new node type:** expanded nodes render via the EXISTING `nodeTypes` registry
  (`graph-nodes.tsx`) — an expanded `knowledge_node` looks identical to one loaded on initial mount.

### GRAPH-03 — Tier filter (segmented control)

- **Placement:** lives in the existing toolbar row (`graph-toolbar.tsx`), inserted between the page
  title and the right-aligned action group — OR as a new row directly below the toolbar if horizontal
  space is tight at the toolbar's current `h-11`; planner's call based on live measurement, but the
  control must NOT go inside `FilterRail` (that rail is node-type filtering, a distinct concern from
  edge-tier filtering per CONTEXT's explicit framing as "the budget-prune analog").
- **Three segments, cumulative (not exclusive) semantics:** "Confirmed only" (EXTRACTED) →
  "+ Inferred" (EXTRACTED + INFERRED) → "+ Ambiguous" (all three) — a single-select segmented
  control where each option's LABEL communicates the cumulative set, matching CONTEXT's literal
  wording. Implemented as a 3-button group (`role="radiogroup"`, each button `role="radio"
  aria-checked`), NOT a `Tabs` primitive (tabs imply mutually exclusive content panes, not a filter
  breadth toggle) — reuse `Button` primitive with `variant="outline"`/active-state classes mirroring
  `filter-rail.tsx`'s checkbox active/inactive token pair (see Color section above).
- **Default:** all three shown (widest state, "+ Ambiguous" selected) — rationale: GRAPH-03's own
  wording frames the control as narrow→wide, and the design-case demo path (`32-CONTEXT.md`
  "Specific Ideas") explicitly walks confirm→see-suggestions-appear→filter-to-confirmed-only —
  suggestions must be VISIBLE by default for that demo path to work; defaulting to hidden would break
  the very story this phase exists to enable. Documented as Claude's Discretion per CONTEXT.
  Structural (non-`kne-*`) edges are unaffected by every filter state — always visible.
- **Filtering mechanic:** reuses the EXISTING both-endpoints-visible edge filter
  (`knowledge-graph.tsx` lines ~254-266) — filtering happens at the EDGE level (an edge whose tier is
  excluded by the current filter state is dropped from `filteredEdges`, exactly like today's
  visible-node-endpoint check), so orphaned nodes degrade exactly as today (a node with only
  filtered-out edges still renders, just with no visible relationship lines — consistent, no special
  new "hide orphan nodes" behavior invented).
- **Persistence:** filter state is local `useState` (like `visibleTypes`/`showInstances`) — NOT
  persisted to `localStorage` (unlike the taxonomy banner). No CONTEXT decision requires persistence;
  keeping it session-only avoids a new storage key for a control this minimal.

### Promote affordance (Phase-30 closure)

- **Trigger:** clicking a suggestion-tier (`tier === "INFERRED" || tier === "AMBIGUOUS"`) `kne-*` edge
  (its label or the edge line itself — React Flow edges support `onEdgeClick`) opens a `Popover`
  anchored near the click point (`PopoverAnchor` positioned at the click coordinates, or a simpler
  `PopoverTrigger`-less controlled-open pattern keyed to a `selectedEdgeId` state — planner's call).
  EXTRACTED edges are NOT clickable for this affordance (no popover, no cursor change) — nothing to
  promote.
- **Popover content (fixed order, mirrors `node-detail-pane.tsx`'s per-type block shape):**
  1. Header: `text-sm font-semibold` — "Suggested relationship"
  2. Relation-type row: label "Relation" / value = `edge.relationType` (plain text, e.g.
     `co_occurs_with`)
  3. Tier badge row: label "Tier" / `Badge` styled per the Color section above (INFERRED/AMBIGUOUS)
  4. Confidence row: label "Confidence" / `{Math.round(confidence * 100)}%` — reuses
     `node-detail-pane.tsx`'s existing confidence-formatting convention verbatim
  5. Provenance summary row: label "Source" / a short plain-text summary (e.g. "Synthesized from
     region confirmation" or the token-provenance's human-readable descriptor) — exact string sourced
     from whatever `provenance` field the data-layer prerequisite (see Files table) exposes; if that
     field is unavailable at execution time, this row is OMITTED entirely (never shows "undefined" or
     a raw JSON blob — T-11-05-style plain-text-only discipline)
  6. Divider (`Separator`, matches `node-detail-pane.tsx`)
  7. "Promote to confirmed" button — `variant="default"`, full-width inside the popover, `Check` icon
     leading, `Loader2 animate-spin` swap while the mutation is in flight (existing app-wide async
     button convention — same shape as every other mutation-triggering button in this codebase)
- **On success:** popover closes; the edge's local state is patched to `tier: "EXTRACTED"`
  (optimistic-safe since the server call already succeeded) so it IMMEDIATELY re-renders with solid/
  full-opacity styling — no full graph refetch required, though a refetch is also acceptable if
  simpler to implement correctly.
- **On 4xx (409 not-promotable / 404 not-found):** popover stays open; `sonner` error toast:
  "Couldn't promote — {reason}" — generic reason text only (backend intentionally returns a generic
  409 detail, never a raw exception; do not attempt to surface `exc.reason`, it is server-log-only by
  design, see `knowledge_edges.py`).
- **Scope discipline (hard ceiling from CONTEXT):** ONE popover, ONE button, error toast on 4xx — no
  review queue, no bulk operations, no dismiss/deactivate action (Phase 30 explicitly did not ship
  that endpoint). If any of the above balloons the implementation, the planner may cut the Promote
  button and ship edge-detail-only (rows 1-6), logging the button as a deferred item — GRAPH-01..03
  remain the phase's actual requirements; this affordance is explicitly discretionary/closable scope.

## Legend (Claude's Discretion — small, included)

A compact legend clarifies the three-tier line encoding without requiring the reviewer to guess from
dash/opacity alone (impeccable's "no information by color alone" posture, extended here to "no
information by line-style alone" for an unfamiliar visual language). Placement: bottom-left corner of
the canvas, as a small `MiniMap`-adjacent overlay (React Flow supports arbitrary `Panel`
children) — NOT a modal, NOT a persistent rail (would compete with `FilterRail`'s existing 18% zone).

```tsx
<Panel position="bottom-left" className="rounded-md border border-border bg-background/90 px-3 py-2 text-xs text-muted-foreground">
  <div className="flex items-center gap-3">
    <LegendSwatch label="Confirmed" style="solid" />
    <LegendSwatch label="Suggested" style="dashed" />
    <LegendSwatch label="Uncertain" style="faint" />
  </div>
</Panel>
```

Each `LegendSwatch` renders a tiny `<svg>` line (12px wide) using the exact same
`stroke`/`strokeDasharray`/`opacity` values as `tierEdgeStyle` above (single source of truth — do not
hand-duplicate the values; either import the same constants or accept minor duplication with a code
comment cross-referencing `toFlowEdges`). Copy: "Confirmed" / "Suggested" / "Uncertain" — plain,
reviewer-facing language, NOT the internal enum names (`EXTRACTED`/`INFERRED`/`AMBIGUOUS` stay
internal/technical; the UI never shows raw enum strings to a reviewer, consistent with this app's
existing "Instance"/"Component"/"Field" Badge labels in `node-detail-pane.tsx` being plain nouns, not
`entity_instance`/`email_component` raw type strings).

No dismiss control on the legend (unlike `TaxonomyBanner`) — it's a small persistent reference, not an
onboarding banner; always visible, no localStorage key needed.

## Accessibility

- **Tier filter control:** `role="radiogroup"` with `aria-label="Filter by trust tier"`; each segment
  `role="radio" aria-checked`; keyboard arrow-key navigation between segments (standard radiogroup
  pattern) — mirrors the existing `FilterRail` checkbox rows' `aria-label` discipline.
- **Edge click target:** React Flow edge labels/paths have a known small hit-target problem — ensure
  the clickable region includes a reasonable stroke-width hit-slop (React Flow's default
  `interactionWidth` already provides this; do not shrink it below the library default for tier-styled
  edges).
- **Popover:** Radix `Popover` already provides focus trap + `Escape`-to-close + `aria-expanded` on
  the trigger — no additional wiring needed beyond correct trigger/content composition. Promote button
  gets a real `disabled` state (not just a visual dim) while the mutation is in flight, preventing
  double-submit — mirrors the CAS-guarded backend, defense in depth on the client.
- **Contrast:** AMBIGUOUS edges at `opacity: 0.45` against the canvas background (`Background gap={16}
  size={1}`, effectively near-transparent dot grid over `--background`) — this is a GRAPHICAL element
  (a line), not text, so the strict 4.5:1 text gate does not apply; the existing 3:1 non-text
  guideline is the relevant bar. `--muted-foreground` at full opacity already clears text-contrast
  minimums against `--background` (established in prior UI-SPECs); at 0.45 opacity the effective
  contrast drops but the edge remains a SECONDARY visual cue (the tier filter + legend + popover all
  provide non-color-dependent ways to identify tier), so a "faint but not invisible" AMBIGUOUS line is
  the intended, accepted design — reviewers are never required to distinguish AMBIGUOUS from
  background by contrast alone to use the feature (filter + click-to-inspect always available).
- **Reduced motion:** the expand-in-flight pulse uses Tailwind's `animate-pulse` — NOT currently gated
  by `motion-reduce:`; per this phase's own discipline (see TOKEN-05 precedent in Phase 28), add
  `motion-reduce:animate-none` to the pulse class so it degrades to a static (non-animated) ring under
  `prefers-reduced-motion: reduce`, consistent with every other entrance/loading animation in this
  app.
- **Security:** no `dangerouslySetInnerHTML` anywhere in this contract; every DB-origin string
  (`relationType`, provenance summary, error `reason`) renders as plain escaped React text, mirroring
  `node-detail-pane.tsx`'s T-11-05 discipline. `expandNode`'s input is Zod-validated at the boundary
  (`nodeId: z.string().uuid()`, `depth` clamped server-side) — never trust a client-supplied depth
  beyond the ≤2 ceiling.
