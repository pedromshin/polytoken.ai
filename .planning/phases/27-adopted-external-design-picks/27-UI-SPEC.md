# Phase 27 — UI Design Contract

> Visual and interaction contract for landing the 5 researched external resources' narrowly-scoped
> takeaways (ADOPT-01..05) in the app and its docs. Generated by gsd-ui-researcher (fully autonomous
> session — no user prompts; every choice derives from `27-CONTEXT.md`'s locked `[auto]` decisions,
> `.planning/research/CHAT-STUDIO-DESIGN-UPLIFT.md`'s locked verdicts, `26-UI-SPEC.md`'s shipped
> baseline, and a fresh read of every consumer file this contract touches). Verified by
> gsd-ui-checker.

**This is an additive phase, not a correction phase.** Every new class/keyframe/utility this
document introduces is genuinely NEW surface area (a ported component, a CSS technique, a
documentation file) — it does not re-litigate anything Phase 26 already fixed. **No token VALUE is
added, changed, or removed** anywhere in this document (`:root`/`.dark` stay exactly as Phase 26 left
them — Phase 28's job). Where new CSS custom properties would normally be tempting (motion
durations/easings), this contract deliberately uses **literal values instead**, per the hard
constraint — only EXISTING color tokens (`--primary`, `--border`, `--foreground`, etc.) are ever
referenced via `hsl(var(--x))`. Zero new npm dependencies — `@radix-ui/react-accordion` and
`lucide-react` (FileTree's only two runtime dependencies) are both already installed.

**Non-interference (unchanged from Phase 24/26):** `packages/genui/src/renderer/spec-renderer.tsx`
is UNMODIFIED. `GenuiPartBoundary`/`InteractiveWidgetBoundary`'s own internal chrome is NOT edited —
ADOPT-03's chat-side ring wraps `GenuiPartBoundary` from its caller (`message-turn.tsx`), never
reaching inside it.

**Design docs convention (established by ADOPT-01, for every future phase onward):** this phase
creates `docs/design/` — the repo's first standing design-reference directory. From this phase
forward, `gsd-ui-researcher` and `gsd-ui-checker` should read `docs/design/product-register-and-bans.md`
(product-register rules + 13-item absolute-bans checklist + the available transition utilities) as
background context on future UI phases, the same way they already read `UI-SPEC.md`/6-pillar
history. Nothing enforces this mechanically yet — it is a documented convention, not a hook.

### ADOPT-02 — FileTree (ported Magic UI component)

**New file:** `apps/web/src/components/file-tree.tsx` — attribution header required (Magic UI,
`magicuidesign/magicui`, MIT, fetch date). **Zero new deps** — only `@radix-ui/react-accordion`

**Critical implementation constraint:** build directly on `AccordionPrimitive` from
`@radix-ui/react-accordion` — **do NOT import `Accordion`/`AccordionTrigger` from `@nauta/ui`**. That

and zero inherited weight.

export interface FileTreeNode {

export interface FileTreeProps {

### ADOPT-03 — `<GeneratingRing>` (hand-ported Magic UI CSS technique)

**New files:** `apps/web/src/components/generating-ring.tsx` + additive CSS in
`apps/web/src/app/globals.css`. Attribution header required (Magic UI's `shine-border` +
`animated-shiny-text`, MIT, fetch date) on both.

**CSS technique — pure `background-position` keyframe sweep, zero JS, teal-only:**

```css
/* ADOPT-03: <GeneratingRing> — hand-ported CSS TECHNIQUE from Magic UI's
   shine-border + animated-shiny-text (MIT) — a background-position sweep,
   not shine-border's own component API. Teal-only via var(--primary). This
   custom utility is NOT registered as a Tailwind theme animation (it lives
   in plain globals.css, per 27-CONTEXT.md), so Tailwind's `motion-safe:`
   variant cannot target it via a class-name prefix — the
   prefers-reduced-motion query below achieves the identical contract by
   hand, directly in this stylesheet. */
@layer utilities {
  .generating-ring {
    position: relative;
    isolation: isolate;
  }
  .generating-ring::before {
    content: "";
    position: absolute;
    inset: -2px;
    border-radius: inherit;
    padding: 2px;
    background-image: linear-gradient(
      90deg,
      transparent 0%,
      hsl(var(--primary) / 0.85) 45%,
      hsl(var(--primary)) 50%,
      hsl(var(--primary) / 0.85) 55%,
      transparent 100%
    );
    background-size: 250% 100%;
    background-position: 50% 0%;
    -webkit-mask: linear-gradient(#000 0 0) content-box, linear-gradient(#000 0 0);
    -webkit-mask-composite: xor;
            mask-composite: exclude;
    pointer-events: none;
  }
  @keyframes generating-ring-sweep {
    from { background-position: 0% 0%; }
    to   { background-position: 200% 0%; }
  }
  @media (prefers-reduced-motion: no-preference) {
    .generating-ring::before {
      animation: generating-ring-sweep 2000ms linear infinite;
    }
  }
}
```

Ring geometry: 2px thickness (`inset:-2px` + `padding:2px`, mirrors Phase 26 FIX-04's already-approved
`border-l-2` precedent), `border-radius: inherit` (**consumer must set its own explicit rounding on
the SAME element receiving the `.generating-ring` class** — e.g. `rounded-lg`), mask-composite
`exclude` renders ONLY the border band (a true ring, never a filled glow-overlay). Under reduced
motion, the gradient simply stays static at `background-position: 50% 0%` (a fixed highlight, no
motion) — no separate override rule needed since the animation itself is gated by the
`no-preference` media query.

**Component:**

```tsx
export interface GeneratingRingProps {
  readonly active: boolean;
  readonly className?: string;
  readonly children: React.ReactNode;
}

export function GeneratingRing({ active, className, children }: GeneratingRingProps): React.ReactElement {
  return <div className={cn(active && "generating-ring", className)}>{children}</div>;
}
```

**Mount (a) — Studio, `generation-sandbox-island.tsx`:** wrap the existing `#sandbox-output-region`
div with `<GeneratingRing active={chromeProps.isPending} className="rounded-lg">`. `q.isFetching`
(the SAME signal `GenerationStateChrome` already derives `in_progress` from) drives `active`.
`GenerationStateChrome`'s own JSX/label is UNCHANGED — the honest "Generating…" text (D-02) stays
exactly as Phase 26 left it; the ring is a purely additional visual signal around the render pane
below it, not a replacement for the label.

**Mount (b) — Chat, `message-turn.tsx`:** wrap ONLY the `isStreaming: true` `GenuiPartBoundary` call
sites — the `genui_spec_streaming` and `interactive_widget_streaming` part branches — with
`<GeneratingRing active className="rounded-lg">`. The finalized `genui_spec` case
(`isStreaming={false}`) is NOT wrapped (nothing is generating once finalized).
`GenuiPartBoundary`/`InteractiveWidgetBoundary` themselves are UNTOUCHED — the ring wraps from
`message-turn.tsx` (the caller), never edits either boundary file.

**Mount (c) — Studio history tab: explicitly NOT mounted (scout finding).** `history-island.tsx`'s
master list (`historyList.useQuery`) only exposes `isLoading` for the WHOLE list fetch — rows
themselves are, by definition, PAST/completed generations with no per-row in-flight state to ring.
`27-CONTEXT.md` hedged this mount as "if trivially reachable" — it is not; this is a reasoned scope
exclusion, not a gap. ADOPT-03's success criterion ("sandbox/history tabs in Studio") is satisfied by
mount (a) — Generation Sandbox is the "sandbox" tab referenced there.

**Accessibility:** `GeneratingRing` is purely decorative (`pointer-events: none` on the pseudo-element,
no ARIA role on the wrapper div) — it must never be the ONLY signal that generation is in progress.
Both mounts already carry a real accessible signal independent of the ring: (a) `GenerationStateChrome`'s
`aria-live="polite"` + "Generating…" text; (b) the existing streaming caret / skeleton bars inside
`GenuiPartBoundary`. Reduced motion freezes the sweep to a static highlight (never fully invisible,
never a flashing/strobing artifact).

## Interaction Contracts

**FileTree (ADOPT-02):** folder rows toggle expand/collapse via native Radix Accordion trigger
behavior (click or Enter/Space when focused); file rows are plain buttons that call `onSelect` —
selecting a file in Plan A calls the SAME `handlePreset` callback already wired to the `<Select>` it
replaces (never auto-runs — "Run preset" stays the separate manual trigger, preserving this
codebase's D-06 manual-trigger-only convention).

**GeneratingRing (ADOPT-03):** purely presentational, no click handler, no keyboard interaction —
`active` is driven exclusively by the SAME boolean signal each mount already computes
(`chromeProps.isPending` / `isStreaming`), never a new state variable.

**Transition snippets (ADOPT-05):** none change WHICH elements are interactive or their existing
handlers — only how the wrapped content's mount/collapse animates. `conversation-rail.tsx`'s
`onCollapsedChange` contract, `delete-conversation-dialog.tsx`'s `onConfirm`/`onOpenChange`, and
`model-picker.tsx`'s `onSelect` are all UNCHANGED.

## Accessibility

- **FileTree:** 44px (`min-h-11`) touch targets on every row; folder triggers get Radix's automatic
  `aria-expanded`; file rows' visible text is their accessible name (no separate `aria-label` needed).
- **GeneratingRing:** decorative-only (`pointer-events: none`, no ARIA role) — never the sole signal
  of an in-progress state; both mounts retain their existing independent accessible signal
  (`aria-live="polite"` region / streaming skeleton).
- **Reduced motion:** `GeneratingRing`'s sweep is gated by `@media (prefers-reduced-motion:
  no-preference)` (animates only when motion is NOT reduced); all 3 ADOPT-05 snippets explicitly
  disable their animation/transition under `@media (prefers-reduced-motion: reduce)`.
- **Color contrast:** every new color reference (`hsl(var(--primary))`, `hsl(var(--foreground) /
  0.12)`, `hsl(var(--border))`) reuses already-shipped, already-contrast-verified tokens — no new
  color pairing requires a fresh contrast check.
- **Security:** no `eval`/`Function`/`dangerouslySetInnerHTML` anywhere in any ADOPT-01..05 file —
  FileTree renders from a typed data prop, `GeneratingRing`/transition snippets are pure CSS.
