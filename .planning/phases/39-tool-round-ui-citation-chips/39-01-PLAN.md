<objective>
Make the chat agent's already-shipped mid-turn server-tool round loop (Phase 34's LOOP-01, now wired
to real tools by Phases 36-38) visible over SSE as its own, correctly-typed transport frames — closing
half of TUI-01 (ROADMAP Phase 39 SC1/SC2). `_run_server_tool_round` already persists `tool_call`/
`tool_result` run events and already streams them over SSE today, but the WEB client's existing
`applyRunEvent` reducer (built for the emit_ui_spec/interactive-widget streaming case, keyed on a
`partial_json` field these events never carry) mis-folds a real server-tool round's `tool_call` event
into a permanently-stuck, empty `interactive_widget_streaming` skeleton part — 39-UI-SPEC.md's
"SSE / Part Contract" section names this a real, already-live bug. This plan adds two NEW,
non-persisted, mirrored SSE frame types (`server_tool_call`/`server_tool_result`) that the sibling web
plan (39-02) folds into the correct `tool_invocation_streaming`/`tool_invocation_result` parts — the
existing persisted `tool_call`/`tool_result` events, the DB schema, and the CHECK constraint are all
left completely untouched (no migration).

Purpose: gives 39-02's `ToolRoundActivityRow`/`ToolInvocationResultRow` something correct to render,
without touching the durable Phase-34 persistence contract chat.getHistory replay depends on.
Output: `_run_server_tool_round` yields 2 additional `ChatRunEvent`s per round; `ChatRunEventType` is
widened additively to type-check them.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `_run_server_tool_round` -> SSE wire | The 2 new mirror events carry tool-execution metadata (tool name, id, result content/error flag) directly onto the client-visible transport, same trust level as the already-persisted `tool_call`/`tool_result` events |
| Client (browser) -> SSE stream | One-way, server-initiated SSE — the client cannot inject or replay frames into this stream |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-39-01 | Information Disclosure | `server_tool_call` mirror event exposing model-authored tool arguments to a browser-visible transport | mitigate | `data` shape for `server_tool_call` is deliberately `{"tool_name", "id"}` only — `arguments` is NEVER included (per 39-UI-SPEC.md: "the activity row never displays them"), narrower than the already-persisted `tool_call` event which does carry `arguments` |
| T-39-02 | Tampering | A compromised/malicious client injecting a fake SSE frame | accept | SSE is a one-way, server-initiated stream (`StreamingResponse` over `text/event-stream`) — there is no client-to-server frame-injection surface; this is the pre-existing SSE trust model, unchanged by this plan |
| T-39-03 | Repudiation / Denial of Service | A client that misses a `server_tool_call`/`server_tool_result` frame (e.g. a dropped chunk) sees no retry, since these are non-persisted | accept | Inherent to a non-critical UI-affordance mirror frame — losing one never corrupts turn state, since the correctness-critical persisted `tool_call`/`tool_result` path (used for `chat.getHistory` replay) is completely untouched by this plan |

</threat_model>
