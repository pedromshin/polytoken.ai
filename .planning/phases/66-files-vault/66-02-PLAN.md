<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → tRPC procedure | Authenticated but hostile: the caller may send extra fields, other users' ids, crafted paths. |
| procedure → service-role client | Service-role credentials can address EVERY user's objects. `ctx.user.id` + `vaultKey` are the only things preventing it. |
| process → env | `SUPABASE_SERVICE_ROLE_KEY` must never reach a response body or a log line. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-66-02 | Elevation | every procedure | mitigate | `ctx.user.id` is the only userId source; no input names a user/key/bucket/prefix; zod strips unknown keys. Proven by the impersonation test + the `files-inputs` source gate (two independent gates). |
| T-66-03 | Information disclosure | `requestDownload` / `requestUpload` | mitigate | Signed URL scoped to one `vaultKey` under `{userId}/`, 60s expiry, attachment disposition (Plan 01); the response carries `{ url }` and nothing else. |
| T-66-04 | Spoofing | every procedure | mitigate | `protectedProcedure` → UNAUTHORIZED on null user; one written-out test per procedure, plus a source gate banning `publicProcedure` under `router/files/`. |
| T-66-05 | DoS | `requestUpload` | mitigate | `size` capped server-side at `VAULT_MAX_UPLOAD_BYTES`; the bucket's `fileSizeLimit` is the layer the client cannot lie past. |
| T-66-07 | Information disclosure | error mapping, `service-client.ts` | mitigate | `VaultStorageError` → generic TRPCError, real detail logged server-side only; missing-secret guard returns "Storage is not configured" without naming which secret; asserted by the storage-failure test. |
| T-66-10 | Repudiation | vault mutations | accept | No audit log tonight. Single-user self-cloud; storage keeps `created_at`/`updated_at`. Backlog if the vault ever gains sharing — at which point it is not optional. |
