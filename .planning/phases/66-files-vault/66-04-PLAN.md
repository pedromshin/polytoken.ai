    `use-vault-drop.ts` — pane-level drag state. Hand-rolled, ~30 lines, zero dependencies (D-66-09:
    `packages/ui`'s `Dropzone` is card-shaped and its drag-active state is a stock accent — and it is
    not ours to fix). Returns `{ isDragging, dropProps }`.
    - `onDragEnter`/`onDragLeave` with a COUNTER, not a boolean: dragleave fires when the pointer
      crosses onto a CHILD element, so a boolean makes the accept state strobe as the user moves over
      the rows. This is the bug every hand-rolled dropzone ships first.
    - `onDragOver` → `preventDefault()` (without it, drop never fires at all).
    - Only engage when the drag actually carries files: `e.dataTransfer.types.includes("Files")`.
      Dragging selected text across the vault must not make the sheet rise.
    - **Window-level guard:** attach `dragover`/`drop` `preventDefault` on `window` while mounted, so
      a near-miss drop does not make the browser NAVIGATE AWAY to the file — losing the user's page
      (and any in-flight upload) because they missed the pane by 10px.
    - Reset the counter on `drop` and on unmount. A stuck counter leaves the sheet permanently risen.

    `use-vault-upload.ts` — the queue. `useVaultUpload({ path })` → `{ uploads, start(files), cancel(id) }`,
    where `uploads: { id, name, progress, status: "pending"|"uploading"|"done"|"error", error?: string }[]`.
    - per file: client pre-check `size <= VAULT_MAX_UPLOAD_BYTES` (§B — import it) and name through
      `VaultNameSchema`; a reject never reaches the network and shows the reason. This is a courtesy
      to the user, NOT the control — the server enforces both regardless (T-66-05).
    - `requestUpload` → PUT via XHR per §E, with `upload.onprogress` → `progress`, `xhr.abort()` →
      `cancel`. Concurrency limit 3 (a 50-file drop must not open 50 sockets).
    - on all-settled: `vaultApi.useUtils().files.list.invalidate({ path })` so the rows refresh. Only
      once, after the batch — invalidating per file re-renders the list N times.
    - **the browser never chooses a key.** It sends `{ path, name, size }` and receives a URL. If you
      find yourself passing a key from the client, stop — that is T-66-02 and Plan 02 exists to
      prevent it.
    - a file dropped while `path=["a"]` uploads into `a/` — capture the path at start, so a user who
      navigates mid-upload does not have their file land in whatever folder they ended up in.
    - **directories:** a dropped folder yields an entry with no real file body. Detect and skip it
      with a toast "Folders can't be uploaded yet — drop files instead." (Folder upload is OUT.)
      Without this, dropping a folder does nothing at all and the vault looks broken.

    `vault-drop-layer.tsx` — §D's signature, and nothing more. It wraps the pane's children; it is
    NOT an overlay that covers them. Classes literal (§C), `aria-hidden` on the copy line (the state
    is announced by the tray, not by decorative text).

    `upload-tray.tsx` — per-file rows, docked at the pane's bottom, `bg-bright border-rule`. Each:

    swept treatment — `border-rule` + `text-ink` + `role="alert"` + "Upload failed — {reason}." +
    "Retry" — and **NEVER madder** (D-66-05: an upload failure is a STATUS; madder is only for the
    irreversible). Done rows auto-dismiss after ~4s; error rows persist until dismissed — an error
    that vanishes on a timer is an error the user never read.

    `vault-surface.tsx` — mount `useVaultDrop` on the PANE (the comment Plan 03 left marks the spot),

    (tell #4). The hidden input's `onChange` shares the exact same `start(files)` path as the drop:
    one ingest funnel, two doors.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| dropped file → the vault | File contents AND names are fully attacker-controlled. Nothing here interprets bytes. |
| browser → signed upload URL | The browser PUTs directly to storage. The URL scopes what it can write; the client never picks the key. |
| DataTransfer → the app | A drag can carry text, links, files, or directories — from any origin. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-66-05 | DoS | `use-vault-upload.ts` | mitigate | Client pre-check against `VAULT_MAX_UPLOAD_BYTES` (imported, not re-declared) + concurrency limit 3. The real enforcement is the server's `.max()` and the bucket's `fileSizeLimit` — the client check is UX and is documented as such. |
| T-66-06 | Tampering (XSS) | whole surface | mitigate | No `dangerouslySetInnerHTML` (law gate, Task 3); no inline preview (D-66-04); uploaded bytes are never rendered, only listed by name as escaped text. Downloads are attachment-disposition for ALL types (Plan 01). |
| T-66-13 | Elevation | upload path | mitigate | The client sends `{ path, name, size }` and receives a URL; the key is derived server-side from `ctx.user.id` (Plan 02). Gated by the write test asserting `requestUpload` receives no key/userId field. |
| T-66-14 | Tampering | window drop handler | mitigate | Window-level `preventDefault` on dragover/drop stops the browser navigating to a dropped file (which would destroy in-flight uploads and the page). Handlers detach on unmount — a leaked global listener would swallow drops on every other surface. |
| T-66-15 | Repudiation | delete | accept | No trash, no audit log (both OUT). The confirm dialog IS the mitigation, and its copy states the consequence — including the recursive one for folders. If sharing ever ships, this becomes unacceptable and needs a real answer. |
</threat_model>
