**The single most important thing this plan gets right (RESEARCH Landmine 2):** the WRITE-TIME
cross-tenant ownership check. A caller must not be able to wire a foreign user's knowledge node, ledger
row, genui panel, or email thread into their own conversation's context — that would inject another
tenant's private content into this caller's prompt at turn time (56-04). This is validated at
`createContextEdge`, NOT deferred to read time (the Python read path in 56-04 has no per-request user
identity to re-verify against — Landmine 2). An adversarial two-user test in the pattern of the v1.7
Phase 44 suites is a first-class deliverable, not an afterthought.

## Phase Decision D-56-A (tier-gating, resolving RESEARCH Assumption A3)
An explicit user-drawn context edge to a `knowledge_node` may be created (and later injected) REGARDLESS
of the node's trust tier (INFERRED / AMBIGUOUS / EXTRACTED). Rationale: drawing an edge is a deliberate,
addressed user action — it IS the "selection" that suggest-only requires — and is categorically distinct
from `list_injectable_edges`'s EXTRACTED-only gate, which governs AUTOMATIC/blind injection into every
prompt. `createContextEdge` therefore performs an OWNERSHIP check only, never a tier check. This keeps the
feature useful (INFERRED is the default tier for everything freshly captured) and stays consistent with
suggest-only (the edge exists only because the user drew it). Recorded here and mirrored in 56-04.

Output: the ownership dispatcher in ownership.ts, the new router file, its registration, and a test file
whose adversarial coverage is the acceptance bar. No apps/web work; no api-client dist rebuild needed this
phase (Phase 63 is the first apps/web consumer — 54-04 gotcha).
