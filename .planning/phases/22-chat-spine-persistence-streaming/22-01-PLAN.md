<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Supabase anon/authenticated role → chat tables | A direct Supabase client (bypassing the backend) could read/write chat rows if RLS is absent |
| conversation delete → ledger rows | Hard delete must not erase accounting data (D-14) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-22-01 | Information disclosure | chat_conversations/messages readable by anon | mitigate | RESTRICTIVE deny-all RLS for anon + authenticated on all five tables (0020 pattern); only postgres/service_role (the tRPC + Python backends) reach rows |
| T-22-02 | Tampering | forged chat rows via direct client write | mitigate | same deny-all RLS `WITH CHECK (false)` blocks direct inserts/updates |
| T-22-03 | Repudiation | run events mutated/deleted after the fact | mitigate | chat_run_events is append-only by app convention; unique (run_id, seq); no UPDATE/DELETE paths planned |
| T-22-04 | Tampering | conversation delete wipes billing/audit rows | accept→mitigate | ledger FK is ON DELETE SET NULL, not cascade — accounting rows persist with a null conversation_id (D-14) |
| T-22-05 | Tampering | outcome/status/locus set to out-of-set values | mitigate | SQL CHECK constraints on role/status/type/execution_locus restrict to the enumerated sets |
</threat_model>
