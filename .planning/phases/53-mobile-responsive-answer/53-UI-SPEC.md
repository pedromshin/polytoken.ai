> pass, not a redesign.** It consumes the Phase 48 token system, the Phase 51 re-skin, and

**This phase's central exception — the `.touch-target` floor guard becomes load-bearing
for the first time** (`globals.css`, D-48-07: `min-height: 44px; min-width: 44px`,
breakpoint-static by design). Two distinct application mechanisms, chosen per Judgment
Call #1 to resolve an apparent contradiction (see below):

1. **Unconditional** — any NEW mobile-only chrome this phase adds (back buttons, the
   global mobile nav trigger, filter-chip toggles, sheet close affordances) is built
   `.touch-target`-compliant from the start (`size-11` / `min-h-11 min-w-11`), no variant
   needed — it only ever renders on/for a mobile viewport in the first place.
2. **`pointer-coarse:` variant** (Tailwind v3.4 core, zero config — this stack is pinned
   to v3.4 per the design-system skill) — for EXISTING desktop-authored chrome that must
   stay visually identical for mouse/trackpad users but grow its hit-area for any
   touch-capable pointer, **regardless of `md`/viewport width**. This is the mechanism for
   the Phase-52 panel toolbar and `KnowledgePreviewNode`'s remove button (see Judgment
   Call #1 for why viewport-width alone cannot govern this case).

| `KnowledgePreviewNode`'s remove button | `size-6` | `pointer-coarse:touch-target` (same fix family — same canvas surface, same drift pattern Phase 52 fixed for its sibling node type; consistency extension, not scope creep) |

| `SidebarTrigger` (new consumer, see Component Inventory §1) | default `h-7 w-7` | explicit `className="size-11"` override at the call site (unconditional — mobile-only placement) |

**Documented deferred gap (not fixed this phase):** React Flow's own stock `Controls`

## Breakpoint & Mount Contract (D-48-07 mechanics — binding on this phase)

**The single switch line stays the Tailwind `md` breakpoint (768px).** No new breakpoint
is introduced. Two independent mechanisms, used for two independent concerns — do not
conflate them (this is Judgment Call #1's core resolution):

1. **Layout presentation** (what renders, CSS-only): plain Tailwind `md:`-prefixed
   responsive classes. No JS branch, no hook — e.g. `hidden md:flex` /
   `flex md:hidden` pairs switching between the mobile feed/list/stack markup and the
   desktop canvas/three-pane/graph markup.
2. **Mount/unmount** (whether a subtree's JS ever executes, for cases where CSS-only
   `hidden` would still pay the mount cost — e.g. `dynamic(ssr:false)` React Flow
   islands): the ONE shared hook, `useIsMobileViewport()`.

### `useIsMobileViewport()` — new shared hook

New file: `apps/web/src/hooks/use-is-mobile-viewport.ts` (new shared top-level hooks dir
— `~/hooks/use-is-mobile-viewport`, alongside the existing `~/components`/`~/trpc`
conventions; no existing shared-hooks location exists to reuse).

- `matchMedia("(max-width: 767px)")` — the exact numeric complement of Tailwind's `md:`
  (`768px` and up), so this hook can NEVER disagree with a `md:` CSS class at the same
  instant (off-by-one-px class of bug closed by construction).
- Mirrors `@polytoken/ui/sidebar`'s own internal `useIsMobile()` (`MOBILE_BREAKPOINT =
  768`, same `matchMedia` shape) byte-for-byte in behavior — this phase does not
  duplicate logic casually; it duplicates it because `useIsMobile` is `sidebar.tsx`'s
  private, unexported internal, and this hook needs to be public/shared across
  `/chat` + `/knowledge`. Same numeric contract, so the AppSidebar's own mobile Sheet
  behavior and this phase's canvas/graph mount decisions always agree on which side of
  `md` the viewport is on.
- SSR-safe default: `false` (renders the desktop/canvas presentation on the server and
  first paint; corrects on mount via `useEffect`+`matchMedia` listener — identical
  hydration-safety posture to `useIsMobile`'s own `undefined`-then-resolve pattern,
  coerced to `false` rather than `undefined` since every consumer needs a boolean).
- **Only 2 consumers this phase** (mount/unmount decisions): `ChatCanvasIsland`'s host
  (`page.tsx`'s `ConversationView`, gating whether `ChatCanvasIsland` — the
  `dynamic(ssr:false)` React Flow island — ever mounts) and `KnowledgeGraphIsland`'s host
  (`knowledge/page.tsx`, same gating). Every other mobile presentation in this phase
  (inbox stack, email-detail sheets, filter chips, touch-target sizing) is CSS-only
  (`md:` classes / `pointer-coarse:` variant) — no hook read, per 53-CONTEXT.md's
  "one hook, not scattered `window.innerWidth` reads."

## Component Inventory — Mobile Chrome

### 1. Global mobile nav trigger (closes a found gap — no `SidebarTrigger` exists anywhere in the app today)

**Found in research:** `AppSidebar` (`components/app-sidebar.tsx`) already uses
`@polytoken/ui/sidebar`'s `collapsible="icon"` Sidebar, which — via its own internal
`useIsMobile()` at the identical 768px line — already renders as an off-canvas `Sheet`
below `md`. But **no `SidebarTrigger` is mounted anywhere in the app** (`root layout.tsx`,
every page header). Without one, a signed-in user on a phone can never open the app nav —
this silently fails MOBL-02's "core flows... usable on a mobile viewport" for the
inbox→chat→knowledge navigation path itself, before any of this phase's other work even
matters. Judgment Call #2 closes it with a single, page-agnostic fix.

New markup in `apps/web/src/app/layout.tsx`, inside `SidebarInset`, directly above
`{children}`:

```
<div className="flex h-11 shrink-0 items-center gap-2 border-b border-border/50 bg-background px-2 md:hidden">
  <SidebarTrigger className="size-11" />
  <BrandMark variant="glyph" size="size-5" className="text-primary" />
  <span className="text-sm font-semibold text-foreground">Polytoken</span>
</div>
```

- `md:hidden` — exists ONLY below `md`; desktop keeps its current chrome-free
  `SidebarInset` exactly as today (zero visual change ≥md).
- `SidebarTrigger` default is `h-7 w-7` (28px) — explicitly overridden to `size-11`
  (44px) here (see Spacing Scale table). Default `aria-label`/sr-only text ("Toggle
  Sidebar") is kept verbatim — functionally correct, no copy authorship needed.
- One new bar, present on every route inside the shell (including `/login`? — no:
  `/login` renders outside `SidebarProvider`'s authenticated shell per its own routing;
  unauthenticated users never see this bar, consistent with the AppSidebar itself never
  rendering there).

### 2. `/chat` mobile feed (MOBL-01)

Below `md`:

- **`ChatCanvasIsland` is never mounted.** `page.tsx`'s `ConversationView` reads
  `useIsMobileViewport()`; when `true`, `viewMode` is force-coerced to `"chat"` (the
  existing docked mode) and `ChatCanvasViewToggle` is not rendered at all (`hidden
  md:flex` wrapper) — there is no user-facing way to request canvas mode on a phone, so
  the `dynamic(ssr:false)` React Flow import is never triggered (satisfies "do not
  merely hide it — avoid paying its init cost on phones," 53-CONTEXT.md verbatim).
  `localStorage`'s persisted `chat:canvas-view:{id}` value is still READ (so returning to
  desktop restores the user's prior choice) but never WRITTEN while mobile-forced.
- **The existing docked "chat" mode IS the inline feed** — found in research, not built
  new. `message-turn.tsx` already renders `genui_spec`/`genui_spec_streaming` parts
  inline via the unmodified `GenuiPartBoundary` in emission order, interleaved with text
  — this is exactly the "genui panels rendered inline in flow order" contract MOBL-01
  asks for (matches the Claude-Artifacts-inline-on-mobile precedent cited in
  `breakpoint-decision.md` §3). **Zero new feed-item components** — reuse
  `MessageList`/`MessageTurn`/`Composer` verbatim, per 53-CONTEXT.md's "reuse existing
  message-stream components wherever possible."
- **`ConversationRail` becomes an overlay, not a flex sibling, below `md`.** At `w-[280px]`
  fixed, an inline flex sibling would squeeze the main content column to ~80px at a
  360px viewport (see Judgment Call #3) — unusable, though not a literal CSS overflow.
  Below `md`, `ConversationRail`'s expanded state renders inside a `Sheet` (`side="left"`,
  default `w-3/4 sm:max-w-sm` — already appropriately sized for a phone, zero override
  needed) instead of the current `Collapsible` `w-0`/`w-[280px]` inline toggle. The
  existing `PanelLeft`/`PanelLeftClose` toggle button in `page.tsx`'s top bar (already
  `size-11`) becomes the `SheetTrigger`. Rail content (`ConversationRow` list,
  new-chat button) is unchanged — only its mobile MOUNT POSITION changes (overlay vs.
  inline column). Desktop (`≥md`) keeps the current `Collapsible` inline behavior
  unchanged.
- **Top toolbar** (`ConversationView`'s `h-11` row: `ChatCanvasViewToggle` +
  `ModelPicker` + `SaveStatusIndicator` + `CostMeter`): hiding `ChatCanvasViewToggle`
  below `md` (above) removes ~140px, leaving `ModelPicker` (compact `Select` trigger) +
  `CostMeter` (compact popover trigger) — both already narrow enough to fit at 360px
  with the existing `gap-2`/`gap-3` spacing; no additional class changes needed here.
  `SaveStatusIndicator` only renders `viewMode === "canvas"`, which is now impossible
  below `md` — it never renders on mobile, freeing more width automatically.
- **Composer**: unchanged (`min-h-[44px]` `Textarea`, `size-11` send button — already
  touch-target-compliant, confirmed in research).

## Judgment Calls (unattended-run discretion resolutions)

Explicit record per the constraints — each is a deliberate decision, not an oversight,
made because 53-CONTEXT.md left it to "Claude's Discretion" or because implementing
MOBL-01/02 required a call CONTEXT didn't fully spell out file-by-file:

1. **Two separate mechanisms for "mobile," not one.** 53-CONTEXT.md's own text names
   both a viewport-width concern (canvas→feed, `md` breakpoint) and a touch-capability
   concern (44px targets, "even under denser style packs") under the umbrella word
   "mobile." These are NOT the same axis: a touchscreen laptop or an iPad in landscape
   can sit at ≥768px (`md` and above) and still need 44px touch targets, while the
   Phase-52 canvas toolbar (a canvas-only construct) only ever renders when the canvas
   IS mounted — i.e. only at `≥md`. Resolving "the Phase-52 toolbar's size-6 icons get a
   mobile-only touch-target expansion" literally via a `max-width` viewport check would
   therefore be a no-op (the toolbar never renders below `md` in the first place, since
   the canvas isn't mounted there). The correct, and actually more useful, mechanism is
   Tailwind v3.4's native `pointer-coarse:` variant (`(pointer: coarse)` media feature,
   zero extra config on this stack's pinned v3.4) — it targets ANY touch-capable input
   regardless of viewport width, correctly covering the realistic case (a touch tablet
   running the canvas at ≥768px) that a width-only check would silently miss. Viewport
   width (`md:`/`useIsMobileViewport()`) governs LAYOUT PRESENTATION and MOUNT decisions;
   pointer capability (`pointer-coarse:`) governs TAP-TARGET SIZE. Both are D-48-07-legal
   (neither is a per-breakpoint TOKEN dimension — both are Tailwind-variant/component
   concerns resolved above the token layer, exactly what D-48-07 §1 permits).
2. **Global `SidebarTrigger` added to root layout, not each page.** No `SidebarTrigger`
   existed anywhere in the codebase before this phase (grep-verified) — without one, the
   app's own primary navigation is unreachable on any phone, silently failing MOBL-02
   before any of the phase's named surfaces even come into play. A single `md:hidden` bar
   in `layout.tsx` (inside `SidebarInset`, above `{children}`) covers every route
   (including ones not named in MOBL-02's flow list, e.g. `/entities`, `/studio`) with
   one change instead of N page-specific special cases — lower risk, and consistent
   with `AppSidebar`'s own single-source-of-truth posture.
3. **`ConversationRail` becomes a `Sheet` below `md` instead of staying an inline
   `w-[280px]`/`w-0` flex sibling.** At a 360px viewport, an inline 280px rail would
   squeeze the main chat column to ~80px — not a CSS overflow (flex shrinks, no
   scrollbar appears), but plainly unusable, failing MOBL-02's "usable on a mobile
   viewport" in spirit even where it technically passes a literal `overflow-x` check.
   Reusing the already-vendored `Sheet` (not a new component) mirrors the exact pattern
   `AppSidebar` already uses for the SAME class of problem (a rail that must not compete
   with the main content column for width on a phone).
4. **`/knowledge`'s filter facets re-presented as a horizontal chip bar, not a vertical
   list or a second Sheet.** 53-CONTEXT.md leaves "whether /knowledge mobile list reuses
   the entities-list idiom" open. A horizontal `overflow-x-auto` pill row (reusing
   `filter-rail.tsx`'s exact data + color recipe) keeps filters visible and reachable
   above the list at all times without consuming a second full-screen surface or a
   nested disclosure — the common, well-tested mobile pattern for a small, fixed set of
   category toggles (6 types here). A vertical checkbox list (desktop's own shape) would
   consume ~300px of scarce mobile vertical space before the list even starts.
5. **`NodeDetailPane`'s own internal close `X` is hidden (`hidden md:inline-flex`) when
   it renders inside the mobile `Sheet`**, since `SheetContent` already ships its own
   corner close control — two independent close affordances stacked in the same corner
   would be confusing and redundant. Desktop keeps `NodeDetailPane`'s own close button
   exactly as today (the mobile `Sheet` doesn't render on desktop, so this is additive,
   not a removal there).
6. **Inbox back-affordance is plain component state, not URL/history-synced,** and
   auto-select-on-load does NOT auto-navigate to the detail view on first paint (see
   Component Inventory §4 point 6) — a user landing on `/` for the first time on a phone
   always sees the list first, never gets silently deposited into a detail view they
   didn't tap into. Promoting this to real routing (enabling the hardware back button)
   is flagged as a natural v-next follow-up, not built this phase (`selectedEmailId`
   itself is already non-URL-driven today — this phase doesn't regress or improve that
   axis, only adds the missing "how do I get back" affordance on top of it).
7. **Email-detail's `CanvasShell` panel collapse is IN scope**, even though
   53-CONTEXT.md's Integration Points list didn't itemize `emails/[id]/_components/`
   file-by-file — MOBL-02's own success-criteria text explicitly names "email detail" in
   the required flow list, and the surface's literal fixed-width side panels (256px +
   288px) make "no horizontal overflow at 360px" mathematically unsatisfiable without
   SOME collapse mechanism. Reusing the identical `Sheet`-based pattern from Judgment
   Calls #3/#5 (not inventing a third mechanism) keeps this addition low-risk and
   consistent. The editing tools THEMSELVES (draw mode, region/role/entity pickers) are
   explicitly NOT redesigned — same carve-out 53-CONTEXT.md gives the Phase-52 toolbar,
   applied here by the same reasoning.
8. **No new slide/reveal motion is authored for the inbox list↔detail swap.** The
   already-adopted `t-modal-reveal`/`t-panel-reveal`/`t-dropdown-reveal` utilities
   (ADOPT-05, ~27) each have ONE designated consumer per their own doc comment in
   `globals.css` — this phase doesn't repurpose or extend that closed set. An instant
   swap (no transition) is the zero-risk default; a future phase could add a dedicated
   mobile slide transition as its own scoped design decision if the plain swap reads as
   jarring in practice.
9. **React Flow's own stock `Controls` buttons are left at their native ~26px size**
   (documented as a deferred gap in the Spacing Scale section) rather than restyled —
   they are third-party markup outside `@polytoken/ui`, and React Flow's own touch
   gesture handling (pinch/pan) is the primary mobile interaction for canvas navigation
   regardless of this specific button's hit-area.
