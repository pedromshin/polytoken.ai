<objective>
Close the Phase 44 verification gap (44-VERIFICATION.md, SC2 / TENA-03): three FastAPI
SSE endpoints backing the chat turn engine — `POST /v1/chat/stream`,
`POST /v1/chat/regenerate`, `POST /v1/chat/widget/submit` — enforce ZERO per-user
ownership on the client-supplied `conversation_id`, even though the Next.js BFF already
forwards a server-verified `X-User-Id` on every request. Any authenticated user who
obtains another user's `conversation_id` can read that conversation's context (via the
model reply conditioned on the victim's history), inject a message, or promote a
knowledge edge under the victim's conversation.

This plan adds `require_user_id` + a fail-closed conversation-ownership assertion to all
three endpoints (mirroring the phase's own `emails.py` `_assert_importer_owned` pattern),
adds an ownership-lookup method to `ChatConversationRepository`, and threads the caller's
`user_id` through the confirm_action dispatch path into `PromoteEdgeUseCase.execute` so
its existing 44-03 ownership guard actually runs.

Purpose: satisfy the phase goal's literal bar — "unreachable across users via ANY route" —
which the chat-SSE surface currently violates. This is the single highest-priority
tenancy item in the codebase per both 44-08's own SUMMARY and the verifier.
Output: three enforced endpoints, a new ownership port method + impl, four flipped
regression tests, and an updated sweep inventory.
</objective>

| Next.js BFF -> FastAPI | The BFF resolves `supabase.auth.getUser()` server-side and forwards `X-User-Id`. This plan makes the three chat SSE endpoints actually READ and ENFORCE that identity. |
| client -> conversation_id | `conversation_id` is client-supplied and untrusted for scoping; it must be validated against the caller's ownership, never trusted alone. |
