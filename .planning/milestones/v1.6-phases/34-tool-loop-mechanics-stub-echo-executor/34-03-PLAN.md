<objective>
Wire the primitives from Plan 34-01 into `_execute_turn` as a bounded in-stream round loop: on a
completed server-tool call, execute the ToolExecutor, feed the result back as a native tool_result
block, and continue streaming in the SAME ChatRun — capped at 4 rounds behind the max_tool_rounds
gate, fail-closed on exhaustion. Production wires an EMPTY executor mapping (no real server tool
exists until Phase 36), so this proves the mechanics against the echo stub only.

Purpose: LOOP-01 (the round loop, gate, new part types) + LOOP-03 (cap exhaustion -> visible text).
This phase IS gate G4 for Phases 35-39.
Output: the round loop in run_chat_turn.py, empty-mapping container wiring, end-to-end echo tests.
</objective>

| Boundary | Description |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-34-01 | Denial of Service | round loop | mitigate | hard cap `_MAX_TOOL_ROUNDS=4`; per-tool `asyncio.wait_for(~10s)`; `breaker.should_abort()` re-checked at each round boundary |
| T-34-02 | Denial of Service | tool output size | mitigate | `cap_tool_output(~2000 chars)` at the executor boundary (in-round messages are not covered by history trimming) |

| T-34-05 | Elevation of Privilege | OpenRouter tool round | mitigate | gate on `max_tool_rounds > 0` (0 on every OpenRouter/browser entry); OpenRouter adapter also drops tool blocks — defence in depth |
| T-34-SC | Tampering | npm/pip/cargo installs | mitigate | no new packages introduced this phase (stdlib asyncio only); slopcheck N/A |
