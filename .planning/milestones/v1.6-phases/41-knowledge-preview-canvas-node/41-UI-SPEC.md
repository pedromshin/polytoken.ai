## Spacing Scale

Unchanged 4-point grid (identical to every prior chat/studio/knowledge UI-SPEC — no new spacing
token; every value below is a bare Tailwind utility already in this app's vocabulary, all clean
multiples of 4):

| Token | Value | Usage this phase |
|-------|-------|-------------------|
| xs | 4px | `gap-1` (footer icon-adjacent text spacing, if any), node-dot focus-ring offset |
| sm | 8px | `gap-2` (header icon-to-label gap), `px-3`→12px is the next tier but `gap-2` used for popover form field spacing |
| — | 12px | `p-3`/`px-3` (mini-graph canvas padding, header/footer horizontal padding — NOT `p-4`, see rationale below) |

the same padding tier `chat-canvas.tsx`'s own toolbar (`px-3`) and `graph-legend.tsx`'s legend
chip (`px-3 py-2`) already use — not a new value.

## 1. Node Shell (chrome, header, drag handle, remove)

Mirrors `GenuiPanelNode` (`apps/web/src/app/chat/_canvas/genui-panel-node.tsx`) almost exactly —
same family member, not a redesign:

```tsx
<div
  className={`flex h-[240px] w-[320px] flex-col overflow-hidden rounded-lg border border-border/60 bg-background transition-shadow duration-150 animate-in fade-in-0 zoom-in-95 [animation-duration:250ms] motion-reduce:animate-none ${selected ? "ring-2 ring-primary ring-offset-1 shadow-elevation-2" : "shadow-elevation-1"}`}
>
  <Handle type="target" position={Position.Left} />
  {/* header */}
  <div className="node-drag-handle flex h-9 shrink-0 cursor-grab items-center justify-between gap-2 border-b border-border/60 bg-muted/40 px-3 active:cursor-grabbing">
    <span className="flex min-w-0 items-center gap-2">
      <Share2 className="size-3 shrink-0 text-primary" aria-hidden />
      <span className="truncate text-xs font-normal text-muted-foreground">
        {headerLabel}
      </span>
    </span>
    <button
      type="button"
      aria-label="Remove knowledge preview"
      className="flex size-6 shrink-0 items-center justify-center rounded-md text-muted-foreground transition-colors hover:text-destructive focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-1"
      onClick={(e) => { e.stopPropagation(); onRemove(id); }}
    >
      <X className="size-3.5" aria-hidden />
    </button>
  </div>
  {/* mini-graph body — see §2 */}
  {/* footer — see §3 */}
  <Handle type="source" position={Position.Right} />
</div>
```

**Fixed, not min-* dimensions.** `GenuiPanelNode`/`ChatNode` use `min-h-[…] min-w-[…]` because their
content (a genui spec / a chat composer) can legitimately grow. This node's content is bounded by
construction — a capped mini-graph that never overflows — so it takes exact `h-[240px] w-[320px]`
(same numeric family, `h`/`w` instead of `min-h`/`min-w`). This guarantees the layout math in §2
(a fixed 280×140 logical drawing box) always has the exact space it was computed for.

**`headerLabel` resolution order:** `node.data.label` (if the creator supplied one) → the resolved
focus node's own title once `knowledge.expandNode` settles (`nodes.find(n => n.id === focusNodeId)
?.label`) → the fallback literal `"Knowledge preview"` while loading with no custom label. Always
`truncate`d — never causes header overflow (product-register ban #8).

**Remove — the first node-level remove affordance on this canvas.** Neither `ChatNode` nor
`GenuiPanelNode` currently exposes an inline remove control (React Flow's own Backspace-key
deletion exists but is undiscoverable and not wired to the debounced-save path today). `41-CONTEXT.md`
explicitly calls for "the standard canvas node chrome (drag handle, remove)" — this phase introduces
it for real, as a `size-6` icon-only button (matches `NodeDetailPane`'s `size-7` close-button
register, one step down since this sits inside a `h-9` header rather than a `h-auto` panel header).
`onClick` removes this node from `nodes` state (and any edges touching it, if `41-CONTEXT.md`'s
persistence layer models edges to/from this node type) and immediately triggers the SAME debounced
`persistence.scheduleSave` call `chat-canvas.tsx`'s `handleNodeDragStop` already uses — no new save
mechanism. `stopPropagation` prevents the click from also selecting/dragging the node.

**Entrance:** identical to `GenuiPanelNode`'s existing `animate-in fade-in-0 zoom-in-95
[animation-duration:250ms] motion-reduce:animate-none` — no new entrance choreography invented for
this node type.

## 2. Mini-Graph Rendering (static, non-interactive)

### Rendering technique — locked

**SVG for edges, absolutely-positioned HTML `<Link>`s for nodes.** Not pure SVG (Radix `Tooltip`
requires a real DOM element trigger, and CSS `truncate` on `foreignObject` text is fragile) and NOT
a second `<ReactFlow>` instance (the confirmed, rejected hazard). One `<svg>` layer (`aria-hidden`,
purely decorative lines) stacked under one `role="group"` div layer of real `<Link>` node dots,
both positioned against the SAME fixed 280×140 logical coordinate box, centered inside the node
body via `mx-auto` + `flex items-center justify-center`.

```tsx
<div className="relative flex flex-1 items-center justify-center p-3">
  <div style={{ width: 280, height: 140 }} className="relative motion-safe:animate-in motion-safe:fade-in motion-safe:duration-200">
    <svg width={280} height={140} className="absolute inset-0" aria-hidden>
      {edges.map((e) => <PreviewEdge key={e.id} edge={e} positions={positions} />)}
    </svg>
    <div role="group" aria-label="Related knowledge nodes" className="absolute inset-0">
      {nodes.map((n) => <PreviewNodeDot key={n.id} node={n} position={positions[n.id]} distance={distanceOf(n.id)} />)}
    </div>
  </div>
</div>
```

**One single fade-in for the whole layer** (`motion-safe:animate-in motion-safe:fade-in
motion-safe:duration-200` — same 200ms tier `save-status-indicator.tsx`/`ToolRoundActivityRow`
already use). **No per-node stagger, no orchestrated reveal sequence** — product-register ban on
"orchestrated page-load choreography" applies here exactly.

### Layout algorithm — hand-rolled elliptical radial (locked, not dagre)

`41-CONTEXT.md` explicitly rejects a second `<ReactFlow>` AND — separately — locks the layout
utility itself to be "hand-rolled, small," which this phase reads as: do not even reuse
`/knowledge`'s own `@dagrejs/dagre`-based `layoutGraph` (a hierarchical top-to-bottom tree layout
designed for a 6-type, unbounded, pannable graph). A capped, always-centered-on-one-focus-node
preview needs a MUCH simpler shape: a two-ring "ego network."

```ts
// knowledge-preview-layout.ts
export function layoutPreview(
  focusId: string,
  oneHop: readonly string[],   // stable order, as returned by expandNode
  twoHop: readonly string[],   // stable order, sorted by connecting 1-hop parent (see below)
  box: { width: number; height: number } = { width: 280, height: 140 },
): Record<string, { x: number; y: number }> { /* … */ }
```

- **Center:** the focus node sits at the box's exact center point (`width/2, height/2`).
- **Ring 1 (1-hop):** placed on an ellipse with `rx = 0.38 * width`, `ry = 0.38 * height`, evenly
  spaced by angle, **in the order `expandNode` returned them** (stable — the same server response
  always renders identically; no randomness, no re-sort on every render).
- **Ring 2 (2-hop):** placed on an outer ellipse with `rx = 0.62 * width`, `ry = 0.62 * height`.
  Sorted by their connecting 1-hop parent's ring-1 angle (a simple `sort`, not a physics pass) so
  a 2-hop node visually clusters near the 1-hop neighbour that introduced it, instead of being
  scattered randomly around the full circle.
- **No collision detection, no force simulation, no iterative relaxation.** At the ≤25-node cap,
  even-angular-spacing on two concentric ellipses does not produce meaningful overlap in practice;
  this is a deliberate simplicity trade-off consistent with "hand-rolled, small."
- Both ellipses share the box's own aspect ratio (2:1 for the 280×140 box), so nothing clips the
  card edges regardless of exact node-dot size.

### Node dot sizes (multiples of 4, no new spacing scale)

| Distance | Diameter | Label |
|----------|----------|-------|
| Focus | 20px | `text-xs font-semibold text-primary`, `max-w-[64px] truncate`, shown BELOW the dot |
| 1-hop | 12px | `text-xs font-normal text-foreground`, `max-w-[56px] truncate`, shown below the dot |
| 2-hop | 8px | none inline — full label available via tooltip only (keeps the widget legible at the ≤25-node cap) |

### Focus-node emphasis

The focus node is the ONLY node with: the glow shadow, the 2px border (vs. 1px for others), the
`Share2` icon, and semibold label weight. This is deliberately the single strongest visual signal
in the widget — "this is the node the preview is anchored on" — everything else fades outward from
it (teal → soft-teal → neutral, per the Color table above).

### Label truncation + Radix tooltip (locked: yes, on every node)

Every node dot (focus, 1-hop, AND 2-hop — not just the ones with visible truncated text) is wrapped
in a `Tooltip` whose trigger is the node's own `<Link>` (`asChild`), revealing the FULL untruncated
label on hover/focus. This is the only way a 2-hop node's identity is ever discoverable (it has no
inline label at all) and it also covers the case where a 1-hop/focus label truncates visually.

```tsx
<TooltipProvider delayDuration={300}>
  {/* … node dot layer … */}
</TooltipProvider>
```

One local `TooltipProvider` per mini-graph instance (matches `canvas-toolbar.tsx`'s own
per-component-tree convention — there is no app-wide global `TooltipProvider` mounted at layout
level in this codebase).

## 3. Cap + "+N more — open in Knowledge" Footer

**Cap, locked at exactly 25** (`MAX_PREVIEW_NODES = 25`, resolving `41-CONTEXT.md`'s "~25"
discretion note to a hard number — same "resolve a `~N` into an exact constant" move `39-UI-SPEC.md`
already made for its own chip cap). Counts the focus node itself, i.e. focus + up to 24 neighbours.

**Trim priority when the server's `expandNode` response exceeds 25 nodes:**
1. Focus node — always kept.
2. 1-hop neighbours — kept in full up to the remaining budget (24). If 1-hop alone exceeds 24, trim
   the 1-hop list itself (stable order, first-N as returned) and drop ALL 2-hop nodes.
3. 2-hop neighbours — fill whatever budget remains after all 1-hop nodes are kept, stable order.
4. Any edge whose endpoint fell outside the kept node set is silently dropped (same technique
   `capBudget` in `expand.ts` already uses server-side — never render a dangling edge).

**Footer — always rendered** (not conditional on overflow existing), because `41-CONTEXT.md`'s own
requirement is "deep-links out," and this footer is that guaranteed escape hatch regardless of query
state. Resolution of the discretion point: `41-CONTEXT.md`'s literal phrasing ties the copy to the
overflow case, but always showing the row (with copy that varies) is strictly more discoverable and
costs nothing — the exact overflow-case copy matches `41-CONTEXT.md` verbatim.

```tsx
<Link
  href={`/knowledge?focus=${encodeURIComponent(focusNodeId)}`}
  className="flex h-7 w-full shrink-0 items-center justify-center gap-1 border-t border-border/60 px-3 text-xs text-muted-foreground transition-colors hover:bg-accent hover:text-accent-foreground focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-1"
>
  {footerCopy}
</Link>
```

| State | `footerCopy` |
|-------|--------------|
| Loading, or success with no overflow | `"Open in Knowledge →"` |
| Success with overflow (`overflowCount > 0`) | `` `+${overflowCount} more — Open in Knowledge →` `` |
| Error / not-found empty state | `"Open in Knowledge →"` (still usable — the focus id is always known client-side regardless of the preview query's own outcome) |

`overflowCount = Math.max(0, serverNodeCount - MAX_PREVIEW_NODES)`, where `serverNodeCount` is
`knowledge.expandNode`'s own `nodes.length` (itself already capped server-side at 50 — if the
server ALSO truncated, this count is an honest lower bound, not a precise total; no attempt is made
to represent "50+ more" specially, consistent with this app's existing "don't over-claim precision"
posture).

The `→` suffix reuses `node-detail-pane.tsx`'s own established micro-copy convention ("Open entity
→", "View N instances →") verbatim — not a new arrow-copy pattern invented for this phase.

## 4. Loading / Empty / Error States

All non-success states render inside the SAME body region the mini-graph would otherwise occupy
(`relative` positioning parent, exactly like `CanvasEmptyState`/`UnknownNodeTypePlaceholder` already
anchor `EmptyState`'s `absolute inset-0` centered variant). The footer (§3) still renders
underneath every one of these states — the deep-link escape hatch is never hidden.

| State | Condition | Rendering |
|-------|-----------|-----------|
| Loading | `expandNode` query pending | Three `Skeleton` circles (`@nauta/ui/skeleton`, `rounded-full`) at 20px/12px/12px roughly mimicking the eventual focus + 2 neighbours, loosely centered, `motion-safe:animate-pulse` (Skeleton's own built-in behavior) — `role="status" aria-label="Loading knowledge preview"`. Mirrors the app's established "static ghost divs, not React Flow, not EmptyState" skeleton convention (`CanvasSkeleton`/`KnowledgeGraphSkeleton`) rather than force-fitting a spinner into `EmptyState` (which has no spin-capable icon slot). |
| Error | `expandNode` query threw | `EmptyState` — `icon={AlertCircle}` `tone="destructive"` `layout="centered"` `size="compact"` `heading="Couldn't load this preview."` `body="Try again, or open the full graph."` `action={{ label: "Retry", onClick: refetch }}` |
| Empty — not found / inaccessible | `expandNode` returns `nodes: []` (fail-closed per `expand.ts`'s T-32-03 — an inactive/cross-tenant/deleted seed) | `EmptyState` — `icon={Share2}` `tone="muted"` `layout="centered"` `size="compact"` `heading="This preview is unavailable."` `body="The knowledge node may have been removed or is no longer accessible."` — deliberately vague per T-32-03's own fail-closed posture (never confirms/denies existence), no action button |
| Empty — no connections | `expandNode` returns exactly 1 node (the seed itself, `edges: []`) | `EmptyState` — `icon={Share2}` `tone="muted"` `layout="centered"` `size="compact"` `heading="No connections yet."` `body="This knowledge node isn't linked to others yet."` — no action button (mirrors `GraphNoSchemaState`'s no-button convention for "nothing to click here") |
| Success | `nodes.length > 1` | The mini-graph itself (§2) |

**Zero new typography introduced:** `EmptyState`'s `text-base font-semibold` heading (16px) is that
shared primitive's own pre-existing contract (consumed via props, never overridden with a
`className` here) — not a new size this phase adds to the 4-role scale.

## 5. Deep-Link Interactions (whole-node vs per-node click targets)

**Locked click-target inventory — exactly 3 kinds of real anchor, nothing else is clickable:**

1. **Each mini-graph node dot** (focus, 1-hop, 2-hop alike) — a real `<Link href="/knowledge?focus=
   {thatNode'sOwnId}">`, computed the SAME fixed way `<ProvenanceLink>`'s internal `hrefFor` switch
   already computes a `kind="knowledge"` href (`` `/knowledge?focus=${encodeURIComponent(id)}` ``) —
   never trusting a server-supplied route string, matching `39-UI-SPEC.md`'s "compile-time switch,
   never model/server-authored" discipline. Every node is independently clickable — not just the
   focus — so a user can jump straight to any neighbour shown in the preview, not only the anchor.
2. **The footer row** (§3) — a real `<Link>` to `/knowledge?focus={the node's OWN focusNodeId}`
   (always the ORIGINAL focus this preview was created for, regardless of which node dot the user
   might separately click).
3. **Nothing else.** The node header, the empty SVG/canvas background between dots, and the node
   body's own background are all inert — clicking them does nothing (the header remains the DRAG
   handle only, per `.node-drag-handle`).

**Why NOT a "whole-node click-through":** `41-CONTEXT.md`'s own phrase "node body click-through" is
resolved here as "each individual node WITHIN the body," not "the body's background as one giant
link." A giant invisible full-card link would conflict with the header drag handle, would make the
remove button's hit area ambiguous, and contradicts this app's own existing precedent — nowhere else
in this codebase (chip lists, `NodeDetailPane`, `EntityInstanceNode` selection) treats an entire
card/row as a single undifferentiated navigation target; navigation always comes from an explicit,
visually distinct link (`"Open entity →"`, a chip, a node dot). Per-node links preserve that
discipline and are MORE useful (any neighbour is reachable, not just the anchor).

**Hit-target sizing for small dots:** the 12px/8px dots use a padded, larger invisible click area
than their visual size — `<Link>` gets a fixed `min-h-6 min-w-6` (24×24px) hit box centered on the
dot's logical point (`absolute` positioning with the visual dot centered inside via flex), meeting
the WCAG 2.5.8 AA 24×24px minimum target size even though the visual glyph is smaller. The focus
node's 20px visual dot already exceeds this minimum on its own.

**Accessible names:** every node `<Link>` carries `aria-label={`Open ${fullLabel} in Knowledge
graph`}` (the FULL, untruncated label — critical for 2-hop dots, which have no visible text at
all). The tooltip (§2) is a visual-hover convenience on top of this, not a substitute for it.

## Accessibility

- **Loading:** `role="status"` + `aria-label="Loading knowledge preview"` on the skeleton wrapper.
- **Error:** `EmptyState`'s own markup (no explicit `role="alert"` in the shared primitive today —
  consumed as-is, not modified this phase).
- **Empty states:** plain, non-live content — matches `GraphNoSchemaState`'s existing posture.
- **Node dots:** every one is a real, keyboard-focusable `<Link>` with a full (never-truncated)
  `aria-label`; the surrounding `role="group" aria-label="Related knowledge nodes"` orients screen
  reader users encountering a dense cluster of small links.
- **Tooltip:** `delayDuration={300}` (a deliberate, non-zero delay so keyboard/mouse users
  traversing the graph aren't flooded with tooltips on every focus step — matches typical Radix
  tooltip UX defaults elsewhere in comparable dense-UI contexts in this app).
- **Reduced motion:** the mini-graph's entrance (`motion-safe:animate-in motion-safe:fade-in`) and
  the loading skeleton's pulse (`Skeleton`'s own built-in `motion-safe:animate-pulse`) both fully
  disable under `prefers-reduced-motion: reduce` — no separate `motion-reduce:` rule needed
  (`motion-safe`/`motion-reduce` are mutually exclusive by Tailwind construction).
- **Focus indicators:** every interactive element (node dots, footer link, remove button, toolbar
  trigger, popover fields/buttons) uses `focus-visible:ring-2 focus-visible:ring-ring
  focus-visible:ring-offset-1` — the same neutral-ring convention `<ProvenanceLink>` and `badge.tsx`
  already established; never the brand `ring-primary` selection idiom (these are links and controls,
  not "selected" canvas nodes).
- **Non-color differentiation:** the error state's `AlertCircle` icon shape + `EmptyState`'s
  `tone="destructive"` carries the signal independent of hue (WCAG 1.4.1) — same posture
  `39-UI-SPEC.md` already established for its own error row.
- **Contrast:** every color pair used (`primary`/`primary-foreground`-adjacent teal fills against
  `background`, `muted`/`muted-foreground`, `destructive` against `background`) is already
  WCAG-AA-verified by `28-UI-SPEC.md`'s TOKEN-01 contrast pass and Phase 32's own knowledge-node
  styling — no new pair introduced.
- **Security:** no `dangerouslySetInnerHTML`, no `eval`. The one untrusted-input boundary
  (`focusNodeId` from the Add-preview form) is Zod-validated (`z.string().uuid()`) before ever being
  used to construct a request or a node — a system-boundary validation point per this project's own
  input-validation guardrail.
- **Visual verification:** connected-env/browser screenshot verification deferred to execution/audit
  time per v1.4/v1.5/v1.6 precedent (`STATE.md` → Deferred Items) — mark `human_needed` if the
  playwright-core loop isn't available in the execution session.

## DO-NOTs (explicit, binding)

1. **No nested `<ReactFlow>`.** Not even a "just for the mini-graph" instance. The confirmed hazard
   (duplicate providers, wheel/drag capture, persistence blindness) that `41-CONTEXT.md` rejects
   applies to ANY second `@xyflow/react` mount, full stop.
2. **No pan/zoom/drag of the mini-graph content.** No wheel handler, no drag handler, no pinch
   gesture anywhere inside the SVG/div layer. The ONLY drag surface on this node is the
   `.node-drag-handle` header (moves the WHOLE node on the outer canvas), exactly like every other
   node type.
3. **No new npm dependency.** No `d3-force`, no second layout library, no reuse of `@dagrejs/dagre`
   even though it's already installed for `/knowledge` — the hand-rolled elliptical layout in §2 is
   the whole point of "hand-rolled, small."
4. **No new design token, no `theme.extend` entry, no new keyframe.** Every class in this document
   resolves to an existing `globals.css` CSS variable or an already-used Tailwind utility.
5. **Never hand-duplicate the tier stroke/dash/opacity values.** Import `tierEdgeStyle` from
   `~/app/knowledge/_components/tier-edge-style.ts` directly — that file's own docstring already
   names "single source of truth" as the rule.
6. **Locked renderer files stay untouched.** `spec-renderer.tsx`, `render-node.tsx`,
   `genui-part-boundary.tsx` — this node never renders a genui spec; there is no reason for this
   phase to touch any of the three, and it must not.
7. **`<ProvenanceLink>` itself is not modified.** This phase CONSUMES its href-computation contract
   (recompute `kind`+`id` → route, never trust a server string) as a pattern to replicate for the
   mini-graph's own node/footer links — it does not edit `provenance-link.tsx`.
8. **No full free-text search/picker UI inside the node or the Add-preview popover.** Manual
   paste-an-ID only, per `41-CONTEXT.md`'s explicit discretion resolution; a real picker is backlog.
9. **`ALLOWED_PROCEDURES`/the model-facing tRPC allowlist is untouched.** `knowledge.expandNode` is
   called directly by app UI (existing, already-shipped procedure) — this is not a model tool call
   and does not implicate that gate.
10. **No whole-node-body click-to-navigate.** Only individual node dots and the footer row are real
    anchors (§5) — the header, background, and empty canvas space between dots are all inert.
11. **No brand-accent (`primary`/teal) usage beyond the exact reuse already documented** (focus/
    1-hop node dots, header icon) — this phase adds zero NEW accent surface area beyond what Phase
    32's `graph-nodes.tsx` already approved for knowledge-node identity.
12. **No per-node stagger / orchestrated entrance animation.** One single fade-in for the whole
    mini-graph layer — never a sequential per-dot reveal.
13. **No in-widget tier legend, no edge tooltip.** `/knowledge`'s own persistent `GraphLegend`
    already teaches the encoding; this bounded card doesn't re-teach it.
14. **No confirmation dialog for removing a preview node.** Low-stakes, easily redone — matches
    `EdgeCreationPicker`'s own precedent for a comparably low-stakes canvas removal.
