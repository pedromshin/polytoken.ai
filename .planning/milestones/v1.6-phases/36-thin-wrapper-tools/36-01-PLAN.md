<objective>
Close the two structural gaps Phase 34 deliberately left open (the `ToolExecutor` port has no
tenant-scoping parameter, and `_build_tool_offer` synthesizes an empty-`properties` stub schema for
every server tool) and ship the FIRST real, production `ToolExecutor`: `lookup_entity`, a thin
wrapper over the existing `EntityResolutionRepository.find_candidates()` (TOOL-01).

Purpose: TOOL-01 requires the chat agent to return grounded, cited, tenant-scoped entity results
with zero new backend. That requires the tool-loop's own contract to carry `importer_id`, which it
does not yet do (Phase 34 built the mechanics against a tenant-agnostic echo stub). This plan is
Wave 1 — interface-first — because Plan 36-02's `search_emails` executor and the container wiring
both build on the same `importer_id`-carrying port and the same citation/envelope helpers.

Output: an extended `ToolExecutor` Protocol, a shared `app/infrastructure/tools/envelope.py`
citation/truncation module, and `LookupEntityExecutor` + `build_lookup_entity_tool()` — not yet
wired into `container.py` (that lands in 36-02 alongside `search_emails`, since both tools share one
`_provide_run_chat_turn` edit and it is cleaner as a single diff).
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LLM (Bedrock Claude) -> tool arguments | `arguments["name_or_id"]` is model-authored, untrusted free text |
| ToolExecutor -> Supabase RPC | `find_candidates`/`find_by_id` queries, must stay importer-scoped |
| ToolExecutor result -> LLM context | The JSON envelope re-enters the model's context as a `tool_result` block |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-36-01 | Information Disclosure | LookupEntityExecutor cross-tenant read | mitigate | `find_by_id`'s returned row's `.importer_id` is compared against the caller's `importer_id`; a mismatch is treated identically to "not found" (falls through to the name-search path scoped to the CALLER's importer_id only) — never returns another tenant's instance, identifiers, or aliases |
| T-36-02 | Tampering | `name_or_id` prompt-injection payload embedded in a plausible-looking entity name | accept | Out of scope for this phase (Phase 38 owns the adversarial fixture suite + the "tool results are data, not instructions" system-prompt hardening line, per synthesis Fork 3/P6); `maxLength: 200` on the tool schema bounds the payload size as defense-in-depth only |
| T-36-03 | Denial of Service | `_search_by_name`'s per-entity-type `find_candidates` loop | accept | Bounded by the importer's active entity-type count (small, single-tenant local/sandbox deployment) and Phase 34's existing `asyncio.wait_for(~10s)` per-tool timeout + `cap_tool_output`'s 2000-char output cap already bound worst-case blast radius |
| T-36-04 | Repudiation | Interface-widening change to `ToolExecutor.execute` (new required `importer_id` kwarg) silently un-enforced by a future executor | mitigate | Made a REQUIRED (no-default) keyword parameter on the Protocol, not optional — a future executor that ignores it still receives the correct value and any executor that fails to type-check against the Protocol is caught by `mypy`/structural typing at the call site |
