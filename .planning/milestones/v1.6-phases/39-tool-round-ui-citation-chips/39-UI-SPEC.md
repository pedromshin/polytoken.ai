## Color

**Zero new color tokens. Zero new brand-accent (teal `primary`) usage.** Every value below reuses
an existing `globals.css` CSS variable. This phase's net effect on the 60/30/10 contract is
neutral-to-negative (it consumes 0% of the 10% accent budget) — worth stating explicitly since a
naive "citation chip" design instinct reaches for a colored badge (see `entity-chips.tsx`'s
violet treatment below for why this phase deliberately does NOT copy that pattern).

### Naming-collision reminder (same one `28-UI-SPEC.md` documented — read before auditing)

`hover:bg-accent` below refers to shadcn's **neutral** `--accent`/`--accent-foreground` hover-fill
slot (whisper-teal-tinted gray, ~10% saturation) — NOT this app's 60/30/10 brand accent (`primary`,
`hsl(164 39% 22%)`, reserved for CTAs/selection/state per the accent-allowlist). `<ProvenanceLink>`
never uses `primary`/`ring-primary` — its focus ring uses the neutral `--ring` token, matching
`badge.tsx`'s own `focus:ring-2 focus:ring-ring` convention, not the canvas nodes' `ring-primary`
selection idiom (chips aren't a "selected" state, they're a link).

### Collapsed row — error

| Element | Token | Class |
|---------|-------|-------|
| Icon (`AlertTriangle`) | `--destructive` | `text-destructive` |
| Label text | `--muted-foreground` (deliberately NOT `text-destructive`, see rationale below) | `text-muted-foreground` |

**Why the error row's text stays neutral while `InlineErrorCard`'s doesn't:** `InlineErrorCard`
(`role="alert"`, bold `text-destructive` text, bordered box) marks a whole TURN failing — high
severity, the user must notice and can retry. A single failed tool round is recoverable *within
the same turn* — the model already has the error content fed back and keeps generating (LOOP-02's
"never silent" motto: the turn's own visible text explains what happened next). Matching
`InlineErrorCard`'s full visual weight here would misrepresent severity and violate the "quiet
single-line entry" decision in `39-CONTEXT.md`. The icon alone (shape + `--destructive` hue) is
the non-color-redundant signal (WCAG 1.4.1); text stays legible-neutral. This is a considered
downgrade in visual weight, not an oversight.

## SSE / Part Contract (cross-cutting — read before wiring `use-chat-stream.ts`)

### The naming collision `39-CONTEXT.md` doesn't spell out, and its resolution

`run_chat_turn.py`'s `_run_server_tool_round` (Phase 34, already shipped) already calls
`self._emit(run.id, "tool_call", {"tool_name", "id", "arguments"})` and
`self._emit(run.id, "tool_result", {"tool_name", "id", "content", "isError"})` — and `_emit`'s
return value is already `yield`-ed through `_execute_turn` → `stream_run_events` → the SSE wire,
**today**, for every real server-tool round. But `use-chat-stream.ts`'s `applyRunEvent` already
owns `"tool_call"`/`"tool_result"` for a **different** purpose (the `emit_ui_spec`/interactive-widget
streaming reducer, keyed on `data.partial_json`) — reusing those same two `type` strings for the
server-tool round would make a real `lookup_entity` round get folded into
`interactive_widget_streaming` (wrong part type, empty content, a permanently-stuck skeleton) the
moment Phase 36's real executor ships. `39-CONTEXT.md` already locks the fix ("Emit new SSE frame
types... Additive only") — this section is that fix, made concrete.

**Resolution: 2 new, non-persisted, mirrored SSE frame types.** The existing DB-persisted
`"tool_call"`/`"tool_result"` run events (Phase 34's durable record, replayed via
`chat.getHistory`) are **untouched** — no migration, no CHECK-constraint change (Roadmap Phase 39:
"No migrations"). `_run_server_tool_round` additionally constructs two **transport-only**
`ChatRunEvent`s (never routed through `_emit`/`append_event`, `seq: null` — matching the existing
nullable-`seq` semantics for a non-durable frame) at the same two dispatch points:

| New event `type` | Emitted at (mirrors) | `data` shape |
|---|---|---|
| `server_tool_call` | Same moment as the existing persisted `"tool_call"` event | `{ "tool_name": string, "id": string }` (deliberately omits `arguments` — the activity row never displays them, per "full result JSON stays out of the transcript") |
| `server_tool_result` | Same moment as the existing persisted `"tool_result"` event | `{ "tool_name": string, "id": string, "content": string, "isError": boolean }` (identical shape to the persisted event's `data` — this lets the client build the SAME `tool_invocation_result` part client-side that `chat.getHistory` will later confirm, avoiding any visual "flash" on turn-terminal refetch) |

### `use-chat-stream.ts` additions

`ChatRunEventType` union + `CHAT_RUN_EVENT_TYPES` set both gain `"server_tool_call"` and
`"server_tool_result"` (additive, mirrors how every prior phase extended this union — no removal,
no rename).

`MessagePart` union gains 3 members:

```ts
| { readonly type: "tool_invocation_streaming"; readonly toolUseId: string; readonly toolName: string }
| { readonly type: "tool_invocation"; readonly toolUseId: string; readonly toolName: string; readonly arguments: Readonly<Record<string, unknown>> }
| { readonly type: "tool_invocation_result"; readonly toolUseId: string; readonly toolName: string; readonly content: string; readonly isError: boolean }
```

Field names (`toolUseId`/`toolName`/`content`/`isError`) are chosen to match
`build_tool_invocation_part`/`build_tool_invocation_result_part`'s persisted part shape **exactly**
(`apps/email-listener/app/application/use_cases/run_chat_turn_tool_loop.py`) — the client-built
part (from `server_tool_result`) and the server-persisted part (from `chat.getHistory`) are
byte-identical in shape, so swapping one for the other on terminal settle is inert.

`applyRunEvent` gains 2 branches (inserted alongside the existing `tool_call`/`tool_result`
branches, same function, same file):

- **`server_tool_call`**: replace-or-append a trailing `tool_invocation_streaming` part — same
  replace-if-same-`toolId`-else-append shape as the existing `genui_spec_streaming` reducer, but
  simpler (no chunk concatenation: this event fires exactly once per round, not progressively).
  `state` stays `"streaming"`.
- **`server_tool_result`**: if the trailing part is `tool_invocation_streaming` with a matching
  `toolId`, replace it with the finalized `tool_invocation_result` part built from this event's
  `data`; otherwise append it. `state` stays `"streaming"`.

`message-turn.tsx`'s part-type switch gains 3 branches:

- `"tool_invocation_streaming"` → renders `<ToolRoundActivityRow toolName={part.toolName} />`
- `"tool_invocation"` → renders `null` (see DO-NOTs — this part exists for the model's own replay
  context and the `tool_invocation_result` row already narrates the round; a second visible row
  for the SAME round would contradict "collapsed... single-line entry")
- `"tool_invocation_result"` → renders `<ToolInvocationResultRow toolName={part.toolName} content={part.content} isError={part.isError} />`

**What needs zero new UI (confirm, don't build):** `PARSE_FAILURE_TEXT` and
`ROUND_CAP_EXHAUSTED_TEXT` (Phase 34's existing loop-level failure strings) are plain `text` parts
already — `message-turn.tsx`'s existing `MarkdownRenderer` branch already renders them correctly.
These are the "Failed/errored rounds render the visible text part Phase 34 already emits" case
`39-CONTEXT.md` refers to — distinct from a per-tool `isError: true` result (which DOES get the
`ToolInvocationResultRow` error treatment below, because it's still a completed
`tool_invocation_result` part, not a loop-level text-only bailout).

## Component 1 — `ToolRoundActivityRow` (TUI-01, in-progress state)

**States, resolved:** `39-CONTEXT.md`/the orchestrator ask for "pending/streaming/complete/error."
In practice this app has exactly one visible **in-flight** rendering (there is no observable
partial content between round dispatch and round settlement — the tool executor call is a single
opaque await, never chunked back to the client) — "pending" and "streaming" are the SAME visual.
Documented here rather than inventing a fake intermediate state. "complete"/"error" are Component
2 (`ToolInvocationResultRow`), below — the activity row unmounts the instant the round settles,
replaced in-place by the part-array swap already described in the SSE contract.

**Precedent chosen, and why it's not `GeneratingRing`:** `39-CONTEXT.md` says "existing
`GeneratingRing`/streaming affordance style" — `GeneratingRing` wraps bounded child content (a
card/panel outline sweep); there is no panel here, just a status line. The correct instantiation
of "the existing streaming affordance family" for a bare status line is the
`SubmittingRow`/`GeneratingIndicator` idiom already in this exact file tree
(`interactive-widget-boundary.tsx`'s `SubmittingRow`, `message-list.tsx`'s `GeneratingIndicator`):
`Loader2` + text, `motion-safe:animate-spin`. Reused verbatim, not reinvented.

### Anatomy

```tsx
<div
  role="status"
  className="flex items-center gap-2 py-1 text-sm text-muted-foreground motion-safe:animate-in motion-safe:fade-in motion-safe:duration-200"
>
  <Loader2 className="size-4 shrink-0 motion-safe:animate-spin" aria-hidden />
  <span>{label}</span>
</div>
```

- `role="status"` — implicit polite live region (matches `save-status-indicator.tsx`'s exact
  convention). The page-level `sr-only aria-live="polite"` announcer in `chat/page.tsx`
  (`liveAnnouncementFor`) is keyed off `StreamState`, which does NOT change mid-round (state stays
  `"streaming"` for the whole turn) — so it can never announce a round starting/ending on its own.
  `role="status"` on this row is the correct, minimal fix: the row's own visible text is also what
  gets announced, no new global announcer function needed.
- Entrance: `motion-safe:animate-in motion-safe:fade-in motion-safe:duration-200` — same 200ms
  fade `save-status-indicator.tsx`'s "Saved" label already uses; gated correctly (no
  `motion-reduce:` needed since `motion-safe:` already fully omits the animation classes when
  `prefers-reduced-motion: reduce` is set — Tailwind's `motion-safe`/`motion-reduce` variants are
  mutually exclusive by construction, no extra work required).
- **No trailing text caret.** The blinking `▍` caret in `message-turn.tsx` is reserved for the
  actively-streaming TEXT part; do not add it here — the spinner already signals "in progress."
- Multiple rounds in the same turn (up to 4, `_MAX_TOOL_ROUNDS`) each get their OWN
  `tool_invocation_streaming` → `tool_invocation_result` pair in the `parts` array, in natural
  emission order — the existing `parts.map(...)` in `message-turn.tsx` already renders every part
  in sequence, so N rounds simply produce N activity rows / N collapsed rows over the turn's
  lifetime. No round-count text is surfaced (`39-CONTEXT.md`: "keep it subtle" — resolved to: not
  shown at all, since the natural sequential appearance already communicates multiplicity without
  adding copy).

### Copy (`label`) — tool-name → human label map

| `toolName` | Label |
|---|---|
| `lookup_entity` | "Looking up an entity…" |
| `search_emails` | "Searching emails…" |
| `search_knowledge` | "Searching knowledge…" |
| any other / unrecognized | "Running a lookup…" |

(The 3 named tools are Phase 36/37's `lookup_entity`/`search_emails`/`search_knowledge` — the only
real server tools this milestone ships. The fallback covers the Phase-34 test-only `echo` tool and
any future addition without a code change being strictly required to stay non-broken, though
adding a real entry to the map is still the right follow-up when a new tool ships.)

## Component 2 — `ToolInvocationResultRow` (TUI-01 completion + TUI-02 citations)

### Anatomy — success (`isError: false`, `content` parses as JSON)

```tsx
<div className="flex flex-wrap items-center gap-x-2 gap-y-1 py-1 text-sm text-muted-foreground">
  <span>{completedLabel}</span>
  {citations.length > 0 && (
    <div className="flex flex-wrap items-center gap-1">
      {visibleCitations.map((c) => (
        <ProvenanceLink key={`${c.kind}:${c.id}`} kind={c.kind} id={c.id} />
      ))}
      {overflowCount > 0 && (
        <span className="inline-flex items-center rounded-md border border-transparent bg-muted px-2 py-1 text-xs font-normal text-muted-foreground">
          +{overflowCount}
        </span>
      )}
    </div>
  )}
</div>
```

- **"Single-line" clarified:** describes the row's *information density* (a one-sentence summary,
  never an expandable block or the full result JSON) — not a hard CSS `nowrap`. `flex-wrap` is
  used deliberately: wrapping to a 2nd visual line when chip count is high is NOT a scrollbar (the
  "no second scrollbar" constraint is about overflow containers, not text reflow) and is strictly
  better UX than truncating/hiding chips or forcing horizontal scroll. This mirrors
  `entity-chips.tsx`'s own `flex flex-wrap items-center gap-1` container exactly.
- **Chip cap:** exactly 5 visible citations (`MAX_VISIBLE_CHIPS = 5`, per `39-CONTEXT.md`'s "~5
  with +N" discretion note, resolved to a hard 5 — mirrors `entity-chips.tsx`'s
  `MAX_VISIBLE_CHIPS` pattern, that file uses 4; this one uses 5 per the phase's own number).
  Dedupe citations by the composite key `${kind}:${id}` before slicing/capping (defensive — the
  per-tool `citations[]` envelope is already deduped server-side per Test 7 in `36-02-PLAN.md`,
  but a cross-round or future-tool duplicate should never silently render 2 identical chips).
- **Overflow "+N"** is a plain `<span>`, never a `<Link>` — non-interactive, matches
  `entity-chips.tsx`'s overflow badge exactly (no expand-on-click; the full-inspector affordance is
  explicitly deferred to studio/debug per `39-CONTEXT.md`).
- **Zero citations, zero results:** render the row with `completedLabel` only, no chip container at
  all (matches `entity-chips.tsx`'s "empty renders nothing" convention, scoped to the chip
  sub-list only — the label text itself still renders, see Empty States below).

### Anatomy — error (`isError: true`)

```tsx
<div role="alert" className="flex items-center gap-2 py-1 text-sm text-muted-foreground">
  <AlertTriangle className="size-4 shrink-0 text-destructive" aria-hidden />
  <span>{errorLabel}</span>
</div>
```

- No chips (an error result carries no `citations[]` — `content` is a plain-text error string, not
  the `{results, citations}` envelope).
- **No retry button.** Unlike `InlineErrorCard`, this is not user-actionable — the round loop
  already fed the error back to the model and it keeps generating (LOOP-02). Adding a retry control
  here would imply an action the UI doesn't actually support and duplicate what the model's own
  next visible text already does.
- The raw `content` string (Python's `"Tool execution timed out."` / `"Tool execution failed."` /
  an executor-specific message) is **never rendered verbatim** — always the fixed `errorLabel` copy
  below. Rationale: consistent with "full result JSON stays out of the transcript" extended to
  error text too — the model-facing string and the user-facing string are allowed to diverge, and
  fixing the user-facing string here means it never has to be re-audited as executor error text
  evolves.

### Anatomy — malformed content (defensive, `isError: false` but `JSON.parse(content)` throws)

`cap_tool_output`'s 2000-char truncation can cut a JSON envelope mid-token, producing invalid JSON
on the wire — this is a real, expected edge case, not hypothetical. `JSON.parse` MUST be
try/caught; on failure, render:

```tsx
<div className="flex items-center gap-2 py-1 text-sm text-muted-foreground">
  <span>{completedLabel} — details unavailable.</span>
</div>
```

No chips, no crash, no thrown error surfacing as a blank turn.

### Copy — tool-name → completed / error label maps

| `toolName` | Completed label (`{n}` = `results.length`) | Zero-results label | Error label |
|---|---|---|---|
| `lookup_entity` | "Looked up an entity — {n} results" (singular "1 result" at n=1) | "Looked up an entity — no results found" | "Couldn't look up that entity." |
| `search_emails` | "Searched emails — {n} results" | "Searched emails — no results found" | "Couldn't search emails." |
| `search_knowledge` | "Searched knowledge — {n} results" | "Searched knowledge — no results found" | "Couldn't search the knowledge graph." |
| any other / unrecognized | "Ran a lookup — {n} results" | "Ran a lookup — no results found" | "Couldn't complete that lookup." |

`n` = `JSON.parse(content).results?.length ?? 0`. Citation chips render from
`JSON.parse(content).citations` (array of `{kind, id, route}` — `route` is read but NOT used for
navigation, see `<ProvenanceLink>` below).

## `<ProvenanceLink>` (TUI-02)

**File:** `apps/web/src/components/provenance-link.tsx` (locked path, `39-CONTEXT.md`).
**Props:** `kind: "email" | "entity" | "knowledge"`, `id: string`, `label?: string`.

### Anatomy

```tsx
<Link
  href={hrefFor(kind, id)}
  onClick={(event) => event.stopPropagation()}
  className="inline-flex max-w-[160px] items-center gap-1 rounded-md border border-transparent bg-muted px-2 py-1 text-xs font-normal text-muted-foreground transition-colors hover:bg-accent hover:text-accent-foreground focus-visible:outline-none focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-1"
>
  <Icon className="size-3 shrink-0" aria-hidden />
  <span className="min-w-0 truncate">{label ?? fallbackLabel(kind, id)}</span>
</Link>
```

- **Real `<Link>`, never `onClick`-only.** Renders a genuine `<a href>` — middle-click/ctrl-click/
  new-tab all work natively, satisfying `39-CONTEXT.md`'s "real links" specific idea. The
  `onClick={stopPropagation}` guard mirrors `entity-chips.tsx`'s exact precedent (a chip inside a
  clickable row must not also trigger the row's own click handler) — harmless when the chip is NOT
  inside a clickable row (the common case in a transcript), required when it is (defensive,
  zero-cost to include).
- **Route computed internally, never trusted from the caller.** `hrefFor` is a fixed 3-way switch
  inside `provenance-link.tsx` — even though the server's `citations[]` envelope also carries a
  `route` string (`ToolCitation.route`), this component **recomputes** the href from `kind`+`id`
  rather than consuming that string directly. Rationale: `citations[]` is data that ultimately
  originates from a tool call in a model-adjacent pipeline; never let model-adjacent data drive a
  navigation target string directly — the SAME "compile-time switch, never model-authored"
  discipline `use-data-bindings.ts` (Phase 33) already established for tRPC procedure selection,
  applied here to route selection. One canonical routing table, in one file:

  ```ts
  function hrefFor(kind: "email" | "entity" | "knowledge", id: string): string {
    switch (kind) {
      case "email":
        return `/emails/${encodeURIComponent(id)}`;
      case "entity":
        return `/entities/${encodeURIComponent(id)}`;
      case "knowledge":
        return `/knowledge?focus=${encodeURIComponent(id)}`;
    }
  }
  ```

- **Size:** `rounded-md` (interactive-control radius family, per
  `docs/design/product-register-and-bans.md`'s radius allowlist — chips are explicitly listed as
  a `rounded-md`/`rounded-lg` consumer). `px-2 py-1` / `text-xs` (Label role, 12px/400/1.4) /
  `gap-1` between icon and label — same anatomy shape as `badge.tsx`'s `badgeVariants` base
  (`rounded-md border px-2.5 py-0.5 text-xs`), adapted for a link (focus-visible instead of focus,
  hover-fill instead of hover-darken).
- **Icon-per-kind (resolved: included, not omitted).** `39-CONTEXT.md` marks this optional; resolved
  to "include" for scannability in a mixed-kind chip row, reusing icons this app's own knowledge
  graph already assigned to the equivalent node kinds (`graph-nodes.tsx`) rather than inventing new
  ones:

  | `kind` | Icon | Precedent |
  |---|---|---|
  | `email` | `Mail` | `graph-nodes.tsx`'s `email` node |
  | `entity` | `Box` | `graph-nodes.tsx`'s `entity_instance` node |
  | `knowledge` | `Share2` | `graph-nodes.tsx`'s `knowledge_node` |

  Icon size `size-3` (12px) — matches `widget-status-badge.tsx`'s `ICON_CLASS` convention for a
  small inline badge icon.
- **Truncation:** `max-w-[160px]` on the `<Link>`, `truncate` + `min-w-0` on the inner `<span>`
  (the `min-w-0` is required — a flex child cannot shrink below its content's intrinsic width
  without it, a common truncation-in-flex bug). CSS truncation is visual-only; the full label text
  stays in the DOM, so the link's accessible name is never clipped for screen readers even when
  visually ellipsized — no separate `aria-label` needed.
- **Fallback label** (when `label` prop is omitted — the expected case for Phase 36/37's
  `ToolCitation(kind, id, route)`, which carries no label field): `` `${Capitalize(kind)} · ${id.slice(0, 8)}` ``
  — e.g. `"Email · a3f21b8e"`, `"Entity · 9c22f1a0"`, `"Knowledge · 5e1d88b2"`. Short, fits well
  inside the 160px cap without truncating in the common case, and gives the user SOME signal about
  what the chip resolves to pending a real label. Phase 41's knowledge-preview node (a future
  consumer) can pass a real `label` when it has one.

## Empty / Error States

| Scenario | Rendering |
|---|---|
| Round dispatched, executor running | `ToolRoundActivityRow` — spinner + gerund label (Component 1) |
| Round completes, `results.length > 0` | `ToolInvocationResultRow` success — label + count + up to 5 `<ProvenanceLink>` chips + optional "+N" |
| Round completes, `results.length === 0`, `isError: false` | `ToolInvocationResultRow` success — label + "no results found" text, no chip container rendered at all (cold-start-safe per `36-02-PLAN.md`'s own "never an error just because nothing matched" convention — this is NOT an error state) |
| Round completes, `isError: true` | `ToolInvocationResultRow` error — icon + fixed error copy, `role="alert"`, no chips, no retry button |
| Round completes, `content` fails `JSON.parse` (truncation edge case) | `ToolInvocationResultRow` degraded — label + "details unavailable.", no chips, no crash |
| Loop-level parse failure / round-cap exhaustion | **No new component** — already-existing `text` part rendering via `MarkdownRenderer` (Phase 34's `PARSE_FAILURE_TEXT`/`ROUND_CAP_EXHAUSTED_TEXT`) |
| A `tool_invocation` part (paired with the result that follows it) | Renders `null` — no visible output (see DO-NOTs) |

## Accessibility

- **Live region:** `ToolRoundActivityRow` carries `role="status"` (implicit `aria-live="polite"`)
  — its own visible label doubles as the screen-reader announcement, since the page-level
  `StreamState`-keyed announcer (`chat/page.tsx`) cannot observe intra-turn round boundaries.
- **Error semantics:** the error variant of `ToolInvocationResultRow` carries `role="alert"`,
  matching every other inline error surface in this app (`InlineErrorCard`,
  `interactive-widget-boundary.tsx`'s submit-error row) — but with reduced visual weight (neutral
  text, not bold `text-destructive`) per the severity rationale documented under Color above.
- **Non-color differentiation:** the error row is never signaled by color alone — `AlertTriangle`
  icon shape + presence carries the signal independent of the `--destructive` hue (WCAG 1.4.1).
- **Reduced motion:** the activity row's entrance (`motion-safe:animate-in motion-safe:fade-in`)
  and its spinner (`motion-safe:animate-spin`) both use the `motion-safe:` variant — fully absent
  under `prefers-reduced-motion: reduce` (the row still appears/disappears, just without the fade
  or spin; the `role="status"` text content is unaffected either way).
- **Focus:** `<ProvenanceLink>` uses `focus-visible:ring-2 focus-visible:ring-ring
  focus-visible:ring-offset-1` — keyboard-only focus indicator (never shows on mouse click),
  matching `badge.tsx`'s existing focus-ring convention adapted to `focus-visible`.
- **Accessible names:** `<ProvenanceLink>`'s CSS `truncate` never removes text from the DOM/
  accessible tree — full label always available to assistive tech regardless of visual clipping.
- **Contrast:** every color pair used (`muted`/`muted-foreground`, `accent`/`accent-foreground`,
  `destructive` against `background`) is already WCAG-AA-verified in `28-UI-SPEC.md`'s computed
  contrast pass (TOKEN-01) — no new pair introduced, so no new contrast computation is required
  this phase.
- **Security:** no `eval`/`Function`/`dangerouslySetInnerHTML` anywhere in this contract. The one
  untrusted-input handling point (`JSON.parse(part.content)`) is explicitly required to be
  try/caught with a defined fallback (see "malformed content" above) — this is a system-boundary
  validation point per the project's own input-validation guardrail, and it's spelled out exactly
  because `content` is server-tool output that has passed through an LLM-adjacent pipeline.
- **Visual verification:** connected-env/browser screenshot verification is deferred to
  execution/audit time per the v1.4/v1.5 precedent (`STATE.md` → Deferred Items) — mark
  `human_needed` if the playwright-core loop isn't available in the execution session.

## DO-NOTs (explicit, binding)

1. **No new npm dependency.** Everything above uses already-installed packages
   (`lucide-react`, `next/link`, existing Tailwind classes). Do not vendor a new chip/badge/tag
   registry component for this — `<ProvenanceLink>` is hand-authored per `39-CONTEXT.md`.
2. **No modal, no toast, no popover for the activity row or the result row.** Both are inline
   transcript content, always in-flow with the turn — `39-CONTEXT.md`: "Never a modal/toast."
3. **No second scrollbar.** The chip row uses `flex-wrap`, never `overflow-x-auto`/a nested
   scroll container. If a future phase needs more than "~5 chips + overflow," that's a new
   decision, not a scroll affordance bolted onto this one.
4. **Locked renderer files stay untouched.** `spec-renderer.tsx`, `render-node.tsx`,
   `genui-part-boundary.tsx` are not touched by this phase — the 3 new part types render via NEW
   branches in `message-turn.tsx`'s own part switch (which is not one of the 3 locked files),
   exactly like `interaction_result`/`interactive_widget` already do.
5. **No new design token.** Every class in this document resolves to an existing `globals.css`
   CSS variable or an unconfigured-but-already-used Tailwind utility (`size-3`, `size-4`, `gap-1`,
   `gap-2`, `rounded-md`, `truncate`, etc.). Do not add a new CSS custom property, a new
   `theme.extend` entry, or a new hand-authored keyframe for this phase.
6. **No full tool-result JSON in the transcript.** Neither row ever renders `content` verbatim
   (success or error) — only the derived label/count/chips. The debug/inspector view is explicitly
   deferred to studio (`39-CONTEXT.md` Deferred Ideas).
7. **`tool_invocation` parts render `null`.** Do not give this part type its own visible row — it
   would duplicate the narrative the paired `tool_invocation_result` row already carries.
8. **No retry button on a failed tool round.** The round loop self-recovers (LOOP-02); a retry
   control here would be a dead affordance.
9. **No round-count copy** ("2nd lookup…", "round 3 of 4") — resolved discretion, see Component 1.
10. **`<ProvenanceLink>` never trusts a caller-supplied route string.** Always recompute
    `href` from `kind`+`id` via the fixed internal switch — never accept or forward a raw `route`
    field from a citation envelope directly into `<Link href>`.
11. **No brand-accent (`primary`/teal) usage anywhere in this phase's new components.** Chips and
    rows stay entirely within the neutral (`muted`/`accent`/`ring`) + `destructive` palette — this
    phase's net contribution to the 10% accent budget is zero.
12. **`ALLOWED_PROCEDURES`/tRPC surface is out of scope** — this phase renders citation data
    already returned by tool executors; it does not call any new procedure.
