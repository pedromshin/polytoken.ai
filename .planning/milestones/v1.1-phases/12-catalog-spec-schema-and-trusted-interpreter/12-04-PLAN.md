<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| hardcoded demo spec → SpecRenderer | The demo specs are author-controlled (not LLM), but they flow through the SAME interpreter Phase 13's untrusted specs will — so the no-eval / isolation guarantees are exercised here against real specs. |
| @nauta/genui subpath → apps/web bundle | The client bundle imports the renderer via dynamic ssr:false; the bundle must not pull node-only modules (e.g. crypto) into the browser. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-14 | Tampering | demo spec rendered via interpreter | mitigate | SHOWCASE_SPEC/MALFORMED_SPEC pass through `SpecRenderer` → `renderNode` whose no-eval / per-node-boundary guarantees (Plan 03) hold; the demo cannot introduce an eval path because rendering is createElement-only. The malformed fixture exercises (does not bypass) the SPEC-03 boundary. |
| T-12-15 | Denial of Service | crypto import in client bundle (registry-version) | mitigate | `REGISTRY_VERSION` is consumed in the SERVER page.tsx (header chip), not inside the ssr:false island; if the bundler pulls `registry-version.ts` into the client, confirm the build succeeds (web:build gate) and the hash is import-time-stable. If a client-side hash error surfaces, isolate REGISTRY_VERSION usage to the server component (it already is). |
| T-12-16 | Information Disclosure | spec JSON inspector exposes internals | accept | The JSON inspector is a developer-only surface (UI-SPEC §6/§14) showing a hardcoded generic showcase — no PII, no secrets, no tenant data this phase (D-17). |
| T-12-SC | Tampering | npm installs | accept | No new third-party runtime packages — @nauta/genui is a workspace package; @nauta/ui resizable/scroll-area/badge already exist. No package-legitimacy checkpoint required. |
</threat_model>
