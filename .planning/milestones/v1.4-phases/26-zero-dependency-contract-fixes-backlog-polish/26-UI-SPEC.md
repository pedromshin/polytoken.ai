# Phase 26 — UI Design Contract

> Visual and interaction contract for correcting eleven drift-against-existing-tokens defects on
> `/chat` + `/studio`'s own hand-built chrome, plus two independent backlog fixes (declared-state
> text binding, canvas auto-layout). Generated by gsd-ui-researcher (fully autonomous session — no
> user prompts; every choice derives from `26-CONTEXT.md`'s locked `[auto]` decisions, the code-level
> audit in `.planning/research/CHAT-STUDIO-DESIGN-UPLIFT.md`, and a fresh read of every file the
> audit cites). Verified by gsd-ui-checker.

**This is a correction phase, not a new-surface phase.** Every fix below targets EXISTING tokens
that already exist in `apps/web/src/app/globals.css` / `packages/tailwind-config` — no custom
property is added, changed, or removed anywhere in this document (that is Phase 28's job). No new
npm dependency. No second brand hue. No third typography weight. Where a fix requires a genuinely
new visual affordance not covered by an existing token pairing (e.g. FIX-04's node differentiation,
FIX-09's dock shadow), the affordance is built ONLY from the existing `primary`/`muted`/`border`/
`destructive` token set and Tailwind's STOCK utility scale (no arbitrary hex, no arbitrary shadow
value).

**Explicit revision of a prior phase's decision:** `23-UI-SPEC.md` locked "never special-case chat"
(byte-identical `ChatNode`/`GenuiPanelNode` shells). FIX-04 supersedes that ONE clause per
`26-CONTEXT.md`'s explicit `[auto]` decision — everything else `23-UI-SPEC.md` established (node
minimum dimensions, drag-handle convention, selection ring, streaming dot) is unchanged.

**Non-interference (unchanged from Phase 24):** `packages/genui/src/renderer/spec-renderer.tsx` is
UNMODIFIED. `GenuiPartBoundary`/`InteractiveWidgetBoundary`'s own chrome (owned by Phase 24) is not
touched by this phase — `ChatNode`/`GenuiPanelNode` shells wrap these components but never reach
inside them.

## Design System

| Property | Value |
|----------|-------|
| Tool | none — no `components.json` / shadcn CLI-managed registry exists in this repo (same posture as every prior chat/studio UI-SPEC). `packages/ui/src/*` (`@nauta/ui`) is the hand-maintained component set. Do not run `shadcn init`. |
| Preset | not applicable |
| Component library | Radix UI primitives wrapped by `@nauta/ui` (shadcn pattern, CVA variants) |
| Canvas library | `@xyflow/react` (React Flow) + `@dagrejs/dagre` layout — both already dependencies (Phase 23) |
| Icon library | `lucide-react` — sole icon set on both surfaces (already verified consistent by the research doc's gap-check; this phase adds new icons only from this same set) |
| Font | System default sans stack (Tailwind `font-sans`, `apps/web/tailwind.config.ts:17-20`) — unchanged, no webfont this phase |
| Styling engine | Tailwind **v3.4.4** (confirmed, NOT v4) — matters for any CSS syntax added by FIX-01/FIX-10 (must be v3-compatible `@layer`/`@apply`, never v4 CSS-first syntax) |

Existing tokens this phase consumes as-is (`globals.css` `:root`/`.dark`, values unchanged):
`background`, `foreground`, `card`, `popover`, `primary` (teal, `hsl(164 39% 22%)`, IDENTICAL in
light/dark), `secondary`, `muted`, `accent` (the latter three are byte-identical stock grays in both
modes today — a known Phase 28 gap, not fixed here), `destructive`, `border`, `input`, `ring`. All
consumed via Tailwind's HSL-var utility classes only — never a hardcoded hex/named Tailwind palette
color (`amber-*`, `red-*`, `emerald-*`, `white` are the exact violations this phase removes).

```tsx
  readonly body: string;
  readonly layout?: "centered" | "inline"; // "centered": full-pane vertical center (default);
                                            // "inline": compact icon+text row for a bounded container
  readonly tone?: "muted" | "destructive"; // icon/border tint; default "muted"
  readonly size?: "compact" | "spacious"; // icon-size + heading-size register; default "compact"
  readonly action?: {
    readonly label: string;
    readonly icon?: LucideIcon;
    readonly onClick: () => void;
    readonly disabled?: boolean;
  };
  readonly caption?: string; // small text-xs text-muted-foreground line below body (unknown-node-type's "Type: {x}" line)
}
```

### POLISH-01 — Declared-state display via `dataRef`-bound nodes (prompt-only, no visual chrome)

**Scope note:** this is a generator-PROMPT fix in the Python email-listener app
(`apps/email-listener/app/infrastructure/llm/genui_generator_adapter.py`'s `_SYSTEM_PROMPT_TEXT` +
`packages/genui/artifacts/genui-prompt.json`'s catalog reference) — it produces NO new UI chrome and
touches NO renderer code. `packages/genui/src/schema/spec-schema.ts`'s `text` node has only
`content: z.string()` (no `dataRef`); only `list` (`dataRef` → iterate an array) and `conditional`
(`dataRef` → branch on a value) resolve live state at render time
(`packages/genui/src/renderer/render-node.tsx`'s `resolveDataRef`). The bug (999.8): the generator
currently emits `{"type":"text","content":"{{count}}"}` — a static string the (locked, unmodified)
`SpecRenderer` never interpolates — even though the paired `button.onClick:{type:"setState",

**Design-contract implication (the only part in this researcher's remit):** whatever declarative
pattern the prompt is taught to emit (e.g. a `conditional` chain keyed on `state.<name>` with
threshold/equality branches, each branch a plain `text` node with its own static, realistic copy —
NOT a `{{mustache}}` placeholder) must render through the EXISTING, already token-compliant
`SpecRenderer` output — no new visual treatment, no new node type, no styling task. Any regression
fixture/test this produces (per `26-CONTEXT.md`: "a counter bound to state" prompt shape → a
`dataRef` binding) is a Python/pytest concern, not a UI-SPEC concern. Nothing in this section
requires planner/executor UI work beyond verifying the existing render output stays token-compliant
(it will, since `conditional`/`list` already render through the unmodified, compliant renderer).
