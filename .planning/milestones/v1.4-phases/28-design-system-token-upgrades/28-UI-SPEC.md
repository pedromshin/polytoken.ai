## Color

### TOKEN-01 — `secondary`/`muted`/`accent` tonal differentiation

**Role contract (from `28-CONTEXT.md`, locked):** one neutral family, hue **164** (the primary's own
hue), saturation whisper ≤8-12% so it reads cohesive with the teal brand without becoming a second
hue. `muted` = the recessive fill (closest to today's value). `secondary` = a step more
present/darker (real fills — table header bands, secondary buttons). `accent` = the interactive-hover
tint (slightly more teal-leaning than the other two — this is shadcn's hover-fill semantic slot, see
the naming-collision note below, NOT this app's 60/30/10 "accent" color).

**⚠️ Naming-collision note (read before auditing 60/30/10):** this app's own accent-allowlist
(established Phase 22, reused by every prior UI-SPEC) reserves the word "accent" for `primary`
(teal, `hsl(164 39% 22%)`) — the ONE saturated brand color used at ≤10% coverage for CTAs/selection/
state. shadcn's `--accent`/`--accent-foreground` CSS variables are a **different, pre-existing
semantic slot** (the neutral hover-fill Radix components like `SelectItem`/`CommandItem`/
`DropdownMenuItem` paint on `data-[highlighted]`) — it has always existed in this codebase, was
already byte-identical-gray before this phase, and stays a **neutral** (whisper-teal-tinted) color
after this phase. Widening its usage via existing `hover:bg-accent` call sites does **not** expand
this app's 10% brand-color budget — `--accent`'s post-TOKEN-01 saturation (10-12%) keeps it
unambiguously in the neutral family, never a second brand hue.

#### Final values

| Token | Light (before → after) | Dark (before → after) |
|-------|--------------------------|--------------------------|
| `--muted` | `0 0% 96.1%` → **`164 6% 95.3%`** | `0 0% 14.9%` → **`164 6% 15%`** |
| `--muted-foreground` | `0 0% 45.1%` → **`164 5% 41.5%`** | `0 0% 63.9%` → **`164 6% 65%`** |
| `--secondary` | `0 0% 96.1%` → **`164 8% 93.5%`** | `0 0% 14.9%` → **`164 8% 17.7%`** |
| `--secondary-foreground` | `0 0% 9%` → **`164 4% 11%`** | `0 0% 98%` → **`164 10% 97%`** |
| `--accent` | `0 0% 96.1%` → **`164 10% 94.3%`** | `0 0% 14.9%` → **`164 12% 17.5%`** |
| `--accent-foreground` | `0 0% 9%` → **`164 6% 12%`** | `0 0% 98%` → **`164 8% 97%`** |

Role read confirmed by the numbers: light-mode lightness order is `muted (95.3) > accent (94.3) >
secondary (93.5)` — muted is the lightest/most recessive, secondary is the darkest/most present
(table header bands, per `28-CONTEXT.md`), accent sits between the two but carries the highest
saturation (10%) of the three, marking it as the "livelier" interactive tint. Dark-mode order is
inverted correctly for a dark surface: `muted (15) < secondary (17.7) ≈ accent (17.5)` — secondary/
accent are *lighter* than muted in dark mode (a raised/more-present surface must be *lighter*
against a near-black background, not darker — mirrors how `--card`/`--popover` already sit at the
same lightness as `--background` while the browser still tells them apart via `--border`).

#### Maximum-drift guard (`28-CONTEXT.md`: "the change must read as the same app, slightly more
defined, not a retheme")

All 6 values stay within **≤2.8 lightness points** of their shared pre-phase baseline (light `96.1%`,
dark `14.9%`) — well under the ≤3-point ceiling:

| Token | Mode | Δ from baseline |
|-------|------|------------------|
| muted | light | 0.8 |
| secondary | light | 2.6 |
| accent | light | 1.8 |
| muted | dark | 0.1 |
| secondary | dark | 2.8 |
| accent | dark | 2.6 |

Saturation stays ≤12% throughout (well inside the "whisper of teal" ceiling implied by `28-CONTEXT.md`'s
"hue 164 whisper ≤8% sat" guidance for `muted`, extended slightly to 10-12% only for `secondary`/
`accent`'s more-present/interactive roles — still far from `primary`'s own 39%).

### TOKEN-03 color note (shadow tint)

The elevation ambient layer's light-mode color (`hsl(164 25% 15% / …)`) is a dark, moderately
desaturated teal — chosen so the shadow reads as "cool/brand-tinted" rather than pure black, per
`28-CONTEXT.md`'s "ambient tinted with the teal hue at very low alpha instead of pure black." Dark
mode deliberately uses neutral black (`hsl(0 0% 0% / …)`) instead — a colored shadow is imperceptible
against a near-black (`3.9%` L) surface, so forcing a teal tint there would be invisible (i.e.
purely theoretical) while still constraining what "reduce shadow reliance" can mean; neutral black
at modest alpha is the pragmatic, honest choice. See full recipe under TOKEN-03 below.

## TOKEN-03 — Elevation/Shadow Scale

**Numeric backing (cited per `28-CONTEXT.md`'s explicit deferral of this citation to this phase):**
`.planning/research/CHAT-STUDIO-DESIGN-UPLIFT.md` records styles.refero.design's shadcn/ui "clinical
blueprint on frosted paper" style extraction as carrying harder numeric backing this repo lacked: a
**stacked hairline shadow recipe** (a 1px hairline layer + a soft ambient layer, rather than one flat
shadow) and a **two-value radius allowlist** (see TOKEN-04). This elevation scale adopts the stacked-
hairline shape directly: every level below stacks a `0 0 0 1px` hairline layer (using the existing
`--border` token — no new color) underneath a soft ambient layer.

**Compliance constraint (hard ceiling, cross-checked against
`docs/design/product-register-and-bans.md` item 9, the "ghost-card" ban — "pairing a 1px border with
a soft, WIDE (≥16px blur) drop shadow... is banned... pick one signal... or a defined shadow at ≤8px
blur"):** `packages/ui/src/card.tsx` already has a real 1px `border` AND is about to receive a shadow
— to stay inside the ban's own explicit exception, **no elevation layer's blur radius may exceed
8px**. Verified below: the largest blur used anywhere in this scale is exactly `8px`
(`elevation-3`'s second layer) — at the ceiling, never over it.

### CSS custom properties (`apps/web/src/app/globals.css`)

Add inside the existing `:root { … }` block (after `--sidebar-ring`):

```css
--elevation-1: 0 0 0 1px hsl(var(--border) / 0.6), 0 1px 2px 0 hsl(164 25% 15% / 0.06);
--elevation-2: 0 0 0 1px hsl(var(--border) / 0.6), 0 2px 4px 0 hsl(164 25% 15% / 0.08),
  0 1px 2px -1px hsl(164 25% 15% / 0.04);
--elevation-3: 0 0 0 1px hsl(var(--border) / 0.6), 0 4px 8px 0 hsl(164 25% 15% / 0.10),
  0 2px 4px -2px hsl(164 25% 15% / 0.06);
```

Add inside the existing `.dark { … }` block (after `--sidebar-ring`):

```css
--elevation-1: 0 0 0 1px hsl(var(--border)), 0 1px 2px 0 hsl(0 0% 0% / 0.24);
--elevation-2: 0 0 0 1px hsl(var(--border)), 0 2px 4px 0 hsl(0 0% 0% / 0.32),
  0 1px 2px -1px hsl(0 0% 0% / 0.20);
--elevation-3: 0 0 0 1px hsl(var(--border)), 0 4px 8px 0 hsl(0 0% 0% / 0.40),
  0 2px 4px -2px hsl(0 0% 0% / 0.24);
```

Note: dark mode's hairline layer uses `hsl(var(--border))` at full alpha (dark `--border` is already
a low-contrast near-surface value, `0 0% 14.9%`, so it doesn't need the `/0.6` softening light mode
needs against a much brighter `--border`, `0 0% 89.8%`).

### Named consumers (exactly 4 — all already carry a `shadow-*` class today, all trivially-clean swaps)

| File | Before | After |
|------|--------|-------|
| `packages/ui/src/card.tsx:12` | `"rounded-xl border bg-card text-card-foreground shadow"` | `"rounded-xl border bg-card text-card-foreground shadow-elevation-1"` |
| `apps/web/src/app/chat/_components/composer.tsx:76` | `"w-full shrink-0 border-t border-border/60 bg-background shadow-sm"` | `"w-full shrink-0 border-t border-border/60 bg-background shadow-elevation-2"` |
| `apps/web/src/app/chat/_canvas/chat-node.tsx:150` | `` `...bg-background shadow-sm transition-shadow duration-150${selected ? ` ${SELECTED_RING}` : ""}` `` | `` `...bg-background transition-shadow duration-150 ${selected ? `${SELECTED_RING} shadow-elevation-2` : "shadow-elevation-1"}` `` |
| `apps/web/src/app/chat/_canvas/genui-panel-node.tsx:157` | `` `...bg-background shadow-sm transition-shadow duration-150${selected ? ` ${SELECTED_RING}` : ""}` `` | `` `...bg-background transition-shadow duration-150 ${selected ? `${SELECTED_RING} shadow-elevation-2` : "shadow-elevation-1"}` `` (then TOKEN-05 classes append — see below) |

**Rationale for the resting/selected split on the two canvas node shells:** both files already carry
an inert `transition-shadow duration-150` class (present since Phase 23, never actually driven by a
changing shadow value until now) — this phase gives it a real job: resting = `elevation-1`, selected
= `elevation-2` (a visible "lift" reinforcing the existing `ring-2 ring-primary` selection idiom, not
replacing it). **Composer's dock effect:** `elevation-2`'s ambient layers use `y`-offsets of `+2px`/
`+4px` with blur `4px`/`8px` respectively — because blur radius extends the shadow in all directions
regardless of offset sign, and the composer sits flush at the viewport's bottom/side edges (nothing
renders below/beside it to catch a downward shadow), the visible portion of this shadow reads
entirely as a soft rise ABOVE the composer's `border-t` — i.e., a "top shadow" effect, achieved
without inventing a 4th, negative-offset-only shadow token.

--radius-2xl: calc(var(--radius) + 8px); /* = 16px at the current 8px base */
```

### Tailwind config (`packages/tailwind-config/web.ts`, inside the existing `borderRadius` object —

borderRadius: {

```

### `card.tsx` — the "swap" that requires **zero className string change**

Tailwind's own (unconfigured) default theme already defines `xl: 0.75rem` (12px) and `2xl: 1rem`

`card.tsx:12` does not change** — the swap is entirely in `web.ts`'s config, which redirects what
that class name means. This is the exact "VISUALLY IDENTICAL, now token-driven" outcome

## TOKEN-05 — Entrance/Stagger Animation

**Zero new CSS.** Every class used below (`animate-in`, `fade-in-0`, `zoom-in-95`,
`slide-in-from-bottom-1`, `duration-[…]`, `motion-reduce:animate-none`) is already generated by the
already-registered `tailwindcss-animate` plugin (`packages/tailwind-config/web.ts`) or Tailwind core
(`duration-[…]` arbitrary values, the `motion-reduce:` variant) — this phase is pure component-level
`className`/inline-`style` wiring, no `globals.css` change.

**Reduced-motion gating:** `tailwindcss-animate`'s utilities ARE real Tailwind utilities (unlike
Phase 27's hand-authored `.generating-ring`, which lives outside Tailwind's utility system and
therefore cannot take a `motion-safe:`/`motion-reduce:` prefix) — so Tailwind's own
`motion-reduce:animate-none` variant works directly here, setting `animation: none` and fully
canceling the entrance keyframe under `prefers-reduced-motion: reduce`. Every consumer below carries
this suffix.

**No stacking with Phase 27's `.t-*`/`GeneratingRing`** (the "one animation per reveal" precedent,
`28-CONTEXT.md`): none of the 3 consumers below currently carry a `.t-*` class or `.generating-ring`
— `GenuiPanelNode` is explicitly NOT a `GeneratingRing` consumer (that scope exclusion was locked in
`27-UI-SPEC.md` ADOPT-03 mount (c)), so there is no conflict to resolve.

### (a) Genui panel mount — `apps/web/src/app/chat/_canvas/genui-panel-node.tsx`

Applied to the **`GenuiPanelNode` outer shell div** (the top-level returned element, NEVER inside
`GenuiPanelNodeBody`/`GenuiPartBoundary`/`InteractiveWidgetBoundary`/`SpecRenderer`). This fires once
per node MOUNT (React reuses the same DOM node across re-renders as long as the node isn't
unmounted/remounted — drag ticks, streaming updates, and `selected` toggles all re-render this
component without recreating its DOM node, so the entrance never replays mid-drag or mid-stream;
`GenuiPanelNodeBody` is already `memo`'d specifically to keep drag/stream re-renders cheap, per the
file's own doc comment — this phase does not touch that mechanism).

Final className (combining the TOKEN-03 elevation swap + this phase's entrance classes):

```tsx
className={`flex h-full min-h-[240px] w-full min-w-[320px] flex-col overflow-hidden rounded-lg border border-border/60 bg-background transition-shadow duration-150 animate-in fade-in-0 zoom-in-95 duration-[250ms] motion-reduce:animate-none ${selected ? `${SELECTED_RING} shadow-elevation-2` : "shadow-elevation-1"}`}
```

250ms duration matches Phase 27's `.t-modal-reveal` (also a content-reveal moment); `zoom-in-95`
(scale 0.95→1) + `fade-in-0` (opacity 0→1) mirrors the same scale-and-fade shape as `.t-modal-reveal`/
`.t-dropdown-reveal` without literally reusing either class (this is a React Flow node, not a Radix
dialog/popover — a distinct component, deliberately not force-fit into the `.t-*` set).

**No stagger on panel mount** — `28-CONTEXT.md` reserves stagger for the Studio list items only
(item (b) below); if `POLISH-02`'s auto-layout drops several panels on the canvas at once, each
mounts (and therefore animates) independently on its own React Flow insertion tick — simultaneous,
not staggered, and visually unproblematic for the 2-4-panel case this app actually produces.

### (b) Studio list items — stagger, capped at 6

**Formula (both consumers):** `delayMs = Math.min(index, 5) * 40` → delays `0, 40, 80, 120, 160,
200`ms for the first 6 items (index 0-5), then flat `200ms` for every item beyond (the "capped ~6
items" instruction from `28-CONTEXT.md`). Total cascade spread tops out at `200ms` (stagger) +
`200ms` (each item's own animation duration) = `400ms` worst case for the last visible item — the
PER-ITEM duration (`200ms`) stays inside impeccable's "150-250ms on most transitions" ceiling
(`docs/design/product-register-and-bans.md`'s Motion guidance); only the aggregate cascade exceeds
it, which is the expected, accepted shape of any staggered-list reveal (a single transition's
ceiling does not bound a multi-item cascade's total spread).

Shared per-item classes: `animate-in fade-in-0 slide-in-from-bottom-1 duration-200
motion-reduce:animate-none`, plus an inline `style={{ animationDelay: "<delayMs>ms" }}` (the delay
value is per-row/per-card DATA, not a static class — Tailwind's JIT scanner cannot pick up a
dynamically-interpolated class name like `` `delay-[${n}ms]` ``, so `animationDelay` is set via
inline style, the standard escape hatch for this exact case).

**Consumer 1 — `apps/web/src/app/studio/_components/history-island.tsx`** (the `<ul>` inside
`HistoryMasterList`): change `rows.map((row) => …)` to `rows.map((row, index) => …)` and apply the
classes/style to the `<li>` wrapper:

```tsx
<li
  key={row.id}
  role="listitem"
  className="animate-in fade-in-0 slide-in-from-bottom-1 duration-200 motion-reduce:animate-none"
  style={{ animationDelay: `${Math.min(index, 5) * 40}ms` }}
>
  <HistoryRow … />
</li>
```

**Consumer 2 — `apps/web/src/app/studio/_components/page-ideas-island.tsx`** (`IdeaCard`, mapped in
the grid): add an `index: number` prop to `IdeaCard` and apply the classes/style directly to its
`<Card>` element (the grid's `filtered.map((idea) => …)` becomes `filtered.map((idea, index) => …)`,
passing `index` through):

```tsx
<Card
  className="flex flex-col gap-0 animate-in fade-in-0 slide-in-from-bottom-1 duration-200 motion-reduce:animate-none"
  style={{ animationDelay: `${Math.min(index, 5) * 40}ms` }}
>
```

Both consumers already re-render their full list on every filter/page change (React remounts new
`key`s when the filtered set changes), so the stagger naturally replays whenever the visible set
changes — this is the correct, expected behavior for a filter/paginate interaction, not a bug to
guard against.
