## Overview

Epoch A of the two-epoch endgame ([research/two-epoch-endgame/ENDGAME-PLAN.md](../research/two-epoch-endgame/ENDGAME-PLAN.md) §2).
The intent: polytoken becomes a *used* product — Band 1 closes the live loop on the user's real
email FIRST, gating Band 2 (total UI re-skin, editable genui panels, mobile-responsive answer)
and Band 3 (the E3 email-cluster workflow, depth-first, proven live on the real inbox).

**Outcome against that intent — read this before trusting the phase checkmarks below.** All six
phases executed every plan and all 27 requirements are code-complete and independently verified
at code level. But the milestone shipped with its three live-acceptance legs unexecuted: Google
OAuth on the deployed app (LIVE-03, §A), a real email actually flowing in (LIVE-04, §B.3–6), and
the six-leg cluster scenario on the real inbox (CLUS-07, §H) — which was *this milestone's
declared acceptance bar*. The audit verdict was `gaps_found` and said complete-milestone must not
run until those were done; the user chose to proceed anyway and accept them as tech debt on
2026-07-14, knowingly overriding v1.9's own STANDING RULE ("a milestone isn't done until the user
has touched the capability live").

So: **Band 1 was never fully green, yet Bands 2 and 3 shipped on top of it.** The hard ordering
constraint held on paper (code-level verification) but not in the sense the rule was written to
protect. None of the three gaps is missing work — code, infrastructure, migrations (through 0036,
applied to local + staging + prod) and the runsheets are complete and waiting in
[MORNING-CHECKLIST.md](../phases/49-live-loop-gate-deploy-oauth-real-email/MORNING-CHECKLIST.md).
Executing §A → §B.3–6 → §H closes all three with no further development.

## Milestone Summary

**Decimal Phases:** none — v1.9 ran 49–54 as straight integers, no urgent insertions.

**Key Decisions:**

- **D-49-05 — committed palette-ban gate:** the re-skin's token discipline is enforced by a checked-in `palette-ban.test.ts` regression test, not by review discipline. Rationale: the Phase-48 audit found off-token stragglers precisely where a grep-scoped manual sweep had missed them.
- **§E.2 — coverage gate ratcheted 80 → 65 (user-approved 2026-07-12):** the email-listener pytest gate blocked every ECS image deploy at 68.10% actual. Ratcheted rather than auto-lowered silently, with a step-up ladder (70@72% / 75@77% / 80@82%, never lower further) tracked as a todo. Rationale: unblock deploys without pretending the debt away.
- **AWS/Terraform resource renames re-parked (explicit, not drifted):** hazard A (two unsynced sources of truth — Terraform vars vs hardcoded GitHub Actions env), hazard B (ECR `force_delete=false` destroy+recreate risk), hazard C (local-only un-backed-up tfstate). High blast radius, zero user-facing value.
- **GitHub rename executed with its OIDC apply in the same sitting (2026-07-13):** renaming the repo breaks the IAM trust policy pinned to the old path, so the rename and `terraform apply` had to be one operation to avoid a deploy-pipeline outage.
- **Feature-detected schema columns (`tableColumnExists`):** every read/write of a column whose migration may not be applied is gated by a per-process-cached probe that fails closed to "not present". Rationale: 0036 shipped to code before it was applied anywhere; the app had to degrade cleanly rather than 500.
- **`web_search` shipped dark, then flipped:** exposure was code-gated behind `WEB_SEARCH_TOOL_ENABLED` until the 10-fixture adversarial prompt-injection suite passed — the same discipline as the v1.6 tools.
- **Mobile = don't mount, not just hide:** below `md`, the React-Flow canvas islands on `/chat` and `/knowledge` are never mounted (not CSS-hidden), per the market-validated pattern (ChatGPT removed Canvas 2026-05-28; Claude Artifacts render inline on mobile).

**Issues Resolved:**

- **Prod tasks crashed at import** — `requirements.txt` had drifted from `pyproject.toml`, so every prod ECS task died on `ModuleNotFoundError: jsonschema` and the circuit breaker rolled back each rollout. This was invisible until the coverage ratchet let images actually deploy. Fixed and guarded by `tests/test_requirements_sync.py`.
- **Canvas node loss on reload** — an async `setNodes` updater race in `chat-canvas.tsx` silently dropped every restored canvas node beyond the default chat node. Found by the Phase-50 UAT burn-down, i.e. by actually exercising the thing.
- **Three live chat bugs** found during the §G.2 E2E re-run: genui_spec replay 400 (`e947cd3`), multi-tool-call mangling (`949a04c`), final-round stranding (`065dda0`).
- **`chat_cost_ledger` NOT NULL user_id bug** (`bf6014c`).
- **E2E suite 7 failures** root-caused to test-suite topology bugs — zero product regressions — then green 32/32 twice.
- **Regenerated panel reset its style pack** (Phase 52) and a `/login` brand-copy drift (Phase 51), both caught in-phase.

**Issues Deferred:**

- **LIVE-03 / LIVE-04 / CLUS-07** — the three user-only live legs (§A, §B.3–6, §H). See the Overview above; this is the milestone's defining caveat.
- **999.17 — editable-panel chrome unreachable on mobile:** Phase 52 wired overlays/toolbar only into the canvas node; Phase 53 never mounts the canvas below `md`, and the docked/mobile transcript path reads raw `chat.getHistory` parts. PANL-01..04 editing is desktop-canvas-only, and canvas-side edits don't surface in the docked view of the same conversation. Graceful degradation, but an undocumented cross-phase trade-off — a real cost of running 52 and 53 in parallel.
- **Orphaned conversation on attach-chat failure** (`email-thread-node.tsx:103-125`) — created before attach; on degrade it is neither deleted nor opened. Cosmetic rail clutter, no corruption.
- **§E.1 — hosted DB passwords stale (28P01)** in `.env.staging`/`.env.production`; the Management-API path has captured PASS evidence, so this is low priority.

**Technical Debt Incurred:**

- Coverage gate at 65 for `apps/email-listener` (step-up ladder tracked in `todos/pending/2026-07-12-coverage-ratchet-step-up.md`).
- Playwright viewport specs (360/768/1024) not authored — the mobile sweep stayed manual (§G item 6), so MOBL has no CI lock-in.
- `/login` renders inside the authenticated `SidebarProvider` shell (pre-existing, self-flagged in 53-01) — future security pass.
- `web_search` provider is keyless DuckDuckGo scraping; a keyed provider (Tavily/Brave) is a documented optional swap.

- Display typography role (`text-xl`) declared but unused — page titles fragment across text-lg/text-2xl (inherited).
- Vercel project still named `nauta-web` (rename documented as optional/skippable; zero hardcoded refs in application code).

**The finding that outlived the milestone — backlog 999.18 (HIGH PRIORITY):** the user's verdict on
the live app was *"the whole UI is still ugly/experimental, not a production UI — not just tokens
and colors."* Phase 51 "Total UI Re-skin" was scoped and executed as token-hygiene only (its own
UI-SPEC said "refinement, not redesign"); no phase in the project's history has ever done actual
visual/UX design. The base palette is still stock-shadcn, and the brand guide defines the
"polytoken register" only as voice/tone — no visual identity was ever specified. A full production
UI/UX rebuild is the top candidate for the next milestone, with v1.9's 16-surface screenshot
harness and committed token gates as the regression rails. Companion: 999.19, the user's own
description of the target product (auto-collected sources, user-selected canon, edges-as-context).
