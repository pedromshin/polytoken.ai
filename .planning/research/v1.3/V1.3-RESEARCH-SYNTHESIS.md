## R1 — Unified vs hybrid design generation + screenshot→UI

**How arbitrary-design tools actually reproduce "any design":**
- **v0 (Vercel) = code-emit** `[FRESH✓]`. Its Platform API returns `latestVersion.files[]` — raw React/TSX source strings, not a spec. Screenshot input is a first-class `attachments` type (`screenshot`/`figma`) → "analyze the layout, colors, and components… generate code that closely replicates the design." The internal vision→code pipeline is undocumented.
- **v0 rendering = Vercel Sandbox (Firecracker microVM)** `[FRESH✓]`. v0 **abandoned its old iframe/srcdoc preview** because it couldn't run server-side code (API routes, DB, env). Bolt uses **StackBlitz WebContainers** (WASM in-browser Node). tldraw "Make Real" = code-emit HTML/CSS rendered in an iframe on canvas `[CORPUS]`.
- **Declarative-spec products exist and work for chat-genui but are catalog-bounded:**
  - **Thesys C1 / Crayon** `[FRESH✓]`: OpenAI-compatible API returns a "C1 DSL" (XML-like structured string); `<C1Component>` parses → live React from **pre-registered components referenced by name** (Zod→JSON-schema sent to the model). **No sandbox** — because no executable code is ever emitted. This is a commercial mirror of our engine.
  - **Google A2UI = declarative JSON + trusted catalog; MCP Apps = sandboxed iframe code**; Google explicitly advocates a **hybrid** `[CORPUS]`.
  - **Microsoft Adaptive Cards** `[FRESH✓]` — declarative-JSON-over-fixed-trusted-catalog, in production since 2017, explicitly citing "no code execution risk." Together with A2UI, **two major vendors independently validate our `packages/genui` declarative core.**
- **Reliability ceilings (hard numbers)** `[FRESH✓]`: v0's specialized AutoFix pipeline reaches **93.87% error-free** (vs 78.43% bare Claude Opus) — and "error-free" = compile/lint cleanliness, **not** design fidelity. **Design2Code (Stanford, 484 real webpages): best commercial models only ~0.27–0.28 CW-SSIM** (pixel similarity) + 80–89% element match; dominant failure = layout/element **recall**. OSS screenshot-to-code tools all converged on a **render-and-compare feedback loop**, not one-shot — repair loops are structurally necessary for design fidelity.
- **Decisive answer to "should we unify?"** `[FRESH✓]`: **no production system does BOTH declarative-safe and code-emit-expressive through one unified engine** — the market has bifurcated into two camps, each accepting the other's weakness. Full R1 report: `.planning/research/v1.3/R1-DESIGN-GENERATION-ARCHITECTURE.md`.
- **Screenshot→UI is real but imperfect** `[FRESH✓]`: WebSight (arXiv 2403.09029, HuggingFace) — synthetic-only training data (823K→1.92M pairs, Tailwind), a fine-tuned VLM; **no clean published accuracy numbers**; the real-world domain gap is the dataset's founding premise. Full-fidelity "clone any wildcard site" is not a solved problem.
- **Design-token extraction — a sharp reliability split** `[FRESH✓]` (feeds v1.4): from a **live URL, DOM/CSS-based DTCG extraction is a solved, OSS-available path** — designlang (MIT CLI, headless-Chromium DOM → W3C-DTCG + Tailwind/CSS-vars/Figma/native), @projectwallace/css-design-tokens (npm, CSS→DTCG types), extract-design-system (Playwright + agent-skill → `tokens.json`), Superposition, MiroMiro. From a **raster screenshot it's vision-model and fuzzier** — Peel (Claude/Gemini backend → colors/type/component-styles), Codia VisualStruct (CV → token JSON). **Implication:** "paste a URL" → reliable off-the-shelf DTCG tokens (we already consume DTCG — Phase 17); "drop a screenshot" → vision extraction, lower precision. Prefer URL/DOM input where possible; treat raster-only as best-effort.

**Promptable design system — real, documented prior art** `[FRESH✓]`:
- shadcn **`registry.json` / `registry-item.json`** = "a distribution system for custom code" carrying components, `cssVars`, `tailwind`, deps.
- v0 frames its registry as **"a distribution specification designed to pass context from your design system to AI models"**, now evolved into **"Design Systems 2.0" — a design-system *skill/adapter*** telling the model which components/props/tokens are safe to use (old shadcn-registry flow labeled "legacy"). This is almost exactly the user's "promptable design system the builder agent sets params on."
- Tokens-as-data: **W3C DTCG spec is stable (2025.10)** `[CORPUS]` — we already ship 6 DTCG style packs (Phase 17).

**Repair loop + eval-driven dev — validated** `[FRESH✓]`: v0 layers (1) mid-stream "LLM Suspense" live fixes (~100ms, e.g. bad icon imports via vector search), (2) post-stream deterministic + small AST-aware autofixers (<250ms), (3) a dedicated RL-tuned `vercel-autofixer-01` (86.14% error-free, single-pass; Fireworks) — all under eval-driven development. Mirrors our Phase-16 eval harness + Phase-20 repair loop (validate→autofix→run→heal≤2→fallback).

### Architecture read (the unify-vs-hybrid question)
The evidence is decisive on one point: **"reproduce any design system / wildcard screenshot" fundamentally requires code-emit** — a fixed declarative catalog (ours, C1, A2UI) can only compose *pre-registered* components and therefore *cannot* clone an arbitrary design. The whole arbitrary-design industry (v0/Bolt/tldraw) is code-emit.

But that does **not** mean collapse to one path. The right frame is a **refined hybrid with an explicit routing rule + a shared conditioning layer**, with the center of gravity shifted toward code-emit for the *appearance* use-case:
- **Code-island (code-emit, jailed) = primary path for design/appearance/arbitrary-layout** ("make it look like X", "clone this screenshot"). Our existing iframe/srcdoc jail is adequate for **client-only UI** (which is what design-reproduction mostly is); Sandpack/Vercel-Sandbox is the documented upgrade path if full-app/npm/server fidelity is later needed (v0's exact reason for leaving iframe). See [[code-island-sandbox-architecture]].
- **Declarative catalog = path for the chat's own interactive UI** — the **dual-channel widgets** (forms, pickers, proposal cards, data displays) where reliability, caching, and a safe widget→agent round-trip matter more than pixel-arbitrary styling. C1 proves this path is production-viable for chat-genui.
- **Promptable design system (shadcn-registry / Design-Systems-2.0-style adapter) = the shared conditioning input BOTH paths consume.** A screenshot→design-rules extraction produces tokens+rules that theme the declarative renderer AND condition the code-island generator.

**Recommendation (to confirm, not yet locked — user chose research-first):** **keep hybrid, do NOT collapse to one engine.** Both R1 tracks (the in-context synthesis and the standalone R1 report) converge here, and the reliability ceilings make it decisive: routing 100% of traffic through code-emit's probabilistic-correctness profile (~90–94% error-free, ~0.28 CW-SSIM) is wrong when a deterministic zero-eval path already serves the common case perfectly. So: code-island (jailed code-emit) = primary for *appearance*; declarative catalog = the chat's own dual-channel UI; **unify only via a shared promptable-design-system conditioning layer** (shadcn-registry / Design-Systems-2.0 adapter). **Highest-leverage v1.4 investment surfaced by R1:** add a **rendered-visual-compare step to the code-island repair loop** — it currently validates AST/syntax only, not visual fidelity, and every OSS screenshot-to-code tool needed render-and-compare. **This whole decision belongs to the LATER design-engine milestone (v1.4), NOT the v1.3 foundation** — so v1.3 is not blocked on it. Continue the fresh-web-validation pass (below) before locking it in a v1.4 discuss/plan.

## R2 — 2D infinite canvas + cross-panel shared state  `[MODEL — pending web-validation]`

- **Candidate libs:** **tldraw** (purpose-built infinite canvas, rich shape/tool model, snapshot store, arbitrary-React shapes, "Make Real"/"computer" precedent for AI-on-canvas; watch license + bundle) vs **React Flow / `@xyflow/react`** (already a dependency here — used by `/knowledge`; node/edge graph model, custom React nodes, `toObject()` serialization, data-carrying edges) vs **Excalidraw** (whiteboard, less suited to structured panels).
- **Reuse signal:** we already ship `@xyflow/react` + `@dagrejs/dagre`. React Flow's node/edge model maps *directly* onto both "panels wired by data edges" and the later "orchestrator→subagent graph" (R4) — strong reuse case. tldraw is the stronger *freeform-canvas* experience but a second heavy canvas dep. **Leaning: reuse React Flow for the panel-graph; revisit tldraw only if freeform drawing becomes a requirement.** NEEDS web-validation (perf at scale, live-updating React inside nodes).
- **Shared state between panels:** an observable per-chat store (signals/Zustand-style) + explicit **data-carrying edges** ("panel B consumes panel A's output") is the clean model; CRDT/Yjs only if multiplayer/robust-offline is needed later. Persist per chat via a canvas snapshot (`toObject()` / store snapshot) alongside the chat row.
- **Chat + canvas coexistence:** options are chat-as-docked-rail driving a canvas workspace vs chat-as-a-node on the canvas. Precedents: tldraw computer, Claude artifacts, ChatGPT canvas. **Leaning: chat is a first-class node on the canvas** (so the orchestration-viz pillar composes), with a convenience docked view.

## R3 — Dual-channel + streaming generative UI

- **Streaming transport (our stack: FastAPI+Bedrock → tRPC → Next.js):** Bedrock **`ConverseStream`** yields token/tool deltas server-side; bridge to the web via **SSE or tRPC streaming (`httpBatchStream`/subscriptions)**. Render **partial/streamed structured output** — the declarative spec must tolerate an incomplete tree (render-what's-valid, defer the rest). `[MODEL]` C1 and Vercel AI SDK both stream UI; AI SDK's `streamObject`/partial-object + RSC is the canonical reference `[MODEL, pending validation]`. Closes the long-deferred **GEN-04**.
- **Dual-channel round-trip (clarify-with-widgets) — authoritative blueprint** `[FRESH✓]`: **Vercel AI SDK "Generative UI"** (ai-sdk.dev, AI SDK 7) is exactly our pattern: the LLM emits schema-validated **tool-call args**, dev code maps **tool-name → a pre-built React component referenced by name**, tool output spread as props; the UI renders by tool-part state (`input-available`→loading, `output-available`→rendered result, `output-error`). The LLM never authors JSX. The RSC `streamUI` variant is **deprecated** ("experimental, not for production") in favor of AI SDK UI's typed `message.parts`. So our round-trip = agent emits an interactive genui node → user manipulates → the manipulation returns as the **tool result** that resumes the streamed run. **MCP Apps (SEP-1865, now an official MCP extension, 2026-01-26, OpenAI+Anthropic)** `[FRESH✓]` standardizes the security posture for the reverse direction: UI-initiated tool calls go over `postMessage`/JSON-RPC with **host pre-review + explicit user approval**. Also relevant: **AG-UI** (CopilotKit) + **Google A2UI** `[CORPUS/MODEL, pending validation]`. Pitfalls: stale UI, double-submit, resuming a paused run. **Use the declarative catalog for these widgets** (reliable + safe round-trip), not code-emit.
- **Proposal cards:** a bounded case of the above — agent offers pickable next-step cards; click → structured tool result. Lowest-risk dual-channel behavior; build first.
- **Anticipatory prompting (agent decides WHEN/WHAT to prompt):** **largely greenfield** — no strong published product/protocol; closest is HCI **mixed-initiative interaction** literature + proactive-Copilot-suggestion patterns `[MODEL]`. Pragmatic approach: a trigger/heuristic layer over chat+canvas state (idle, ambiguous intent, completed artifact, detected next-best-action) that proposes a genui prompt, **gated by an eval on appropriateness** (false-positive prompting is the main risk). Treat as a **SPIKE**.

## R4 — Orchestration-visualizer references + seams  `[MODEL — pending web-validation]`

- **Tool survey (abstraction level / live-viz):** n8n & Make = low-level *integration-step* nodes, per-node run status; Flowise/Langflow/Dify = LLM-app nodes with run streaming; **LangGraph (+ Studio)** & AutoGen (+ Studio) & CrewAI = *agent/graph* execution with run trees; Rivet/Vellum = agent IDEs with live graph run state. `[MODEL]`
- **"A few abstractions higher than n8n"** = **agent/task/intent-level nodes** (an orchestrator node that expands into a subgraph of subagents), i.e. hierarchical/nested graphs over a **run/span tree** (compare OpenTelemetry trace-tree / LangSmith run trees). `[MODEL]`
- **Remote-desktop + computer-use (forward-looking, brief):** building blocks = Claude computer-use / OpenAI CUA agents, **noVNC/WebRTC** remote desktop in-browser, **E2B / Vercel Sandbox** desktops. `[MODEL]`
- **SEAMS the v1.3 foundation must leave open (most important deliverable of R4):**
  1. **Panels-as-nodes generality** — a genui panel is one node *type*; the node model must admit other types later (agent, run, remote-desktop) without a rewrite.
  2. **A node-type registry** — additive, versioned, mirroring the genui component registry pattern.
  3. **Data/context-carrying edges** — edges aren't just visual; they transport state/context between nodes (the dual-channel + shared-state substrate AND the later dataflow-between-agents substrate).
  4. **A run/event schema stub** — even if v1.3 only ever has one "run" (the chat), model chat turns as events on a run so a live orchestration run-tree drops onto the same schema later.
  5. **An agent/run abstraction behind the chat loop** — the chat orchestrator is "an agent with one run"; keep the boundary so multi-agent orchestration is an extension, not a new system.

## Proposed Milestone v1.3 — "Conversational GenUI: Chat, Canvas & Dual-Channel" (foundation slice)

Local/sandbox only. Each phase eval/verify-gated (GSD). Requirement themes (new prefixes): **CHAT-**, **STREAM-**, **CANVAS-**, **STATE-**, **DCUI-** (dual-channel UI), **ANTIC-**, **SEAM-**.

- **Phase 22 — Chat spine + persistence + streaming.** `/chat` route; conversation/message/chat-group data model + Drizzle/Postgres persistence + per-chat state store schema; chat orchestration loop (FastAPI → Bedrock `ConverseStream`); streamed transport to web (SSE / tRPC stream); render streamed text + **streamed declarative spec** (partial-tree tolerant). *Closes GEN-04.* Everything hangs off this.
- **Phase 23 — 2D infinite-canvas medium + shared state.** Adopt canvas lib (recommend reuse `@xyflow/react`; validate vs tldraw); render existing genui specs/artifacts as draggable/pannable **panels-as-nodes**; per-chat shared-state store panels read/write; **data-carrying edges**; canvas persistence per chat. Builds R4 seams 1–3 now, surfaces nothing agent-y yet.
- **Phase 24 — Dual-channel genui: proposal cards + clarify-with-widgets + round-trip.** Agent emits declarative interactive genui as a turn; **widget→agent round-trip** (tool-result resumes the streaming run); wire into the loop; persist genui in the chat/canvas. Proposal cards first (lowest risk), then form/picker clarification.
- **Phase 25 — Anticipatory prompting (SPIKE→phase).** Trigger/heuristic layer over chat+canvas state deciding WHEN/WHAT to proactively prompt; eval on appropriateness (false-positive-prompting is the risk). Greenfield → spike first, then formalize. Builds R4 seams 4–5 (run/event + agent/run abstraction).

**Explicitly deferred to later milestones:** universal design-engine rework / screenshot→structure-clone + the unify-vs-hybrid lock (**v1.4**, gated on continued R1 research); the full **orchestration visualizer** (v1.5); **remote-desktop + automation + cross-chat context** (north-star). v1.3 only leaves the seams open.

Recommend closing v1.2 (at least the audit + the Phase-21 live verify) before formally opening v1.3 via `/gsd:new-milestone`, or consciously stacking v1.3 planning on top.
