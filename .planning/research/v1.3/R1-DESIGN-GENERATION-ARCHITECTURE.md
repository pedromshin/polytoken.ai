# R1 — Design Generation Architecture: Unify vs. Hybrid

**Date**: 2026-07-01
**Scope**: How leading prompt/screenshot -> UI systems reproduce arbitrary designs, and whether this codebase's genui engine should unify onto a single code-emit path or keep the current hybrid (zero-eval declarative core + jailed-eval code-island).

---

## Executive Summary

Every production system surveyed that claims to reproduce **arbitrary** designs (v0, Bolt.new, Lovable) does so via **code-emit**: the LLM writes real source files (React/Vue/HTML + Tailwind), and the environment runs real Node.js/npm tooling. None of them achieve "any design" through a declarative spec. Conversely, every system surveyed that prioritizes **safety and predictability over unlimited expressiveness** (Google's A2UI, Microsoft Adaptive Cards, the Vercel AI SDK's own "Generative UI" tool-call mechanism) uses a **declarative JSON spec rendered against a fixed, pre-vetted component catalog** — architecturally identical in spirit to this codebase's `packages/genui` core. No system in this survey does both through one unified mechanism; the two approaches solve different problems and every vendor that offers both ships them as **separate paths**, not a single unified engine.

Screenshot-to-code reliability, even for the best commercial multimodal models on a real-world benchmark (Stanford's Design2Code, 484 pages), tops out around **80-89% "LLEM"** (a structural/text element-match metric) and CW-SSIM (pixel similarity) scores around **0.27-0.28 out of 1.0** — i.e., far from pixel-perfect, with layout and visual-element recall as the dominant failure modes. This is consistent with v0's own published composite-model numbers: their best model reaches **93.87% "error-free"** generation only after a dedicated AutoFix repair model and post-hoc linting pass — meaning even Vercel's most specialized, RFT-tuned code-emit pipeline still needs ~1 in 15 outputs auto-repaired, and that number measures "compiles/lints clean," not "matches the target design."

**Recommendation**: keep the HYBRID architecture, but tighten the routing and invest in making the code-island's design-fidelity mechanisms (token injection, exemplar RAG) do more work — do not unify onto code-emit as the primary/only path, and do not abandon the zero-eval declarative core. See Section 6.

## 1. Per-system findings

### 1.1 Vercel v0 — code-emit, with a published multi-stage repair pipeline

**Generation path**: code-emit. v0 "takes a text description and turns it into a working React component," trained on "best practices for React, Tailwind and shadcn/ui," defaulting to shadcn/ui + Tailwind CSS as its component/styling substrate [1][2]. v0 is explicitly "trained on the default implementations of the shadcn/ui components" but remains "capable of generating code using other component libraries... and even frameworks" — i.e., it is not hard-constrained to a catalog, only biased toward one via training and RAG-style registry context [3].

**Architecture**: v0 uses a "composite model" family — a base generation model plus a dedicated `vercel-autofixer-01` **AutoFix model** trained via reinforcement fine-tuning (RFT) with Fireworks AI specifically to reduce error rates. The base model streams output while AutoFix "handles many issues mid-stream," followed by "a final pass to catch remaining issues" and a linter for "style inconsistencies and simple errors" [1].

**Published reliability**: v0 reports an "error-free generation rate" of **93.87%** for v0-1.5-md and **89.80%** for v0-1.5-lg, versus **78.43%** for Claude Opus and **58.82%** for GPT-4.1 as bare base models (no AutoFix), and the standalone AutoFix model itself achieves **86.14%** error-free output while running "10 to 40x faster" than comparable alternatives [1]. Important caveat: "error-free" here almost certainly means *compiles/lints without errors*, not *matches the design intent* — the source article does not define the metric precisely, and no visual-fidelity number is given. **Confidence: high** that this repair architecture exists as described (official Vercel blog); **medium** on interpreting exactly what "error-free" measures, since the precise eval methodology wasn't in the fetched excerpt.

**Registry / "promptable design system"**: v0 supports a **Registry** mechanism — "a distribution specification designed to pass context from your design system to AI Models." Teams build a registry (via Vercel's open-source `registry-starter` template, itself built on the shadcn/ui registry format), define components/tokens in `registry.json`, deploy it to a public URL, and connect it to v0 via "Open in v0" links or MCP. This gives v0 "a starting point and context on your specific design system" so generations stay on-brand without per-generation manual overrides [4][5][6]. Token/theme customization itself is conventional: CSS variables in files like `src/app/tokens.css` following the shadcn CSS-variable convention, plus `next/font` for typography — i.e., v0's "parametric design system" is really "give the model your existing shadcn-shaped token files as context," not a novel token protocol [4].

**Sandboxing**: not fully verified in this pass. v0.dev's own live preview environment is understood in the wider ecosystem to run in a Node-capable browser sandbox (commonly discussed as WebContainers-adjacent), but I could not confirm v0's specific preview sandbox mechanism from an official source in this research pass — **flagged as unverified**, do not cite a specific sandbox technology for v0 without further confirmation.

### 1.2 Vercel AI SDK "Generative UI" — declarative tool-call-to-component (NOT code-emit)

This is architecturally distinct from v0 itself, and is frequently conflated with it. The AI SDK's Generative UI (introduced in AI SDK 3.0, built on the same underlying tech that powers v0's chat interface) works by:

1. The developer defines a fixed catalog of **tools** (Zod-schema-typed functions) with an associated `render` function per tool.
2. The LLM is given the tool catalog and, based on conversation context, **calls a tool** (structured function-call, not code).
3. The SDK's `streamUI`/`render` mechanism invokes the developer's pre-built `render` function for that tool call, which returns a real React (Server) Component — the **LLM never emits component/JSX source code**, it only selects from and parameterizes a fixed catalog [7][8].

This is the same architectural family as this codebase's `packages/genui` declarative core (LLM picks from/parameterizes a fixed set, host code owns all rendering), just implemented via tool-calling + RSC streaming instead of a single validated JSON envelope. Note: AI SDK RSC-based Generative UI is documented as "experimental," with AI SDK's own docs recommending "AI SDK UI" (a more conventional client-state approach) for production use — worth noting as a caution about RSC-streaming-based generative UI specifically, not about the declarative-tool-call pattern in general, which remains the SDK's most widely used mechanism (tool calling underlies most agentic UIs built on the SDK) [7].

### 1.3 Bolt.new (StackBlitz) — code-emit inside a real in-browser Node.js runtime

**Generation path**: code-emit, confirmed via Bolt's (leaked, and now substantially reflected in the public OSS fork's) system prompt, which frames the assistant as "an expert AI assistant and exceptional senior software developer" operating inside "an in-browser Node.js runtime that emulates a Linux system," instructed to emit full project trees wrapped in `<boltArtifact>`/`<boltAction>` tags (file writes, shell commands, npm installs) [9][10].

**Execution/sandbox**: Bolt.new runs on **StackBlitz WebContainers** — a WASM-based Node.js runtime executing entirely inside the browser tab, with a Rust-based virtual filesystem and a virtualized TCP stack mapped onto the browser's ServiceWorker API. StackBlitz's own announcement states explicitly: **"100% of code execution occurs in the browser security sandbox"** — there is no remote VM, no server-side container. Isolation is whatever the browser tab itself provides (process/tab sandbox via SharedArrayBuffer-based cross-origin isolation), **not** an independent OS-level or kernel-level security boundary like Docker or a microVM [11][12]. This is architecturally the closest thing in this survey to "real code execution, browser-sandboxed" — genuinely more permissive than this codebase's iframe+CSP+AST-allowlist code-island, since WebContainers gives the generated code a real npm/Node environment with no AST restriction at all (only the browser's own tab isolation stands between generated code and, e.g., other tabs or the OS).

**Repair loop**: the open-source fork `bolt.diy` shows a manual "send preview error back to chat" UX pattern (errors from the WebContainer preview are surfaced as actionable alerts a user can forward to the AI), but I found **no evidence of an autonomous multi-attempt self-heal loop** comparable to v0's AutoFix or this codebase's validate→autofix→run→self-heal≤2 loop. This is a **documentation gap, not a confirmed absence** — bolt.diy's own README/FAQ candidly notes prompt tuning per-provider is still a roadmap item, suggesting the reliability engineering here is less mature/published than v0's [13].

**Reliability**: no official StackBlitz benchmark found. A secondary, low-confidence source (a review-aggregator blog, not StackBlitz) cites a 1.4/5 Trustpilot score and describes an anecdotal failure pattern of fix-one-break-another loops with high token cost from re-sending full codebases on each correction — **treat as unverified anecdote**, not a benchmark [14].

### 1.4 Lovable.dev — code-emit in isolated cloud containers (thin public documentation)

**Generation path**: code-emit — React + Vite + Tailwind, confirmed via Lovable's own engineering blog post on their "Visual Edits" feature [15].

**Execution**: unlike Bolt, Lovable runs "ephemeral development servers" in what their own blog describes as containerized Node.js environments — "over 4,000 instances," one isolated project per container, with HMR pushed to the browser. This is server-side/cloud container isolation, not browser-WASM sandboxing [15]. (A secondary/community source claims fly.io as the hosting substrate; **this is not confirmed by Lovable's own materials** and should be treated as unverified.)

**Overall transparency**: markedly thinner than StackBlitz or Vercel. No equivalent technical deep-dive on sandbox security model, threat model, or repair-loop architecture was found from Lovable's own sources. This is a genuine documentation gap in the space, not a finding to build architecture decisions on.

### 1.5 Google A2UI — declarative protocol, explicitly safety-motivated, directly analogous to this codebase's declarative core

A2UI ("Agent-to-UI," `github.com/google/A2UI`, current stable release v0.9.1, v1.0 as release candidate) is an open protocol where **agents send a declarative JSON format describing UI intent**; the client renders it using its own native, pre-approved component library (Flutter, Angular, Lit, React, etc. — framework-agnostic by design) [16][17][18].

**Architecture**: components form a **flat list with ID references** rather than a nested tree — explicitly chosen because a flat list is "easy to generate incrementally, correct mistakes, stream" (i.e., an LLM can emit/patch one component at a time rather than needing a complete valid tree upfront). A separate **Data Model layer** holds application state; components bind to data via JSON-pointer-style paths (e.g., `{"value": {"path": "/booking/date"}}`), and user interactions surface back to the agent as named action events [19].

**Explicit safety rationale (verified, high confidence)**: Google's own docs state the core security property directly: **"Declarative data, not code. Agent requests components from client's trusted catalog. No code execution risk."** [19]. This is Google explicitly choosing the same tradeoff this codebase's `packages/genui` core already made — and publishing it as the headline security property of the protocol, not a footnote.

**Relation to A2A/MCP**: A2UI is transport-agnostic — its docs state messages can "travel over any transport (A2A, AG-UI, SSE, WebSockets)" — positioning it as a UI-payload format that can ride on top of Agent2Agent (A2A) or other agent transports, rather than a competing or dependent protocol [19]. **Confidence note**: A2UI is very new (announced ~December 2025 per the MarkTechPost coverage found [17]); treat it as an early/evolving spec, not a mature, widely-battle-tested production standard yet — the protocol's ideas are well-precedented (see 1.6) even though this specific implementation is young.

### 1.6 Prior art for the declarative pattern: Microsoft Adaptive Cards (decade-mature production precedent)

Adaptive Cards (Microsoft, in production since ~2017 across Teams, Outlook, Windows) are "platform-agnostic snippets of UI, authored in JSON," rendered into native UI by per-platform SDKs. Microsoft's own docs state the safety rationale nearly verbatim to A2UI's, a decade earlier: **"Interactivity is expressed declaratively to help reduce risk of custom code injection"** and "no code apart from that understood by [the] SDK can be passed to the end user" [20]. This is strong, long-running production evidence that the "declarative spec + fixed trusted renderer" pattern this codebase's `packages/genui` core uses is not a toy or compromise — it is the standard approach for any system that must safely render agent/LLM/service-generated UI at scale where the sender is not fully trusted. Note Adaptive Cards is a fixed, versioned schema (v1.5/v1.6) authored largely by humans/services, not typically an LLM emitting freeform designs — so it validates the *safety* pattern strongly, but says nothing about *design-reproduction fidelity*, which is this codebase's actual open question for the declarative path.

### 1.7 Thesys C1 and MCP-UI — not independently verified this pass

I was not able to independently verify detailed architecture claims for Thesys C1 or MCP-UI within this research pass (the dedicated sub-agent assigned to this topic did not return usable findings due to a tooling issue). **This is a gap** — both are relevant comparables (C1 as another "Generative UI API," MCP-UI as a sandboxed-iframe app-island pattern for MCP tool results) and should be researched in a follow-up pass before treating this report as complete on that axis. ## 2. Screenshot/website -> UI pipeline & reliability

**Benchmark**: Stanford SALT Lab's **Design2Code** (arXiv:2403.03163, NAACL 2025) is the most-cited real-world benchmark for this task: 484 manually curated real webpages, evaluated via both automatic metrics and human evaluation [21][22][23].

**Metrics and numbers found** (medium-high confidence, sourced from a secondary aggregator summarizing the paper/leaderboard, not the raw PDF — treat exact figures as approximate):
- Best commercial models (GPT-4o, Gemini 1.5 Pro) reach **CW-SSIM ≈ 0.27-0.28** (pixel-structural similarity, scale 0-1) and **CLIP similarity ≈ 87-89** (out of 100) [24].
- **LLEM** (a low-level element-match metric for text/layout fidelity) exceeds 80% *only* for top commercial models (GPT-4o ≈83.67%, Gemini 1.5 Pro ≈87.17%); open/fine-tuned models (e.g., WebSight+WAFFLE) trail at ≈77.81% [24].
- The paper's own framing (per Stanford's project page) is that models "mostly lag in recalling visual elements from the input webpages and generating correct layout designs" — i.e., **layout structure and element recall are the dominant failure modes**, not, e.g., color accuracy [22][23].
- I could **not** verify a single specific "% rated good/acceptable by humans" headline number from a readable source in this pass (the raw PDF was not machine-readable via WebFetch, and the secondary summary did not surface a human-eval win-rate). **Flagged as unverified** — a follow-up pass should pull the actual paper text (e.g. via arXiv HTML view) for the precise human-eval numbers before quoting a specific "X% human-rated success" figure.

**WebSight dataset**: referenced as a fine-tuning dataset (HuggingFaceM4) used by open models (e.g., "WebSight+WAFFLE" in the Design2Code leaderboard) to improve screenshot-to-HTML/Tailwind generation; per the numbers above, fine-tuned open models still trail top commercial multimodal models on this benchmark, suggesting the gap is not purely a data problem but also a base-model capability gap. **Confidence: medium** — this reference is secondhand via the Design2Code leaderboard summary, not independently verified against WebSight's own dataset card in this pass.

**OSS reference implementation**: `abi/screenshot-to-code` (widely-used open-source project) supports HTML+Tailwind, React+Tailwind, Vue+Tailwind, Bootstrap, and Ionic+Tailwind as output targets, using current-generation multimodal models (Gemini 3, GPT-5.x, Claude Opus 4.x per its README at time of research) [25]. Notably, its README documents a **"Screenshot preview"** capability where "the agent renders its own generated page in a headless browser and visually checks its work" — i.e., a render-and-compare validation step, evidence that even a lightweight OSS single-shot tool has converged on needing a **feedback loop**, not pure one-shot generation, to get usable fidelity [25]. This is corroborated by two academic works surfaced in search but not deeply verified this pass: "VisRefiner: Learning from Visual Differences for Screenshot-to-Code Generation" and "Coding with Eyes: Visual Feedback Unlocks Reliable GUI Code Generating and Debugging" — both titles directly assert that closing the loop with rendered-visual feedback (not single-shot VLM->code) is necessary for reliability; **titles/existence verified via search results, content not independently fetched/confirmed this pass** [26][27].

**Synthesis on pipeline pattern**: the literature converges on a staged/iterative pattern rather than pure single-shot image-to-code: generate → render → visually compare → correct, repeated a bounded number of times. This is the same shape as this codebase's code-island repair loop (validate → autofix → run → self-heal ≤2 → placeholder), just applied to visual/design fidelity rather than (or in addition to) syntactic/runtime correctness. No source in this pass described a distinct "VLM -> structured layout/bounding-box intermediate representation -> tokens -> code" pipeline as a *named, separate* architecture stage in production tools (v0/Bolt/Lovable all appear to go screenshot/prompt -> code more directly, using the model's own multimodal understanding rather than an explicit intermediate IR) — this specific staged-IR pattern shows up more in the *token extraction* tooling (Section 3) than in end-to-end screenshot-to-app tools. **Flagged**: I could not find a production system that explicitly publishes a "VLM -> IR -> tokens -> code" staged pipeline as such; this appears to be more of a reasonable architectural hypothesis than a proven-in-production pattern.

**Realistic reliability ceiling**: based on the numbers above, single/few-shot screenshot-to-code with top-tier commercial multimodal models, even on the best current benchmark, lands around **CW-SSIM ~0.28** (well short of pixel-perfect — 1.0 would be identical) and **~80-89% low-level element match**. Combined with v0's own **93.87% "error-free"** figure (which is a code-quality metric, not a design-fidelity metric, and is achieved only with a dedicated repair model), the honest ceiling for "any design, reliably, on the first or second try" is: **good structural/layout approximation, imperfect pixel fidelity, meaningful residual error rate even after repair loops** — not "solved." This directly supports keeping a repair/self-heal loop (as this codebase already has) rather than assuming code-emit alone gets to production-grade reliability without one.

## 3. Design tokens from images

**Production tools** (verified to exist via search, architecture details not independently confirmed beyond marketing copy — **medium confidence**, no peer-reviewed source found for this specific sub-topic):
- **Peel** (peel.studio) — accepts images/video/URLs, uses Claude or Gemini to extract colors with "semantic roles," font families/weights/sizes/line-heights, and visual components with CSS properties [28].
- **MiroMiro** — extracts colors, typography, spacing, shadows, radii from a website URL; exports as CSS variables, Tailwind config, or Figma variables [28].
- **arvindrk/extract-design-system** (GitHub, open source) — a CLI + "AI agent skill" (for Claude/Cursor/Codex) that extracts colors, typography, spacing, border-radius, and shadows from a public website, outputting JSON + CSS custom properties [28].
- A Chrome extension, "Design Token Extractor," does the same for arbitrary websites via right-click/DOM inspection rather than image analysis [28].

**Pattern observed**: the common technique across these tools, as far as their public descriptions go, is **DOM/CSS analysis of a live website** (extracting computed styles directly) rather than pure image-based extraction via computer vision (k-means color clustering, OCR-based font detection, edge-detection for spacing) — i.e., most "extract tokens from a screenshot" tools in practice mean "point me at a URL and I'll read the CSSOM," which is a fundamentally easier and more reliable problem than true image-only extraction. **I could not verify** a widely-used production tool that does token extraction from a *raster image alone* (no DOM access) with published accuracy numbers — this is a genuine capability gap in the current tooling landscape, worth flagging if this codebase ever needs to support "upload a design screenshot with no accessible source," as opposed to "point at a live URL."

**W3C DTCG spec** (design-tokens.github.io / designtokens.org — the spec this codebase's style packs already conform to): confirmed structure — tokens are JSON objects identified by a `$value` property, with `$type` for type ("color", "dimension", "fontFamily", "fontWeight", "duration", "cubicBezier", "number" as primitives; "strokeStyle", "border", "transition", "shadow", "gradient", "typography" as composites), `$description`/`$deprecated`/`$extensions` as metadata, grouping via nested objects, and token references via `{group.token}` curly-brace syntax or JSON Pointer [29]. This confirms the codebase's existing 6 style packs are aligned with the actual, current W3C spec structure — no gap found here.

## 4. "Promptable design system" in practice

**shadcn/ui registry** (`registry.json`): the most concrete, widely-adopted real-world example of a "parametric design system a builder agent can install/configure rather than generate from scratch." Structure: root `registry.json` with `name`, `homepage`, `items` (array of component definitions), and `include` (to compose multiple registries). Each item has `name`, `type` (`registry:ui`, `registry:block`, etc.), `registryDependencies` (references to other components, including cross-registry via namespaced packages or URLs), `dependencies` (npm packages), and `files` (source locations) [30]. The `shadcn add <url>` CLI command resolves the registry graph and installs vetted files directly — this is explicitly designed so an AI agent can be pointed at a URL and "install" trusted, pre-reviewed components instead of generating raw component code from scratch, closing exactly the reliability gap that free-form code-emit has (Section 2) for the *component* layer, while leaving page composition/layout to the agent [30].

**v0's registry mechanism** (Section 1.1) is built directly on this shadcn registry format — Vercel's own `registry-starter` template is "a free, open-source template built with Next.js and Shadcn/ui Registry to accelerate your AI-Native Design System" [6]. The pattern in production, across both shadcn itself and v0's adoption of it, is: **token/theme values as CSS variables (swappable, parametric) + a fixed catalog of components as files an agent can reference/install (not regenerate)**. This is architecturally very close to this codebase's Catalog -> Spec -> Registry -> Renderer core, with one difference worth noting: shadcn's registry lets the agent pull in *new* component source files (still code, just pre-vetted/human-reviewed) into a real build, whereas this codebase's declarative core never lets the LLM introduce new component code at all — only select/parameterize an already-shipped, already-audited set of ~16 components. shadcn's model is "curated code-emit," a middle point between pure code-emit and pure zero-eval declarative rendering.

**No evidence found** of a published academic/industry treatment of "parametric design systems" as a named, general concept distinct from ad hoc theme systems — this appears to be an emergent, tooling-driven pattern (shadcn, v0, Peel/MiroMiro-style extraction, Radix Colors-style palette generation) rather than a formally theorized one. This codebase's own style-pack system (6 DTCG token sets selected per generation, injected into both the declarative spec envelope and the code-island's exemplar RAG) is a reasonable, evidence-consistent instantiation of this pattern — no counter-evidence found that a more elaborate "fully generative" token system would outperform a curated set of pre-designed packs for reliability purposes.

## 5. Unify vs. Hybrid — decision matrix

| Dimension | UNIFY (code-emit primary, drop declarative core) | KEEP HYBRID (current) | Notes / evidence |
|---|---|---|---|
| **Ability to reproduce arbitrary designs** | High ceiling in principle (matches v0/Bolt/Lovable's approach) | Declarative path capped at ~16-component catalog expressiveness; code-island already covers the long tail | No system studied gets "any design" *without* code-emit somewhere — but also none gets it *reliably* without a repair loop layered on top (Sec. 2) |
| **Safety / attack surface** | Must trust a jailed-eval sandbox for **100% of traffic**, not just the long tail | Zero-eval core handles the (likely majority) common-case traffic with **provably no eval/Function/dangerouslySetInnerHTML** on that path; only long-tail traffic touches the sandboxed code-island | This is the single biggest asymmetry. A2UI and Adaptive Cards both exist specifically because "declarative, not code" eliminates a whole vulnerability class [19][20]. Unifying onto code-emit means the AST-allowlist + iframe jail becomes a single point of failure for *all* traffic instead of a bounded subset |
| **Reliability / correctness** | Even best-in-class (v0's composite model + AutoFix) reports ~90-94% "error-free" — implying ~6-10% of *all* traffic needs the repair loop or fails | Declarative path is validated by Zod at the boundary — a request either produces a valid spec or it doesn't (deterministic, not probabilistic "usually compiles") | Moving 100% of traffic to code-emit imports v0/Bolt's residual failure rate as the new baseline for cases that previously had zero-eval's deterministic correctness |
| **Maintainability** | One code path, but that path is inherently more complex (AST allowlist, repair loop, self-heal budget, safe-placeholder fallback) carrying *all* traffic instead of a fraction | Two code paths to maintain, but each is simpler in isolation; declarative core's simplicity is a maintainability asset, not just a safety one | Vercel, StackBlitz, Lovable all ship v0/Bolt/Lovable as effectively single-path *products* — but none of them also ship a zero-eval fallback, because they don't have this codebase's dual audience (need both "just work reliably" and "reproduce anything") |
| **Latency / cost** | Every generation pays code-emit + sandboxed-run + potential repair-loop cost | Fast path (declarative) is cheap or free of that cost for the common case; only long-tail pays it | Not deeply verified with hard numbers this pass, but structurally true given code-emit requires a full LLM code generation + parse + execute + validate cycle vs. a single structured tool-use call for the declarative path |
| **Precedent for the hybrid shape itself** | None found — no surveyed vendor runs "declarative fast path + code-emit long tail" as one product | None of the surveyed *code-emit* vendors need this because they don't have a declarative product; but the *split itself* (cheap fixed-catalog path + expensive open-ended path) mirrors a common systems pattern (e.g., CDN cache-hit vs. origin-fetch, compiled fast path vs. interpreter fallback) even if no *design-generation-specific* vendor was found doing exactly this | Architecturally novel combination, not unprecedented pattern-shape |

### Recommendation

**Keep HYBRID.** Do not unify onto a single code-emit engine, and do not drop the zero-eval declarative core as the primary renderer. Rationale, directly from the evidence above:

1. **No comparable system reproduces arbitrary designs without code-emit, and no comparable system that needs guaranteed safety uses code-emit as its only path.** The market has effectively already bifurcated into "declarative-and-safe" (A2UI, Adaptive Cards, AI SDK Generative UI, and this codebase's own core) vs. "code-emit-and-expressive" (v0, Bolt, Lovable) camps, each accepting the other's weakness as a tradeoff. This codebase's hybrid isn't a compromise stuck between two "real" approaches — it's a synthesis of both camps' actual production patterns, applied selectively rather than uniformly.
2. **The reliability data argues against defaulting to code-emit.** Even Vercel's most specialized model + dedicated repair-model pipeline reports ~90-94% "error-free" — not "matches the design," just "doesn't crash/lint-fail." Design2Code shows even top commercial models cap out at CW-SSIM ~0.28 for visual fidelity. Routing 100% of this codebase's traffic through that reliability profile, when a materially cheaper and deterministically-correct declarative path already exists for the common case, would be a regression for most real traffic to chase headroom that's only needed for the tail.
3. **Abandoning zero-eval has a real, disproportionate safety cost with no reliability offset.** The current architecture's biggest asset — "no eval/Function/dangerouslySetInnerHTML anywhere on the render path" for the fast path — is exactly the property Google (A2UI) and Microsoft (Adaptive Cards) independently converged on as the correct default for **any system rendering agent/service-generated UI where the generator isn't fully trusted**. Moving 100% of traffic through the sandboxed code-island doesn't just add engineering surface (AST allowlist maintained forever, iframe jail correctness burden on every request) — it removes the one path that has a *provable* (not merely tested) safety property, in exchange for uniform expressiveness that most requests don't need.
4. **What DOES need work, per this research, is the routing/coverage boundary, not the split itself.** Given Design2Code's finding that layout/element-recall (not color/type) is the dominant screenshot-to-code failure mode, and that even OSS single-shot tools have converged on render-and-compare feedback loops for acceptable fidelity — the highest-leverage investment is likely: (a) tightening the criteria for when a request routes to declarative-fast-path vs. code-island, so more "any design" requests that *can* actually be served by the 16-component catalog + style packs stay on the safe/cheap/deterministic path, and (b) making sure the code-island's repair loop includes a visual/rendered feedback signal (not just syntactic/AST validation), consistent with what the screenshot-to-code literature suggests is necessary for real fidelity gains, since the current plan description (validate → autofix → run → self-heal ≤2 → placeholder) is not confirmed to include a rendered-visual-comparison step.

**Third option not recommended but worth naming**: a middle path modeled on shadcn's registry (Section 4) — let the code-island's repair loop, or a future assembly step, "install" additional *pre-vetted* catalog components (still zero-eval, still Zod-validated) rather than raw generated code, narrowing the gap the code-island exists to cover without expanding the trusted-code surface. This is a incremental widening of the declarative core's catalog rather than a wholesale unification — likely lower-risk than either pole, and consistent with how shadcn/v0 actually scaled expressiveness without abandoning a curated-component model. This is a design direction for a future phase, not an immediate recommendation, since it requires its own catalog-growth process.
